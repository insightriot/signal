## Premise correction (overturns a DISCUSS assumption; conclusion stands)

DISCUSS D-v016-2 framing assumed "a PreToolUse hook can't cleanly emit a non-blocking warning." **That is false** (per Claude Code hooks docs, `code.claude.com/docs/en/hooks`): a PreToolUse hook **can** warn non-blockingly via JSON `systemMessage` (→ user) + exit 0 — **without** setting `permissionDecision: "allow"` (which would silently bypass the permission prompt). So write-time size-warn IS possible.

**But the conclusion (size-warn at read-time) stands, now for a stronger reason.** A write-time `systemMessage` means the hook fires on **every Edit/Write in every repo** the user has Signal installed in (matcher = tool name, not path; self-gated by `file_path`), with per-edit cost + cross-project surface. The read-time banner in `/sig:resume` `/sig:status` `/sig:checkpoint` has **zero cross-project blast radius**, is scoped to Signal's own commands, and reuses the established SessionStart `additionalContext` pattern. So FR2 stays read-time **by choice, not by limitation.** (Flagged at the plan gate; not a re-gate — the outcome is unchanged.)

Corollary confirmed: the block half is a clean `exit 2` + stderr (existing hook's mechanism); don't mix exit-2 and JSON in one hook.

## FR1 — block prose in STATE frontmatter (P1 — highest blast radius)

**Where it goes (prior-art + codebase):** a **sibling pure function** — NOT bolted onto `checkProposedStateWrite` (`retrospective.js:358`), which early-returns `{block:false}` for any non-SHIP phase (`retrospective.js:374`); the prose check must fire on **any** phase (AC1.1/1.2 have no SHIP precondition). The hook (`hooks/check-state-write.js:82-90`) composes **two** independent predicates, OR-combines `block`, and the `reason` names which fired. Existing 6 `checkProposedStateWrite` cases in `tests/hook-state-write.test.js` are the AC1.4 no-regression pin.

**The discriminator must be RAW-TEXT, line-based — NOT parsed YAML (R1.3, binding):** `parseFrontmatter` *throws* `StateSchemaError` on malformed YAML, which a prose blob often is (unquoted colons, bare newlines) → a parse-then-check design fails **open** and misses its headline case. So operate on the raw frontmatter block (between the first two `---`).

**Field-specific + blacklist-not-whitelist (advisor correction — a naive "multi-line item → block" rule wedges every blocker):** the two fields have different legit shapes, so the check is per-field, and it blocks only *clearly*-malformed input (when in doubt, **allow** — a false positive wedges a stranger = P1; a false negative just defers to FR2's banner):
- **`completed_phases`** — scalar strings; the writer only ever emits a single physical line. So a raw item that **spans multiple physical lines** OR **exceeds ~150 chars** → block. (The cmmc double-quoted scalar spans ~54 physical lines → the raw multi-line check catches it; length is the backstop for a pathological single-long-line item. This reconciles the "length-primary" reading — length is primary only under the *parsed-value* design R1.3 forbids.)
- **`blockers`** — object mappings; a real blocker legitimately spans 3 lines (`id`/`text`/`raisedAt`), so **NEVER flag a blocker item for being multi-line.** Inspect only the **`text:` value**: block if it's a multi-line block scalar OR exceeds ~500 chars. Shape check is is-mapping-with-`text` (don't demand a strict positive shape).

**Length budgets (R1.1 — generous, named constants, never punctuation):**
- `COMPLETED_PHASES_MAX ≈ 150` chars: legit shapes are `PHASE (YYYY-MM-DD)` (~17–24 chars) and annotated `PHASE (date) — <note>` (observed max **58** chars, cmmc line). 150 gives wide headroom over 58, far below 3874.
- `BLOCKER_TEXT_MAX ≈ 500` chars: legit example `"Marketplace install hangs on first run; tracked under F2"` (~52 chars, has a semicolon — punctuation is NOT prose). 500 is well over 52, below the ~3000-char blob.
- Locked legit-passers (become AC test cases): the 58-char annotated `completed_phases` entry, the 52-char semicolon blocker `text`, `blockers: []`, a legit 3-line blocker object, and Signal's own current frontmatter — **all must pass**.

**R1.2 — full-content re-trip → resolved by SELF-UNBLOCK (the load-bearing fact, per advisor):** the hook is **stateless** — it inspects proposed content only. So a cleanup edit that produces clean frontmatter **always passes**; the user unblocks themselves by deleting/relocating the prose the `reason` string names. And Signal's own writers (`transitionPhase`, `markFresh` via `node:fs`) bypass the hook entirely (R1.5). So (a) is **not** "un-editable" — it's "Claude's Edit/Write is gated until the frontmatter is clean, and cleaning it is a *passing* edit." This kills the "you bricked pre-polluted files" objection — **but only because it's an explicit AC** (AC1.7: *a write producing clean frontmatter is allowed regardless of the file's prior polluted state*). **Plan decision: (a) full-content block + surgical reason + the self-unblock AC** — over (b) delta-only, which is heavier and never *forces* cleanup of existing pollution (weaker for a doc-integrity batch). Day-one strangers start clean (`initState`) so the block never fires for them. Signal's own frontmatter is clean → no self-lock. **Surfaced at the gate (a + self-unblock + why-not-b) for veto.**

**R1.5 (benign, state explicitly):** `state.js` programmatic writers (`transitionPhase` appends a clean `PHASE (date)`; `addBlocker` builds a clean `{id,text,raisedAt}`) go through `node:fs`, never the hook. They're clean by construction — the bypass is intentional and safe (writer A = state.js clean; writer B = agent Edit/Write is the polluter the hook targets).

## FR3 — drain blockquote detector (additive, S)

The `^` anchor is load-bearing: the live file quotes the stamp mid-line in backticks (`FUTURE-IDEAS.md:1698`) — an unanchored regex would flip that live entry to dispositioned and **silently drop a backlog item**. **No baseline break (risk R3.2):** `tests/drain.test.js`'s snapshot fixture has **zero** leading-`> **` blockquote stamps (its "Promoted" strings are `**Status:**`-line stamps the `^>` detector never touches) → 7/5 snapshots stay green (AC3.2). ## FR4 — deriveHeading clause-boundary cut (S)

**Extension point:** `deriveHeading` (`add.js:365-371`), the private helper shared by all 3 consumers (`buildFutureIdeasEntry:382`, `buildOpenQuestionsEntry:603`, `buildMilestoneEntry:648`) — one change covers all. Not exported → tested indirectly via `buildFutureIdeasEntry` (`tests/add.test.js`).

**Rule + edge cases to lock as tests (R2.1):** prefer the first clause boundary before the length cap, fall back to the current 6-word/60-char slice.
- **Boundary set = `—` (em-dash U+2014), `.`, `:`, `,`** — and **only when followed by whitespace/EOL** (so `https://` colon+`/` and `example.com` period+letter never trigger — URL-body guard).
- **Em-dash only, never hyphen-minus `-`** (AC4.1's own example has a hyphen in "closed-work" and the real cut at the `—`).
- **Minimum-length floor** — skip a boundary that yields < ~20 chars / < 3 words (so `OK, so…` / `auth, billing…` don't cut to a 2-char heading); fall through to the next boundary or the word/length cap.
- **60-char cap still applies** after any clause cut (AC4.3); the existing "caps at 60 / ≤63 with `## `" test (`add.test.js:303`) must stay green via the fallback path.
- **`?`/`!` intentionally excluded** from the boundary set (per spec) — a short question heading stays whole under the cap. (Conscious decision, not omission.)
