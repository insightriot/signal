## What changed mid-flight

- **Half the "new" work already existed.** `markFresh` was built in E6, so S1.t5 was command-prose wiring (discuss/plan), not implementation — the retro-honest framing was "lock the contract the wiring depends on," not "TDD a new helper."
- **S3.t1 grew a caller-announce leg** not in its file list: `listDrainCandidatesWithRecovery` is inert without `plan.md` §1b actually calling it and announcing the dangling-fence warning, so the wiring landed in the same task.
- **The scope of the F1 fix widened deliberately.** REVIEW found `isStaleVsOrigin` crashed on a schema-drifted STATE.md; the same latent throw sat in `isStateStale` (pre-existing E6), and since `isStateStale` runs *first* in resume/checkpoint, leaving it unfixed would have meant FR5's schema-drift banner never rendered there. Fixing both was completing FR5, not scope creep — named as such rather than silently expanding or silently leaving the hole.

## What assumptions broke

- **"RED-first TDD + a fully green suite ⇒ the feature is sound on its own edge inputs" — broke hard.** 848 passing tests did not catch that the schema-drift feature *crashes on schema drift*: `isStaleVsOrigin`/`isStateStale` called `readState` (which throws `StateSchemaError` on an ahead/missing-key schema) *outside* their fail-open try/catch. No test had planted a drifted STATE.md against the staleness path, so the suite was green and the command was broken on the exact file the Epic exists to handle gracefully.
- **"`execFileSync` with an args array neutralizes injection" — only half true.** It blocks *shell* injection, but not git *option* injection: a crafted `last_updated_commit: "--output=/path"` glued into a revision range is parsed by git as an option. Not exploitable as composed (a `rev-list` gate rejected it first), but a latent arbitrary-file-write one refactor away.

## What surprised us

- **Two independent review agents caught the *same* crash (F1)** — a strong, uncoordinated signal that it was real, not a plausible-but-wrong finding. The security auditor verified everything empirically (crafted STATE.md fixtures + a local origin-ahead repo), not by reasoning alone, including confirming the billion-laughs YAML bomb is caught and no file is written by the `--output=` payload.
- **The FR1 retro gate correctly *declined* to fire** while `MILESTONE-4.5.md` still marked E10 "pending" — the deliberate chicken-egg where SHIP must annotate the Epic as shipped before the gate enforces the retro. The mechanism worked exactly as E9 designed it.

## What we'd do differently

- **Plant adversarial STATE.md fixtures against every new read path during EXECUTE**, not wait for REVIEW. The F1 crash was findable at S1/S4 with one "fail-open on a drifted STATE" test per new helper — the review caught it, but a checklist item would have caught it a phase earlier and cheaper.
- **Be disciplined about set-before-every-task even for XS prose.** The auto-state-protocol was skipped on S2.t2 (a quick prose task), so a mid-task context-clear there would have mis-reported the last-completed task until I corrected it.

## What to feed back into Signal

- **REVIEW is a load-bearing gate, not a formality — keep it hard at FULL.** A reachable crash slipped past a green 848-test VERIFY; the four-lens independent-agent review is what caught it. The strongest argument for the REVIEW phase's existence is this Epic.
- **Add a Signal checklist item / skill note:** every new file-reading helper a command calls "unguarded" (because a docstring promises never-throws) needs an explicit "fail-open on a malformed/ahead-schema STATE.md" test. This is a recurring class (F1, F3).
- **The `COMMIT_TOKEN_RE` option-injection guard is a reusable pattern** for any user-editable YAML value glued into a git argument — worth codifying in a security reference alongside the `execFileSync`-args-array rule (the array is necessary but not sufficient).
- **Candidate `/sig:add` follow-up:** the drain-recovered-entry seam (a byte range with no `parseEntries` index) suggests drain candidates should carry a stable identifier disposition can consume, so recovery and disposition can't drift — logged as forward work, not this Epic.

## Anti-rationalization moment

The live temptation was at the VERIFY→REVIEW seam: *"848 tests pass, the code demonstrably works end-to-end, REVIEW is ceremony — just PASS it and cut the release."* That rationalization is precisely the one the REVIEW phase exists to defeat, and honoring it (running the full audit with two independent agents) is what surfaced F1 — a crash on the exact input class the Epic was built to make graceful. Working code that passes tests can still be insecure and can still crash on its own motivating edge case; the review catches what the tests, by construction, did not think to assert.
