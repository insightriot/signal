## FR1 — Migrate dangling-delta survives closed-block move + reroot (B24)

`applyMigrate` must not attribute a **pre-existing** dangling `](*.md)` link to the migrate when the append-log evict relocates + re-roots a strictly-closed DECISIONS block. Fix `computeDanglingDelta` to key on the resolved repo-root-relative absolute target with multiset/count semantics (`CONTEXT.md` D1).

**Acceptance criteria**
- **AC1.1** A v3-pending repo whose strictly-closed DECISIONS block carries a pre-existing broken `](*.md)` link **migrates successfully** — block relocates to `archive/M{n}/DECISIONS.md`, live `DECISIONS.md` shortens, no abort. (Invert `tests/migrate-dangling-baseline.test.js:162-219`.)
- **AC1.2 (divergence closed)** Any link `renderDryRun` lists under "Pre-existing dangling links" must NOT cause `applyMigrate` to abort. A single test runs both on the same fixture and asserts consistency.
- **AC1.3 (gate not weakened — NEW non-injected test)** A genuinely migrate-*introduced* dangle (resolved abs-target absent from the pre-apply baseline — e.g. the migrate moves a real target file out from under a still-live link) still aborts + rolls back byte-identical. Must not rely on the existing injected-scanner test.
- **AC1.4 (multiset semantics)** One pre-existing dangle to X + a migrate-introduced *second* reference dangling to the same X → the extra one is attributed and aborts.
- **AC1.5** `computeDanglingDelta` unit test carries the new resolved-target field and still returns only the NEW dangle.
- **AC1.6** Named limitation (delete-one/add-one to the same already-missing target → masked) documented in `BUGS.md` + a decision note; not silently shipped.

- **AC2.3 (tier-aware)** SKETCH (REVIEW skipped) is Epic-close at VERIFY-complete; SPIKE (SHIP skipped) never reaches FR1; FEATURE/FULL require REVIEW-complete. Phase-name match tolerates the `(date)` suffix.

- **AC2.5 (Layer 2)** `checkProposedStateWrite`'s trigger no longer depends on a `- SHIP` entry Signal never writes (`epic-native-flow.md:27`); re-keyed off `phase: SHIP` + tier-complete pre-SHIP phases.

## FR3 — FR5 read-enclosure has a behavioral (interleaving) test (B25)

The RMW lock protection must be proven by a test that fails on a read-outside-lock wrapper, not merely a throw-under-held-lock assertion (`CONTEXT.md` D3).

**Acceptance criteria**
- **AC3.1 (discriminates — core)** The interleaving harness run against a deliberately-broken `reads-outside-lock` twin yields a lost update and **FAILs**; against the real correct wrapper it **PASSes**.
- **AC3.2 (behavioral)** The assertion inspects final on-disk content for a lost append/stamp — not a throw-under-held-lock (already covered).
- **AC3.3 (throw-lock semantics)** In the correct case the *arriving* writer fails fast with a lock error; the test asserts it threw AND the first writer's change landed AND the second's is absent *because it never committed* (not clobbered).
- **AC3.4 (inert in production)** `_afterRead` defaults to `undefined`; no `commands/*.md` caller passes it; prod path byte-identical (mirrors `renameFn`, honors NFR2).
- **AC3.5 (no re-entrancy regression)** `promoteDrainEntry`'s paused writer still routes through the lock-free Core — the hook must not reintroduce a nested self-locking acquire.
- **AC3.6 (full close)** Every entry in `RMW_PATHS` (all 6) carries an interleaving assertion; `regenerateIndex` grows `(baseDir, opts = {})` forwarded to its Core. `M5.E4-REVIEW.md:29` "7 wrappers" wording reconciled.
