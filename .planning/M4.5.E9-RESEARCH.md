## Synthesis preamble — tensions and convergences

**Three convergences worth highlighting:**

1. **The cleanest mechanism is `PreToolUse` matcher on `Edit|Write` against STATE.md, NOT a generic Stop hook.** Project-researcher flagged that `Stop` fires every turn (not at session-end) and superpowers issue #390 documented a real foot-gun (Stop hook hangs indefinitely on slow LLM calls, wedging the session). The phase-researcher's "layered command-internal + Stop hook" recommendation should be revised to **layered command-internal + `PreToolUse(Edit|Write` matcher on STATE.md + `SessionStart(resume)` dirty-state detection**. The `PreToolUse` hook blocks the exact bad write surgically; the `SessionStart` hook addresses the original "context cleared mid-EXECUTE" failure mode by warning on next resume.

2. **The codebase has SHIP-phase scaffolding gaps independent of E9.** Codebase-researcher found that `commands/ship.md` currently doesn't call `tools/lib/state.js` programmatically at all — the "Update STATE.md to reflect completion" step is prose, not a `transitionPhase(baseDir, 'SHIP')` call. E9 will need to either (a) close this gap as part of S1 (turning prose into a real state write), or (b) scope it out as a separate cleanup. **Recommendation: include the state-write fix in S1.t6 — without it, FR1's hard-block has no programmatic phase-completion signal to gate on.**

3. **Backfill-before-enforcement sequencing is critical.** All three implementation-focused agents converged on this: if S1.t6 (enforcement turn-on) ships before S1.t9–t10 (backfill stubs for E1/E2/E3/E6/E7), the very next SHIP attempt blocks because no retros exist for past Epics. Encoded as S1 task ordering in the PLAN.

**Three tensions worth surfacing to the user before PLAN locks:**

1. **ESCALATE assumption #1 (motivating failure mode):** The assumptions analyzer flagged that SHIP-phase enforcement is structurally orthogonal to the failure mode that motivated this Epic. DISCUSS treated SHIP-phase enforcement as sufficient; research confirms it isn't (no `PreSessionEnd` hook exists in Claude Code's public API). This is a DISCUSS-amendment-level concern — must be surfaced and confirmed before PLAN can lock the mechanism. Surfaced separately to user via AskUserQuestion.

2. **AMEND assumptions #2–4, #6, #8 (current_epic reliability, Epic-ID shape, git-log path filter, FR6 milestone schema, hook AC compatibility):** Five PLAN-level fixes needed before locking. Documented in this file's "Assumption verdicts and PLAN actions" section; each gets a concrete fix in the PLAN.

3. **Cross-runtime story (Open Q #5):** Cursor's hook surface is materially incomplete (no `Stop`, no `UserPromptExpansion`, no `SessionEnd` equivalent). If E9 ships hook-based enforcement, Cursor users get the weaker command-internal-only variant. Codex parity is closer. The phase-researcher's "ship command-internal as primary, hook as defense-in-depth" answer naturally handles this — but it should be explicitly documented as a known limitation in PLAN.

## Section 2 — Claude Code hooks API surface

### 2.1 Full hook event catalog (relevant subset)

Per [Claude Code hooks reference](https://code.claude.com/docs/en/hooks) and [plugins reference](https://code.claude.com/docs/en/plugins-reference):

| Event | Fires when | Can block? | Relevance to E9 |
|---|---|---|---|
| `SessionStart` | New session, resume, `/clear`, post-compact | **No** | Matchers: `startup`, `resume`, `clear`, `compact`. Can inject `additionalContext`. **Resume matcher = key for dirty-state warning.** |
| `SessionEnd` | Session terminates | **No** | Matchers: `logout`, `prompt_input_exit`, `clear`, `resume`. **Observe-only — too late to enforce.** |
| `UserPromptSubmit` | User submits a prompt | **Yes** | 30s timeout. `decision: "block"` with `reason`. Too aggressive for E9. |
| `UserPromptExpansion` | Slash command expands before reaching Claude | **Yes** | Matchers = command name. `command_source: "plugin"` exposed. **Lever to detect `/sig:ship` invocation.** |
| `PreToolUse` | Before any tool call | **Yes** | Matchers = tool name (`Bash`, `Edit`, `Write`). **Most surgical place to block a STATE.md write.** |
| `PostToolUse` | After tool succeeds | **Yes** (stops loop) | `decision: "block"` + `reason`. |
| `Stop` | Claude finishes responding (turn end) | **Yes** | Closest thing to "refuse to let user walk away." **But fires every turn, not at session-end.** |
| `SubagentStop` | Subagent finishes | **Yes** | Matchers = agent type. Not relevant to E9. |
| `PreCompact` | Before context compaction | **Yes** | Matchers: `manual`, `auto`. Could surface retro-warning before context lost. |

**Critical finding:** `SessionEnd` **cannot block**. The only way to "prevent the user from leaving without RETROSPECTIVE.md" is `Stop` (refuses to let the turn end) — but `Stop` fires on *every* turn, not at session-end. **There is no `PreSessionEnd` event in the public API.**

### 2.2 Plugin-distributed hooks — auto-registered

Per the [plugins reference](https://code.claude.com/docs/en/plugins-reference):
- Hooks live at `hooks/hooks.json` inside the plugin root.
- **Auto-registered when the plugin is enabled** — no user-side `~/.claude/settings.json` edits required.
- Available env vars: `${CLAUDE_PLUGIN_ROOT}`, `${CLAUDE_PLUGIN_DATA}`, `${CLAUDE_PROJECT_DIR}`.
- Hook commands receive event JSON on **stdin**, run with full filesystem access in project directory — they **can** read `${CLAUDE_PROJECT_DIR}/.planning/PROFILE.md` for tier-gating.
- **Security restriction:** plugin-shipped *agents* cannot declare hooks/mcpServers/permissionMode. Plugin *hooks themselves* are unrestricted.

### 2.3 Hook return contract & exit codes (gotcha-heavy)

- Exit **0** = success; stdout JSON processed.
- Exit **2** = blocking error; stderr shown; JSON ignored. **Only blocking exit code.**
- Exit **1** = **non-blocking** (counterintuitive — common foot-gun).
- JSON form: `{ "decision": "block", "reason": "..." }` or `hookSpecificOutput.permissionDecision: "deny"` (PreToolUse only).
- Hook stdout capped at **10,000 chars**.
- Hooks run **without a controlling terminal** — no `/dev/tty`.

### 2.4 Cross-runtime story — significant gap

- **Cursor** (v1.7+): only `beforeShellExecution`, `beforeReadFile`, `afterFileEdit` — **no `Stop`, no `UserPromptExpansion`, no `SessionEnd` equivalent**. Known forum bug: `PreToolUse` doesn't fire for built-in web search in Cursor Agent. ([forum.cursor.com bug report](https://forum.cursor.com/t/bug-pretooluse-agent-hooks-do-not-emit-for-built-in-web-search-in-cursor-agent-works-in-claude-code/160761))
- **Codex CLI**: much closer mapping — supports `PreToolUse`, `PostToolUse`, `Stop`, `SubagentStop`, `UserPromptSubmit`, `SessionStart`, `PreCompact`, `PostCompact`. Plugin hooks load from `hooks/hooks.json` similar to Claude Code. ([OpenAI Codex docs](https://developers.openai.com/codex/hooks))
- **Implication:** hook-based enforcement portable to Codex, **not Cursor**. Signal's adapter-layer fallback for Cursor has to be inline command check.

### 2.5 Prior art and gotchas (E9-relevant)

- **obra/superpowers** uses three user-gate hooks writing decision traces. **Known gotcha:** [Issue #390](https://github.com/obra/superpowers/issues/390) — Stop hook hangs indefinitely when Haiku API call times out, wedging Claude Code. **Stop hooks on slow operations are real risk.**
- **`--dangerously-skip-permissions`** disables hooks entirely in Claude Code. D-E9-3 ("hard block, no bypass") is **leaky** here — a user can flag-bypass.
- **Exit 1 ≠ block** — must use exit 2. Common foot-gun.
- **`Stop` fires every turn**, not at session-end. A naive Stop hook would block *every* user message during EXECUTE. Must read STATE.md to know whether current phase is EXECUTE-complete-but-no-RETROSPECTIVE-yet.
- **No `PreSessionEnd`** — original failure mode (context cleared mid-EXECUTE) cannot be solved by hooks alone. SessionStart-on-resume can detect dirty state, but cannot prevent original abandonment.

### 2.6 Recommended mechanism (project-researcher's framing)

The cleanest mapping for E9's stated goal:
- **`PreToolUse` matcher on `Edit|Write` against `STATE.md`** — read proposed update; if it adds SHIP entry to `completed_phases` and `RETROSPECTIVE.md` doesn't exist, exit 2 with explanation. **Surgical, blocks exact bad write, doesn't fire on every turn.**
- **`SessionStart` matcher `resume`** — detect dirty-EXECUTE state from prior session, inject `additionalContext` warning. **Addresses original "context cleared" failure mode.**
- **`Stop` hook** — only if you want to nag *during* EXECUTE that RETROSPECTIVE is missing. **High foot-gun risk per superpowers #390. Recommend NOT including in S1.**

Inline check in `commands/ship.md` stays as Cursor-runtime fallback (option c, layered).

### 2.7 Documented gaps (honest acknowledgment)

- **`--dangerously-skip-permissions` bypass for plugin-distributed hooks** is not explicitly documented vs user-defined hooks; assumed equivalent. Worth a smoke test before locking design.
- **Hook chaining order** when multiple plugins register same matcher is undefined.
- **No documented per-plugin hook disable** — users can disable the plugin entirely, but not selectively disable just its hooks.

## Section 3 — PLAN-item recommendations (9 deferred items)

### 3.1 Backfill mechanism location (item #1)
**Recommended:** Standalone `tools/backfill-retros.js` invoked via `node tools/backfill-retros.js` — no slash command, no `/sig:ship` flag.
**Rationale:** Backfill is one-time historical sweep. Bolting onto `/sig:ship` permanently expands a hot-path command's flag surface for a one-shot event. Separate script also keeps SHIP's hard-block logic uncontaminated.
**Rejected:** `/sig:ship --backfill-retros` flag — couples perpetual command to one-shot operation.

### 3.2 Partial-Epic backfill behavior (item #2)
**Recommended:** Generate stubs for partial Epics (E1, E2) covering shipped slices only, with header note: *"Epic incomplete as of backfill date YYYY-MM-DD. This retro covers shipped slices only (e.g., E1.S1, E1.S2 Phase A). When remaining slices ship, either extend this retro or close it and open a follow-on."*
**Rationale:** Matches FR4 tilt. Deferring stubs until Epic-close leaves E1/E2 invisible in index for months and re-introduces the gap the Epic exists to close.

### 3.3 Index maintenance pattern (item #3)
**Recommended:** Hybrid — `tools/lib/retro-index.js` auto-generates structural listing (file path, Epic ID, stub/complete status); hand-written one-line hook per entry preserved across regenerations by merging on Epic ID.
**Rationale:** Hand-curated hooks are the irreplaceable signal; structural fields rot if hand-maintained.

### 3.4 Per-Epic vs per-Slice SHIP behavior (item #4 — load-bearing)
**Recommended:** FR1 hard-block fires **only on Epic-close SHIP**, not per-Slice SHIP. SHIP detects Epic-close by reading `MILESTONE-{N}.md` for the current Epic and checking whether the slice being shipped is the last unshipped slice (no remaining slices in scope or all others marked shipped/shelved).
**Rationale:** D-E9-5 locks per-Epic granularity; firing per-Slice silently overrides that decision.
**Edge case:** Shelved slices (E1.S3-S5) count as "not remaining" for Epic-close detection — otherwise E1 can never close. Encode shelving as first-class slice status.

### 3.5 Tier-specific minimum-content thresholds (item #5)
**Recommended:** SKETCH ≥ 250B, FEATURE ≥ 1000B, FULL ≥ 2500B. Measure empty template byte size per tier and set threshold = `template_floor + ~150B per required reflection section`.
**Rationale:** REQUIREMENTS rough cuts (200/800/2000) close but don't account for "headings + one-line each" failure mode. Adding ~150B per section forces at least a sentence per heading.

### 3.6 Template section heading exact wording (item #6)
**Recommended (locked):**

### 3.7 Promotion of D-E9-* decisions to DECISIONS.md (item #7)
**Recommended:** S1.t1 promotes all 7 D-E9-* decisions to `DECISIONS.md` as single dated entry (2026-05-25). REQUIREMENTS.md retains summary table but links to DECISIONS.md for authoritative rationale. Tests the convention before backfilling E3's decisions.

### 3.8 Enforcement mechanism (item #8 — load-bearing, revised)
**Recommended (revised from phase-researcher's framing):** **Both, layered.** Command-internal check in `commands/ship.md` is primary (works in all runtimes including Cursor/Codex). Claude Code hooks add defense-in-depth via:

**Conditional caveat:** If hook research finds bugs preventing PreToolUse from reading STATE.md content, fall back to command-internal-only and document cross-runtime gap.

## Section 5 — Surfaced assumptions and PLAN actions

Verdict legend: ACCEPT (defensible — keep as-is), DEFER (acknowledge but don't address now), AMEND (PLAN-level fix), ESCALATE (DISCUSS-level concern).

### A1. SHIP-phase enforcement vs. motivating failure mode — **ESCALATE**

**Assumption:** Hard-blocking at `/sig:ship` prevents context-cleared retro loss.
**Why wrong:** The 2026-05-25 motivating incident was retro skipped *because context cleared before SHIP ran*. SHIP-phase check never fires in that scenario. Section 2 confirms no `PreSessionEnd` hook exists in Claude Code.
**PLAN action:** Surface to user via AskUserQuestion before PLAN locks the mechanism. Recommend revised layered approach: command-internal (Cursor fallback) + `PreToolUse` on STATE.md (Claude Code surgical block) + `SessionStart(resume)` dirty-state detection (Claude Code + Codex addresses original failure mode by warning on next resume).

### A2. `current_epic` reliability — **AMEND**

**Assumption:** STATE.md frontmatter has `current_epic` populated; FR1 reads it.
**Why wrong:** Recommended follow-up #2 in CONTEXT.md notes `last_updated_commit` lags HEAD by one commit. CONTEXT.md says "Active Epic: none" right now — what does FR1 do when STATE has no current Epic but SHIP is invoked?
**PLAN action:** Specify behavior in S1.t6 task spec: if `current_epic` is empty, halt with *"No current_epic in STATE.md. Run `/sig:resume` or set current_epic before invoking `/sig:ship`."* If `current_epic` is set but no `MILESTONE-{N}.md` matches, halt with similar guidance. Tested in AC1-extended.
