## 1. Version-decision rubric (FR5 / AC4)

**Signal's adopted convention.** [semver.org](https://semver.org/spec/v2.0.0.html) is deliberately silent on how `0.x` projects split minor vs. patch, so this is Signal's house rule for the pre-1.0 line. It is the written basis for the v0.1.x → v0.2.0 call (D-E5-4); it does not make the call.

**Pre-1.0 baseline** (already stated in `SECURITY.md`): under `0.x.y`, minor (`0.x.0`) bumps *may* carry breaking changes; patch (`0.x.y`) is non-breaking; only the latest patch line receives security fixes.

**Bump the minor — `0.(x+1).0` — if ANY of:**
- A `[BREAKING]`-class change that **requires user action**: a command removed or renamed, a plugin-manifest shape change strangers must react to, or a `.planning/` schema change **without** transparent auto-migration.
- A **substantial new user-facing capability**: a new workflow phase, or a materially new command surface (judgment call, not a single additive meta-command).

**Otherwise bump the patch — `0.x.(y+1)`:** additive docs, new examples, bug fixes, internal refactors, and schema changes that **auto-migrate with no user action**.

**Worked example — M4.5.E6.** E6 introduced a `[BREAKING]`-tagged `STATE.md` schema change *and* a brand-new command (`/sig:checkpoint`) — yet it shipped as **v0.1.2 (patch)**, not 0.2.0. Why: the schema change **auto-migrated with zero user action** (the "breaking" was absorbed at the tool layer), and `/sig:checkpoint` was an *additive* meta-command, not a change to the core phase surface. The rubric encodes that lesson: for the minor-bump trigger, "breaking" means *user-visible breakage requiring action*, not internal churn that the tooling handles for you.

**This release's call (E4 + E5 cumulative surface).** Both Epics are **purely additive** — a worked example (`examples/url-shortener/`), a comparison page (`docs/vs.md`), and launch docs (`docs/launch-post.md`, `docs/demo-script.md`, `docs/tester-brief.md`). No breaking changes, no new commands, no schema changes. By the rubric → **patch → v0.1.4. Recommended: 0.1.4.** Final call is Brett's (D-E5-4); not pre-committed to either 0.1.4 or 0.2.0.

**Why inline, not `docs/versioning.md` (R1).** `docs/versioning.md` is the **shelved E1 Slice 4 artifact** (D-E3-12). Creating it now would either ship a stale stub or balloon into all of E1's versioning-policy work — scope creep that unshelves a paused Epic. This rubric is the minimal written basis the version call needs. The dangling `CHANGELOG.md` reference to `docs/versioning.md` is **softened in S3.t8** to point at `SECURITY.md`'s pre-1.0 line plus this rubric, rather than fulfilled.

## 4. Distribution channels (R2)

Three channels for a **quiet peer release** — deliberately narrow.

- **GitHub Release** — the durable anchor. The release body (§2) links the post; the tag is the thing everything else points at.
- **Direct 1:1 peer share** — hand the tester brief + Mac caveat to a small number of hand-picked peers (who-to-ask criteria are in the brief, FR4).
- **README surfacing** — a one-line post link at publish time.

**Held back — conscious deferral:** HN / Reddit / X / "Show HN". **Not this release.** A sample-of-one, macOS-only 0.1.x isn't ready for a cold public audience — a bad first impression at scale is expensive and hard to undo. Revisit once the first peers' friction logs are in *and* Linux/WSL install is verified (shelved E1, D-E3-12). This deferral is a decision, not an oversight.
