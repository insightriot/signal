## What changed mid-flight

**D-E9-8 was born at PLAN time, not DISCUSS.** The DISCUSS-phase D-E9-3 ("hard block, no bypass") locked the *strength* of the enforcement but didn't specify the *mechanism*. PLAN-phase research (specifically the assumptions-analyzer agent) surfaced an ESCALATE-level concern: SHIP-phase enforcement is **structurally orthogonal to the motivating failure mode** — the context that motivated the Epic was *the conversation context cleared before `/sig:ship` was invoked*, so a SHIP-phase check would never fire in exactly the scenario it was meant to prevent. Hook-API verification confirmed no `PreSessionEnd` hook exists in Claude Code's public API. The DISCUSS-phase decision was technically intact but architecturally insufficient. The user confirmed the three-layer mechanism via `AskUserQuestion` during the PLAN gate; D-E9-8 was promoted to a decision and the three-mechanism stack landed in S1.t6/t7.

**Scope-by-discovery in S1.t6.** RESEARCH § 1.1 surfaced that `commands/ship.md` had a state-write gap relative to verify.md/review.md — SHIP's "Update STATE.md" was prose, not programmatic. Without fixing that, FR1's hard-block had no clean programmatic phase-completion signal to gate against. The fix bundled into S1.t6 (~30 lines of edits + the new `transitionPhase`/`markFresh` calls in §5). Surfaced explicitly in PLAN § 1.5 ("Scope discipline — PASS-WITH-NOTE") so the broaden wasn't silent drift.

**S2 unchanged.** All seven S2 tasks remain as PLAN'd. S2 has not started yet — this retro covers S1 only.

## What assumptions broke

- **A1 (PLAN-RESEARCH analyzer).** Assumed Claude Code's public hooks API included a `PreSessionEnd` hook. Verified false. Drove the layered-enforcement decision (D-E9-8) — without `PreSessionEnd`, no single mechanism addresses the motivating failure mode, so the answer became defense-in-depth.
- **A4 (PLAN-RESEARCH analyzer).** Assumed `git log` path-filter (`-- 'M4.5.E{N}-*.md'`) would reliably enumerate Epic-related commits. False — most work commits touch `commands/`, `tools/`, `tests/`, not just planning artifacts. Switched to commit-message convention scan (`--grep="^M4.5.E{N}"`). Then surfaced a second bug at S1.t10 dry-run (see below).
- **PLAN spec formula vs. AC (S1.t11).** PLAN specified `threshold = template_floor + 150B × section_count`, but PLAN AC said "minimally-filled template (one sentence per section) passes." Those are incompatible — one sentence is ~50-70B of body; 150B per section pushes the threshold beyond what one sentence can clear. AC was the binding constraint (user-facing behavior); resolved with 60B coefficient; deviation documented inline + in commit message. **Plan-internal inconsistency** — a class of issue VALIDATION's 8-dim check didn't catch.
- **Git-log subject-vs-body matching (S1.t10 dry-run).** Assumed `git log --grep='^M4\.5\.E2'` would match only commits whose subject starts with that pattern. False — `--grep` matches the full message including body. `755dabe` ("M4.5.E7.S1.t10: CHANGELOG...") was wrongly attributed to E2 because its body contained "M4.5.E2 precedent (...)" at line 23. Fixed with a subject-line filter applied after the grep returns; regression test added.

## What surprised us

**The dry-run gate (S1.t10) was the most valuable single task in S1.** It caught two real bugs that would have shipped:

1. The body-vs-subject grep bug, attributing the wrong commits to M4.5.E2.
2. The relative-path issue (`../.planning/X.md` vs sibling `X.md`) — a relic of the M5.E1 wiki-shape assumption that survived into the FR4 stub generator.

Without that gate, both would have landed in the live backfill and propagated to five committed stub files. Modeling the gate on M4.5.E6's D15 precedent paid off again — the pattern is now reusable.

**All 5 backfilled stubs passed `validateRetroContent('FULL')` on first try.** Despite the threshold revision in S1.t11, the auto-populated Links section + `[FILL IN]` markers + heading scaffolding pushed every stub above the 687B FULL floor. This means S1.t6 enforcement won't immediately re-block SHIP for any of the already-shipped M4.5 Epics.

**Hook scripts as direct Node CLIs (deviation from PLAN's bash spec) felt right.** Windows lacks bash by default, and the bash→node wrapper added zero testability — tests directly exercise `checkProposedStateWrite` and `detectDirtyExecute` in JS. The deviation was surfaced + justified inline; the existing `session-start.sh` is preserved as the default-source SessionStart handler so this isn't an audit-relevant break of pattern.

**Test count crossed 400 → 492 over a single day's work.** The cumulative test surface is now ~24% larger than at the start of S1, all unit + integration testing the new helpers + hook surfaces. No flaky tests; all green on every run.

## What we'd do differently

**Run hook-API verification during DISCUSS, not PLAN.** The ESCALATE-level D-E9-8 finding was load-bearing enough that surfacing it earlier would have saved a DISCUSS re-litigation. A 5-minute "what hooks does Claude Code actually expose?" research pass at DISCUSS time would have produced the same outcome with less rework.

**Add a "spec-internal consistency" axis to VALIDATION's 8-dim.** The S1.t11 PLAN formula vs. AC contradiction would have been caught by a "does the formula satisfy the AC?" cross-check. Worth promoting to a standard PLAN-validation dimension.

**Surface DRY-RUN gates as a standard PLAN pattern for any Epic that writes to existing user state.** Two of two M4.5 Epics that used a dry-run gate (E6 D15 and E9 S1.t10) caught real issues. The cost of the gate is small; the benefit is real. Worth codifying.

**Test fixture hashes must be valid hex.** Surfaced when the subject-line filter regression test broke an earlier test using `xyz789` as a fake hash. Real git always emits hex; tests should match. Minor but the regression-test fix had to update fixture data.

## What to feed back into Signal

**Add a DISCUSS question: "Does the proposed enforcement mechanism actually fire in the failure scenario you're trying to prevent?"** The D-E9-8 emergence story generalizes — every new gate should explicitly check motivating-failure-mode-vs-enforcement-locus alignment before locking the mechanism.

**Promote "spec-internal consistency" to a VALIDATION axis.** Specifically: does the formula stated in the PLAN satisfy the AC stated in the PLAN? The S1.t11 issue is the canonical example.

**Codify the dry-run gate pattern.** Any Epic writing to existing user state should default to a dry-run-with-user-approval gate before live writes. M4.5.E6 D15 + M4.5.E9 S1.t10 are both load-bearing precedents.

**Document the "PLAN deviation surfaced + resolved" anti-pattern as a *valid* pattern.** Surfacing deviations in commit bodies (with rationale) is the right move when PLAN turns out to have internal contradictions. The wrong move would be silent compliance with whichever side of the contradiction is more convenient.

**Consider documenting hook output format more thoroughly in `references/`.** I made a choice (JSON `additionalContext` payload) based on inference from existing patterns; a documented reference would speed up future hook authoring.

## Anti-rationalization moment

**S1.t10 was almost merged into S1.t9.** At the PLAN gate I considered folding the dry-run gate into S1.t9's acceptance criteria — "it's the same task, really; just generate and write the stubs." I didn't, because M4.5.E6's D15 precedent specifically separated the dry-run-with-user-review from the live write. The separation was load-bearing: the gate caught the body-vs-subject grep bug AND the relative-path issue, both of which would have shipped under a folded design. **Lesson:** treat gates as first-class tasks, not ACs of a sibling task.

**The PLAN spec's 150B coefficient vs. the AC's "one sentence passes" was almost ignored.** At S1.t11 implementation I noticed the formula (`floor + 150B × sections`) produces thresholds that REJECT a "one sentence per section" retro — exactly what the AC says should pass. The convenient thing would have been to use the formula (it was specified) and let the AC slide. The harder-but-right thing was to surface the contradiction, pick the AC as the binding constraint (user-facing behavior wins over implementation hint), document the deviation, and lower the coefficient to 60B. Tests passed; users won't be punished for one-sentence retros. **Lesson:** when the PLAN's internal pieces contradict, the user-facing constraint wins.

**Hook scripts as bash vs Node.** PLAN literally specified `hooks/check-state-write.sh` (`.sh`). Convenient path: write the bash wrapper, have it `exec node`, technically comply with the spec. Right path: surface the cross-platform fragility (Windows lacks bash), justify the simplification (no test value lost), and ship as direct Node invocation. Surfaced and justified in the commit + PROGRESS.md. **Lesson:** literal compliance with specs is not a virtue when the spec doesn't anticipate the deployment context.
