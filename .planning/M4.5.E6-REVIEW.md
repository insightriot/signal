## Important Issues (should fix)

### IMPORTANT-1 — `captureCheckpointContext` writes to disk *before* sensitive-data scrub is acted on

Both specialists converged on this from different angles.

**Location:** `tools/lib/checkpoint.js:264-309` + `commands/checkpoint.md:88-95`

**What's broken.** `captureCheckpointContext` runs `scrubSensitive` at line 276 to detect potential secrets, then unconditionally appends to CONTEXT.md (line 288), DECISIONS.md (line 295), and OPEN-QUESTIONS.md (line 304), then returns `sensitiveHits` to the caller (line 308). The command-spec at `commands/checkpoint.md:93` claims the user's `abort` response should "roll back the writes" via "file length comparison" but the function returns `wrote: [paths]` only — no byte offsets or pre-image to roll back to. The spec's fallback ("re-run captureCheckpointContext with no decisions/questions is the equivalent of 'discard'") is wrong: line 272–274 returns an empty result when both lists are empty, doing nothing — it does not unwind the prior append.

**Why this is Important not Critical.** `.planning/` is local-only until the user `git push`es; the user can manually trim the appended block; there's no auth/multi-tenant context. But the design promise ("we scrub-detect, you confirm, we write or roll back") is broken at both the function-contract layer and the command-spec layer, and a user pasting an AWS key into a `--context` prompt will see it persisted to git-tracked markdown before the abort prompt fires.

**Smallest fix:** Scrub first, refuse to write if hits are detected unless caller passes `acknowledgeSensitive: true`. This inverts the order to match `tools/lib/add.js`'s precedent (which prompts BEFORE writing).

```js
// In captureCheckpointContext, before any atomicWrite:
const scrub = scrubSensitive([...decisions, ...questions].join('\n'));
if (scrub.hits.length > 0 && !opts.acknowledgeSensitive) {
  return { wrote: [], sensitiveHits: scrub.hits, aborted: 'sensitive-data-pending' };
}
// ...then proceed with the writes.
```

And update `commands/checkpoint.md` § 7 to: (a) call `captureCheckpointContext` with `acknowledgeSensitive: false`; (b) if `aborted === 'sensitive-data-pending'`, run the AskUserQuestion prompt; (c) re-call with `acknowledgeSensitive: true` only on `keep`. Drop the fictional rollback paragraph.

**Estimated cost:** ~15 LOC in `checkpoint.js` + ~10 LOC markdown + 2 new tests (refuse-on-hits, write-on-acknowledge). Total ~30 LOC. Tests stay green (existing call sites pass nothing for `acknowledgeSensitive`; default-false matches "no hits → write proceeds; hits → refuse-and-return").

### IMPORTANT-2 — `dispatchTaskWithState` corrupts the success/failure signal when state-write fails after dispatch

**Location:** `tools/lib/execute.js:84-95`

**What's broken.** The wrapper's success path is `await task.dispatch() → await clearCurrentTask({done})`. If the dispatch succeeds but `clearCurrentTask` throws (lock contention, schema error, disk full), the await re-throws and the same catch-block fires `clearCurrentTask({aborted})` — which will likely fail for the same reason — and re-throws the original (state-write) error. **The caller sees "lock error" and may retry the dispatch, even though the task already completed and committed.** Most Signal tasks are non-idempotent at the commit level (e.g., S1.t1's "extract atomic-write.js" already moved the code).

**Why this is Important not Critical.** The orphan detector (`detectOrphans` + `clearOrphansBeforeDispatch`) is the safety net — the next dispatch will see the stuck entry and abort-clear it. So the user experience is "weird error, then everything's fine on next run." But the contract is wrong: a successful task should not surface as a state-write failure.

**Smallest fix:**

```js
try {
  const result = await task.dispatch();
  try {
    await clearCurrentTask(baseDir, { id: task.id, status: 'done', commit: result?.commit ?? null });
  } catch (clearErr) {
    process.stderr.write(`Signal: clearCurrentTask({done}) failed after successful dispatch of ${task.id} (${clearErr.message}); orphan-cleared on next run.\n`);
  }
  return result;
} catch (err) {
  // Existing aborted path stays unchanged.
}
```

**Estimated cost:** ~8 LOC in `execute.js` + 1 new test. ~12 LOC total.

### IMPORTANT-3 — `appendDecision` is misnamed; it only touches `last_decision_at`

**Location:** `tools/lib/state.js:733-754`

**What's broken.** The name implies "append a decision entry to a list" (matching `addBlocker` + `clearBlocker`'s nomenclature). The implementation only refreshes the `last_decision_at` timestamp — no entry is added anywhere. A future contributor grepping `appendDecision` expecting to find a sibling helper that touches DECISIONS.md will be confused. There's no `decisions[]` field in the schema and no list to append to.

**Smallest fix:** Rename to `touchDecisionTimestamp` (or `markDecisionAt`). Touches `tools/lib/state.js`, `tests/append-decision-mark-fresh.test.js`, `references/state-schema.md`, and any markdown that references the name.

**Estimated cost:** ~10–15 LOC across 4 files. No behavior change.

### IMPORTANT-4 — `STALE_SKIP_GRACE_MS = 60_000` reintroduces wall-clock dependency that D11 was chosen to avoid

**Location:** `tools/lib/state.js:591, 626`

**What's broken.** D11 explicitly chose commit-hash-based staleness to avoid clock skew. The grace window is a wall-clock optimization — "if `last_updated` is < 60s ago, skip the git call." On a CI runner with clock drift (or a fresh VM with an out-of-date clock), this misfires: either skipping a legitimate staleness check, or flagging an old write as recent.

**Smallest fix:** Replace the wall-clock comparison with a hash compare. If `last_updated_commit === <current HEAD>`, skip the git call. Same optimization intent, no clock dependency.

```js
// Replace lines 622-629 with:
if (!opts.bypassGrace && lastCommit) {
  const head = resolveHeadCommit(baseDir, execFn);
  if (head && head === lastCommit) return empty;
}
```

**Estimated cost:** ~5 LOC + adjust one test that asserts the grace window. ~8 LOC.

### IMPORTANT-5 — `commands/checkpoint.md` § 7 rollback paragraph promises behavior that isn't implemented

**Location:** `commands/checkpoint.md:93`

**What's broken.** "Roll back the writes" is described as "straightforward by file length comparison" and the fallback claim ("re-run with no decisions/questions is the equivalent of 'discard'") is incorrect — `captureCheckpointContext` returns empty on empty input (no rollback action).

**Smallest fix:** Folded into IMPORTANT-1 above. Once the function refuses-to-write on unconfirmed sensitive hits, the markdown § 7 simplifies to: "If sensitive hits are detected, prompt; on `abort`, do nothing (no writes happened). On `keep`, re-call with `acknowledgeSensitive: true`."

## Performance Findings

This is internal Node tooling on small files — performance is bounded by design.

- **No N+1, no pagination, no bundle, no re-renders** — N/A.
- **Lock contention.** `withStateLock` serializes STATE.md writes. With per-task atomic-commit discipline, lock holds are millisecond-shape. The 5s TTL is generous. In wave-parallel execution this could become a contention point but not a correctness issue (FIFO with retry-on-fail at the caller). No blocker.
- **`isStateStale` grace window** — see IMPORTANT-4. The 60s wall-clock optimization works *most* of the time but is the wrong primitive given D11's hash-based contract.
- **Regex on large files.** `FRONTMATTER_RE` uses `[\s\S]*?` (non-greedy) — fine for STATE.md (<100KB). No catastrophic backtracking surface for the typical input shape.
- **No findings rising to Important or Critical.**
