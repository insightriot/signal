## What went well

- **DISCUSS gates were high-signal.** The four product-level gray areas (example scope / source / form / vs.md format) were exactly the right things to gate; each answer cascaded cleanly into PLAN. The pivotal DISCUSS insight — that an init-only example *stops where the calibration wedge begins* — reshaped the whole Epic for the better.
- **Research was decisive and de-risking.** The 4-agent PLAN pass turned three unknowns into settled facts before EXECUTE: drift was one file (not the feared full regeneration), the guard needed `_schema === 1` (not just "doesn't throw"), and vs.md's accuracy traps were catalogued up front.
- **The currency guard is the durable win.** The `.dogfood/` copy rotted *because* it was untracked; a tracked example + a test that fails on schema drift is the structural fix, not a one-time cleanup.

## What surprised us (and the pivots)

- **Research's top R1 fix didn't survive contact with the toolchain.** PLAN recommended swapping `better-sqlite3` → Node's built-in `node:sqlite`. In EXECUTE it failed immediately: `node:sqlite` is so new that vitest/vite's bundler can't resolve the experimental built-in, and 3 suites wouldn't even collect. The researcher had tested `require('node:sqlite')` in plain Node — not through the test runner's transform pipeline. **Lesson:** "verified the API works" ≠ "verified it works under our tooling." The fallback (the research's documented second choice — a plain JSON-file store) was actually *cleaner* for a showcase: zero deps, no experimental flag, no bundler config, and already within the example's own `PROJECT.md` scope ("SQLite or a JSON file"). Net: the pivot improved the outcome.
- **The example's own paperwork didn't add up.** REQUIREMENTS said "16 testable," VALIDATION/VERIFICATION/SHIP said "18 automated + 6 manual = 24" — but VERIFICATION's own line-by-line table showed **17 + 7 = 24**. Reconciled all summaries to the table-derived truth (the line-by-line evidence was correct; only the tallies miscounted). The 38-vs-39 test count turned out *not* to be an error — VERIFY ran 38, REVIEW added one. **Lesson:** trust the most granular artifact (the line-by-line table) as ground truth; verify before "fixing."

## Process notes

- **Tone matters on positioning docs.** The first `vs.md` draft read as subtly combative (each tool section ended in "but Signal…"). User reframed it around the hammer/screwdriver/drill → toolbox → house metaphor: respect each tool, name the real gap (*assembling them yourself is work*), invite "use one directly if that's all you need," and land Signal as the toolbox + calibration. The rewrite is strictly better and is now the house style for comparison framing.
- **Gate at product altitude.** Mid-PLAN, surfacing the SQLite-library + Node-floor choice as a user decision lost the user ("DEEP in some rabbit hole"). Correction saved as a durable working norm ([[feedback_gate-at-product-altitude]]): under FULL/strict, gate only product/scope/positioning decisions; decide plumbing/tooling/test-mechanics yourself with a stated default. Applied for the rest of the Epic.
- **Lightweight close (user decision).** FULL tier nominally runs VERIFY → REVIEW → SHIP. Because EXECUTE verified all 9 ACs inline with test evidence and the code surface is ~40 lines (a JSON store + a guard), the user chose to fold the remaining phases into this close rather than run formal ceremony, and to leave the work `[Unreleased]` to batch with E5 (the natural release moment; E4 was its prerequisite). The example + vs.md are already visible on `main`/GitHub, so no tag was needed to make them useful.
