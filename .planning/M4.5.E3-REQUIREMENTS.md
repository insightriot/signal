### FR3 — CONTRIBUTING.md

Strangers filing bugs or PRs need a one-page guide.

**Acceptance:**
- New file `CONTRIBUTING.md` at repo root, ~80–120 lines, sections:
  - **Filing bugs** — link to issue template; what info to include (Signal version, Claude Code version, OS, reproducer); link to install-troubleshooting.md for known issues.
  - **Proposing changes** — fork → branch → PR; what a good PR description looks like; link to `.planning/` for the "why we work this way" context.
  - **Local development** — `git clone`, `npm install`, `node tools/validate-plugin.js`, `npm test` (currently 384 tests); `export CLAUDE_PLUGIN_ROOT="$(pwd)"`.
  - **`.planning/` ground rule** — every Signal repo (including this one) keeps `.planning/` tracked in git. PRs that add it to `.gitignore` will be closed; the directory IS the project's memory.
  - **Code style** — no enforced linter today; match surrounding patterns; tests use vitest; one runtime dependency (`yaml`) — don't add more without discussion.
  - **Release process (informational)** — brief pointer that releases are tagged after Epics close; full versioning policy lives in `docs/versioning.md` (E1.S4 deliverable).
  - **Code of conduct** — pointer to a forthcoming `CODE_OF_CONDUCT.md` OR inline 3-line "be kind, assume good faith, no harassment" stanza. Decided per D-E3-7.
- Linked from README's existing footer area (new line near License).
- Linked from GitHub's auto-detection (sits at repo root so GitHub renders the "Contributing" tab).

### FR4 — SECURITY.md

OSS plugins handling user code need a reporting channel. GitHub's security advisory UI surfaces SECURITY.md.

**Acceptance:**
- New file `SECURITY.md` at repo root, ~30–50 lines, sections:
  - **Supported versions** — table showing only the latest `0.1.x` patch line as supported. Older patch versions: "not supported — upgrade to latest." Pre-1.0 caveat: breaking changes can occur at minor bumps per `docs/versioning.md`.
  - **Reporting a vulnerability** — private email to `brett@insightriot.com` (subject line convention: `[security] Signal: <one-line>`); alternative: GitHub private vulnerability report via the security tab.
  - **Response posture** — best-effort solo-maintainer; ack within ~72 hours; remediation timeline depends on severity (no SLA at pre-1.0); no embargo policy at this scale.
  - **Disclosure** — fixed issues will be noted in CHANGELOG.md with the version that contains the fix; pre-disclosure coordination happens via email if requested.
  - **Scope** — what counts (plugin code that runs locally; markdown command files that emit shell commands; tools/ libs that read user files); what doesn't (Claude Code itself — that goes to Anthropic; the user's own project code).
- Linked from README's existing footer area + CONTRIBUTING.md "Filing bugs" section.
