## What changed mid-flight

- **The plan's scope shrank on contact with the ledger.** PLAN research found the "12 known bugs" were really ~9 code fixes + reconciliation: B18 was *already fixed* (M5.E3 `8f1dda8`), B8's both halves shipped v0.1.8, and B19's fix was *shipped-but-holed*. Several "fixes" became residual tests + ledger flips.
- **B15 advisory-scanner treatment.** The plan said "flag-on-truncation" for the 3 advisory scanners; on inspection that forced a return-shape reshape across consumers, so I made only the *blocking* scanner full-file (the AC's actual scope) and kept the advisory cap — a deliberate, documented deviation from the plan's letter.
- **`add.js` reversal ratified mid-EXECUTE.** B14's add.js site overturned a *documented* "symlinks out of scope / no I/O" design decision; surfaced it, Brett ratified (D-M5E4-5), docstring rewritten.
- **B24 discovered.** A T4.1 coverage test surfaced a new P2 (pre-existing dangle in a closed DECISIONS section blocks migrate) — scoped, recommended defer, Brett approved → v0.1.10.

## What assumptions broke

- **"The shipped B19 fix works."** It didn't — the skip predicate keyed on `ann.legend === null`, but the old curated format *carries* the legend block, so it never fired on the real repro. Proven empirically. The Epic's flagship fix was itself a bug.
- **"The B14 fix is done + tested."** The delegated executor's guard passed the *directory* to the confinement check, resolving only the archive parent; a directory symlink at the Epic *leaf* still escaped. The test planted its symlink one level too high, so it read green. **A second false-green, in the fix meant to close a path-confinement gap.**
- **"B23(a) is a straightforward rollback test."** The literal AC ("reroot makes a resolving link dangle") is *unreachable* — the reroot is resolution-preserving. The executor proved it + surfaced the adjacent real behavior + B24.
- **"Executors are the fast path for EXECUTE."** The first executor took ~36 min/task (mostly rediscovery + repeated full-suite runs). At that rate 18 tasks ≈ 10 hours; I switched to direct implementation for the mechanical tasks and reserved executors for the large/isolated/gnarly slices.

## What surprised us

- **Two independent false-green tests in one Epic** — B19's predicate (found in PLAN research) and B14's evict test (found in REVIEW) — both the *exact* failure the Epic was closing. A green suite gave false confidence twice; only adversarial verification (RED-against-`main`; the 3-specialist panel with a PoC) caught them. This is the single most important signal of the Epic.
- **The REVIEW panel's ROI was concrete, not ceremonial.** Two independent reviewers reproduced the B14 evict bypass with a working PoC; the security pass also *cleared* real concerns (command-injection actually improved via B6, ReDoS, re-entrancy) and confirmed `resume.js`'s fail-open holds. The one Important it surfaced was a genuine (bounded) security escape.
- **Reverse-verification paid off.** For B21 and B15 I restored the bug to confirm the test actually fails — cheap, and it's the discipline that would have caught both false-greens earlier had it been applied to the security tests too.

## What we'd do differently

- **Treat every guard/security test as guilty until proven RED against the exact vulnerable shape.** Both false-greens shared a root cause: the RED test exercised a *near* case, not the precise one the code skips. A guard test must plant the hostile artifact at the *exact* location the guard is responsible for (the leaf, not the parent), and be run against unfixed code.
- **Verify delegated security fixes by reproducing the bypass, not by trusting the returned green.** The B14 executor reported "RED-first, all sites hardened" in good faith; the test genuinely passed — but against the wrong shape. Orchestrator verification should re-run the *attack*, not just read the diff + the pass count.
- **Reconcile the bug ledger continuously, not only at FR7.** The B18/B8/B19 ledger-lag (fixed-but-not-flipped, one of them a false-green) cost real PLAN-research time to untangle. Flipping `BUGS.md` at each fix would have surfaced the B19 false-green sooner.

## What to feed back into Signal

- **A "false-green audit" for guard/security tests.** Signal's REVIEW should specifically ask, for every new test asserting a guard *refuses* something: "does this test plant the hostile input at the precise location the guard covers, and would it fail against the unfixed code?" Both this Epic's misses would have been caught by that one question. Candidate: a `test-engineer`/`security-auditor` checklist item, or a `references/` note on "false-green smells" (a RED test that plants the attack one level off the vulnerable site).
- **RED-against-`main` should be the standard for guard fixes, not a per-Epic discipline.** The B19 fix only got a genuine RED test because the false-green history made me paranoid. That paranoia should be encoded (EXECUTE guidance: a fix that closes a security/guard gap must include a test proven red on the pre-fix commit).
- **SHIP-time ledger-reconcile check.** A hygiene test asserting `BUGS.md` has no `confirmed` bug whose described fix already shipped (grep the fix commit) would catch ledger-lag automatically.
- **Executor cost calibration.** A ~30-min/task executor floor makes pure-sequential dispatch impractical for large Epics; Signal's EXECUTE guidance could note the direct-vs-delegate tradeoff (delegate the large/isolated/gnarly; the orchestrator does mechanical tasks directly to avoid round-trip latency) rather than implying uniform delegation.

## Anti-rationalization moment

**Two.** (1) **B24 — I almost defaulted to a neutral "fix-now / defer" menu.** The advisor pushed: don't present it neutrally, *scope the fix and recommend*. Timeboxing the scope revealed the correct fix reworks the load-bearing dangling-delta gate all migrate paths depend on (the naive version *weakens* the gate) — so "defer to v0.1.10, keep the just-hardened gate stable" became a justified recommendation, not a shrug. Presenting it with the real fix-size changed it from a punt into an informed call. (2) **The B14 evict false-green — I almost shipped a green security test as proof.** I *had* spot-checked the FR5 read-enclosure by inspection (correctly), but I trusted the delegated B14 executor's green test without reproducing the bypass. Only because I ran the full 3-specialist REVIEW panel — rather than rationalizing "VERIFY already passed, REVIEW is redundant for a bug-fix Epic" — did the leaf-symlink escape surface before the tag. The temptation to skip REVIEW on a "it's just bug fixes" Epic was real; keeping it caught a real security bug.
