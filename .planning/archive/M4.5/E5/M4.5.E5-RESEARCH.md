## 4. External norms (prior-art researcher)

- **Semver 0.x rubric (label as "Signal's adopted convention," not "per semver.org" — semver is silent on the 0.x minor/patch split):** ship **0.2.0** if any of — new user-facing capability (command/agent/skill/flag/phase behavior) · behavior/contract change (output shape, defaults, `PROFILE.md`/`STATE.md` schema — even with auto-migration) · removal/rename · a release notable enough to write *about*. Otherwise **patch** (0.1.4): fixes, docs, install-UX hardening, test-only, behavior-identical refactors. Reserve **1.0.0** for "people depend on this in production." **Worked example for the rubric:** the M4.5.E6 `STATE.md` schema migration is exactly a contract change that justifies a minor even with auto-migration.
- **Launch-post arc (~700 words):** hook + one-line value prop (40–60) → problem (120–160) → short backstory/credibility (60–90) → what it is (150–200) → how it's different / the wedge (100–140) → honest limits (40–60) → CTA, free + no-signup, "curious what breaks for you" (50–70). Builder-to-builder voice, no superlatives, zero signup friction.
- **Friction-log template:** per-row — step/where they paused · expected · actual · confusion (verbatim quote) · severity (1 minor / 2 major / 3 critical, Nielsen-style) · recovered? Plus header (tester context, task, version) and a "one thing they expected that didn't exist" closer. Rate severity separately from frequency; capture verbatim words.
- **Soft-launch:** set the "0.1.x, early, sample-of-one, expect rough edges" frame up front; invite a small named set individually; make the ask small and specific ("20 min, one project through calibrate→discuss, log where you got stuck"); validate the calibration premise resonates *before* going wide; close the loop with every tester (what you fixed + what you didn't and why).
