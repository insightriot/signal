## TL;DR — three findings that change the PLAN

1. **The synthesizer is the LLM, not JS.** No JS code path writes LANDSCAPE.md or baseline PROJECT.md. `tools/lib/landscape.js` helpers (`readScan`, `readAllScans`, `extractSection`, `extractField`) are read/extract-only and mechanically cannot produce interior character drops. The bug is LLM transcription failure during `/sig:init` Steps 3–4. This invalidates one of D-E7-1's three hypotheses (JS regex bug) and reshapes "tests-first" away from output-byte-equality.

2. **Test architecture is three-layer — not "regression tests against LLM output."** Layer B (deterministic pipeline tests) covers patterns 1/2/3 by testing the seams the LLM operates against. Layer C (prompt-template linting) covers patterns 4/5/6 as shape properties, not exact-match. Layer A captures the buggy run as a fixture for both. **Building a new `embedSection` JS helper is part of the fix for pattern 3** — it pulls the verbatim-embed work out of LLM discretion and into deterministic code that tests can pin.

3. **S2's docs need substantive corrections, not just transcription of R1.** P1 is officially-documented Claude Code behavior (cite plugins-reference). P2 has a canonical CLI fix (`claude plugin uninstall sig --scope user`) the R1 maintainer didn't try. Two new symptoms emerged: pre-rename `signal@signal` cache orphan (Signal-specific historic), and the v0.1.1 SSH-clone fix (cross-link, not re-document). S2 grows from 3 symptoms to 5 + Quick Triage + Canonical Clean Reinstall sequence + Reference table.

## 1. Pipeline map: where synthesis actually happens

From codebase-researcher (full report sources: `commands/init.md`, `tools/lib/landscape.js`, `agents/scanners/*.md`).

```
/sig:init (LLM driver)
   │
   ├── Step 2: Task tool spawns 4 sub-agents (LLM each) ─→ writes .planning/scan/{stack,structure,activity,quality}.md
   │        (Each sub-agent reads repo with Read/Bash/Grep/Glob; outputs markdown per its template.)
   │
   ├── Step 3: LLM calls readAllScans() ─→ returns Map<name, content>
   │        LLM fills LANDSCAPE.md template (commands/init.md:188-250).
   │        Sections 2-6: "mechanical reformat" using extractSection / extractField (LLM may call OR paraphrase).
   │        Sections 1, 7: narrative synthesis.
   │        LLM writes LANDSCAPE.md with the Write tool.
   │
   ├── Step 4: LLM drafts PROJECT.md from LANDSCAPE.md + scan data
   │        Template at commands/init.md:267-319.
   │        LLM writes PROJECT.md with the Write tool.
   │
   └── Steps 5, 6: walkthrough + STATE.md init (out of scope for E7)
```

**Locus of synthesis:** the LLM is the **sole prose generator** for both files. `grep` for `writeFile.*landscape` and `writeFile.*PROJECT\.md` in `tools/` returns zero hits. Every character that lands in the output files passes through the LLM's text-generation pass.

**`landscape.js` involvement (ruled out as bug source):**

- `readScan(baseDir, name)` — file read, returns content or `null`. No string mutation.
- `readAllScans(baseDir)` — loops the 4 scanners, calls `readScan`. No string mutation.
- `extractSection(content, heading)` — regex extract of h2-section body, returns substring `.trim()`'d. Cannot drop interior chars; either matches and returns the full body or returns `null`.
- `extractField(content, label)` — normalizes `**bold**`, regex-matches `Label: value`, returns captured value `.trim()`'d. Same — pure-extract.

**Bug-surface ranking (codebase-researcher, very high confidence):**
1. LLM output during Step 3/Step 4 synthesis — canonical LLM transcription failure mode.
2. Scanner output already corrupted — low. Would require all 4 scanners to drop chars in the same statistical pattern; not observed in R1.
3. `landscape.js` helpers — extremely low. Effectively ruled out by mechanism.

## 3. Test architecture (3 layers)

From phase-researcher. Adopted as E7's test strategy.

### Layer A — Captured fixture (the "red" anchor)

Path: `tests/fixtures/synthesizer-bug-r1/`
```
scan/
  stack.md
  structure.md
  activity.md
  quality.md
expected/
  LANDSCAPE.md     # hand-corrected version of the buggy output
  PROJECT.md       # same
actual/
  LANDSCAPE.md     # the verbatim buggy output from R1 — documentary
  PROJECT.md       # same
```

The buggy `actual/` artifacts go in too, as documentation. The clean `expected/` versions are what the synthesizer SHOULD produce.

### Layer B — Deterministic pipeline tests (patterns 1, 2, 3 + siblings)

Three test types:
- **Template-literal tests.** Parse `commands/init.md` for h2 headings; assert each literal heading is well-formed (no truncation if echoed verbatim).
- **Embed-verbatim tests.** Test the new `embedSection(scan, heading)` helper (proposed; doesn't yet exist) which returns the bytes the LLM is supposed to copy. Assert table-cell content (e.g., `index.js`) preserved exactly.
- **Round-trip tests.** Run `extractSection`/`extractField` against the clean fixture LANDSCAPE.md; assert every section is recoverable. If a real run's output can't round-trip, it's corrupted.

Estimated 8–10 tests.

### Layer C — Prompt-template linting (patterns 4, 5, 6 + free-form siblings)

Lint `commands/init.md` template directly:
- No single template line exceeds N chars (long lines correlate with LLM truncation).
- All `{placeholder}` markers well-formed (no nested braces).
- No instruction spans a sentence-and-fence boundary without an explicit newline.
- Heading template strings all ≥ 8 chars (catches `## ints`, `## Ierred`).

Plus shape-property assertions against fixture output:
- No h2 heading in generated LANDSCAPE.md / PROJECT.md is < 8 chars.
- No table cell in identifier columns < 3 chars.
- No fenced-code block starts mid-line.

Estimated 5–6 tests.

**Total new tests: 13–16.** Sits at the high end of the requirements' 10–14 estimate; the overshoot is justified by the layer separation.

### What this architecture cannot catch
- Novel LLM regressions on prompt regions without fixture coverage.
- Subtle semantic drift (e.g., wrong project description).
- Model-version-induced quality shifts.

**Mitigation:** FR3 (`expressjs/express` rerun, human-in-loop) remains the final check.

## 4. Install troubleshooting — corrections from R1

From project-researcher (full report sourced from `/Users/macstudio/.claude/plugins/installed_plugins.json`, `/Users/macstudio/.claude/settings.json`, `/Users/macstudio/.claude/plugins/cache/`, Claude Code official plugins-reference docs).

### P1 — corrected understanding

**Not a bug. Officially documented behavior.** Per plugins-reference: *"Users get updates only when you bump this field. Pushing new commits without bumping it has no effect, and `/plugin update` reports 'already at the latest version'."* The `version` field in `installed_plugins.json` IS the gate; `gitCommitSha` is informational. Workaround framing shifts from "bug workaround" to "operational pattern."

### P2 — major correction

**Canonical fix is `claude plugin uninstall sig --scope user` (CLI verb).** The R1 maintainer never tried the CLI. The `/plugin` interactive UI lacks Uninstall (confirmed), but the CLI has it (Claude Code v2.1.121+). Per `claude plugin uninstall --help`:
- Removes plugin from named scope (user default).
- `--keep-data` preserves data dir.
- `--prune` removes orphaned deps.
- Docs line 819: "uninstalling from the last remaining scope also deletes the plugin's `${CLAUDE_PLUGIN_DATA}` directory."

Filesystem purge becomes the **fallback**, not the primary fix.

### P3 — stands as documented

**Real Claude Code papercut.** `enabledPlugins` in `~/.claude/settings.json` persists after `claude plugin uninstall`. Verified via direct file read:

```json
"enabledPlugins": {
  "sig@signal": true,
  ...
}
```

Workaround: hand-edit `settings.json` between uninstall and reinstall. Worth filing upstream after S2 ships.

### Cache directory shape (verified)

`~/.claude/plugins/cache/<marketplace>/<plugin>/<version>/`

For Signal: `~/.claude/plugins/cache/signal/sig/0.1.2/` (live), with `0.1.1/` and `0.1.0/` as orphans within the 7-day grace window per docs line 659.

**Pre-rename orphan discovered:** `~/.claude/plugins/cache/signal/signal/0.1.0/` exists from the pre-M4.t19 era when the plugin was named `signal` not `sig`. This is a Signal-specific historic landmine — only affects users who installed pre-rename and upgraded. **New symptom for S2 (Symptom 4).**

### Peer plugin patterns

- **GSD** — two-tier: short README troubleshooting + `docs/USER-GUIDE.md#troubleshooting` for depth.
- **Agent Skills** — no dedicated troubleshooting doc; inline notes in setup guides.
- **Signal** — dedicated `docs/install-troubleshooting.md` matches GSD's depth tier; sits alongside Signal's existing `docs/install-verification.md`.

|---|---|---|

| 6 | `contributoiteria.` | n/a — synthesizer didn't generate the source phrase |

Three things changed between R1 (2026-05-19) and S1.t1 re-repro (2026-05-22):

1. **Plugin version.** R1 ran against v0.1.2 (commit `9f504a4`); re-repro ran against `015525e` (M4.5.E7 DISCUSS + PLAN shipped). The init.md template body is unchanged across this delta — diff is purely `.planning/` artifacts, no `commands/` touch.
2. **Machine + identity.** R1 ran on the biz box (Brett's day-job machine, Claude Code 2.1.144). Re-repro ran on Mac Studio (Brett's primary dev box).
3. **Observer effect, possible.** Re-repro session has the M4.5.E7 plan loaded in context, including the 6 R1 patterns the synthesis is meant to surface. This may have induced more careful generation than the R1 run had — the bug-is-known state is itself a confounding variable.

**Implications for the test strategy.**

- The 3-layer architecture's premise — that on-disk fixtures replace live LLM captures — is **validated**, not invalidated. Layer B / Layer C tests still work because they assert against deterministic fixtures, not live model output. The bug's non-reproduction in re-repro is exactly the failure mode the architecture was designed for.
- The Layer A fixture must now be **synthetic** rather than captured. Per Brett's S1.t1 escalation decision (Option A — "Synthetic actual/ from R1 snippets"), `tests/fixtures/synthesizer-bug-r1/actual/` is constructed by taking the clean re-repro output as the structural base and manually injecting the 6 R1 patterns at the documented locations. See `tests/fixtures/synthesizer-bug-r1/README.md` § Provenance for the per-pattern injection record.
- **FR1 still holds.** "Synthesizer regression tests cover all 6 documented bug patterns" remains satisfiable — the patterns are documented (`docs/install-verification.md` § R1), the fixture records the buggy → clean diff per pattern, and Layer B + Layer C tests will assert RED→GREEN against deterministic fixtures.
- **FR3 weakens slightly.** "Synthesizer rerun on `expressjs/express` produces clean output" is technically already satisfied by the S1.t1 re-repro (which was already clean). S1.t9's value shifts from "verify the fix works against the original bug surface" to "verify the fix did not introduce new regressions." Still useful, but the assertion is one-sided (no R1-style patterns must appear) rather than two-sided (R1 patterns must disappear AND no new ones appear).
- **No PLAN refinements required.** S1.t2 → S1.t10 are unchanged. The synthetic-injection step is folded into S1.t1's fixture-creation work (already done as of this addendum).

The clean re-repro artifacts live at `/tmp/express-repro-e7/.planning/{LANDSCAPE.md,PROJECT.md,scan/}` for the duration of S1; the `scan/` files are also copied verbatim to `tests/fixtures/synthesizer-bug-r1/scan/` so the fixture is hermetic against /tmp/ cleanup.
