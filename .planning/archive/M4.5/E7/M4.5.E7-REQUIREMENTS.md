## Functional Requirements

### FR1 — Synthesizer regression tests cover all 6 documented bug patterns

The fix is verified by tests, not by inspection. Before any fix is attempted, regression tests exist for each of the 6 R1-documented character-drop patterns. Tests fail on `main` at the start of S1 (red), pass after the fix (green), and stay green on subsequent runs.

**Source patterns** (from `docs/install-verification.md` § R1, 2026-05-19 Express dogfood):

1. Heading-boundary drop: `## Ierred goals & uncertainties` ← should be `## Inferred goals & uncertainties`
2. Heading-boundary drop: `## ints` ← should be `## Constraints`
3. Table-cell drop: `is | Top-level entry (224 bytes)` ← should be `index.js | Top-level entry (224 bytes)`
4. Mid-flow drop in fenced code / command: `--checkt/ test/acceptance/` ← should be `--check-leaks test/ test/acceptance/`
5. Concatenated sentence boundary: `...not real cadence).git fetch --unshallow\`` ← missing newline + backtick before remediation prose
6. Mid-sentence drop in PROJECT.md Notes: `Constraints (Team / contributoiteria.` ← truncated mid-word + missing closing context

**Acceptance:**
- One regression test per pattern (6 minimum), each asserting the corrected expected output.
- Each test failable independently (no shared fixture dependency that would mask single-pattern regressions).
- Test names mirror the symptom (e.g., `synthesizer drops chars at heading boundary — issue-E7-pattern-1`) for grep-ability.

### FR2 — Sibling-case regression coverage

Beyond the 6 documented patterns, S1 adds tests for likely-sibling failure surfaces that the same root cause would predict:

- Heading boundaries (other section transitions in LANDSCAPE.md template)
- Table-cell boundaries (other table-row positions in Tech-stack/Activity-signals output)
- Sentence joins (paragraph boundaries in synthesized prose)
- Fenced-code block boundaries (start/end of ```` ``` ```` blocks)

**Acceptance:**
- 4–8 additional tests covering boundary variants.
- If S1's root-cause investigation reveals additional failure surfaces, add tests for those too.
- Total new tests: ~10–14 (366 → ~380).

### FR3 — Synthesizer rerun on `expressjs/express` produces clean output

After the fix lands, S1's verification step reruns `/sig:init` on a fresh shallow clone of `expressjs/express` (the original R1 reproduction surface). Output is inspected for **any** character-drop pattern in the resulting LANDSCAPE.md + baseline PROJECT.md. None of the 6 documented patterns reappear; no new pattern surfaces.

**Acceptance:**
- LANDSCAPE.md headings parse as well-formed h2's with full intended content.
- Table cells contain expected filenames/values, not single-character or truncated strings.
- Code blocks render command-line snippets without mid-flag drops.
- PROJECT.md Notes section contains complete sentences (no mid-word truncation).
- The clean-output evidence is captured in `docs/install-verification.md` § R2 (new row) or appended as supplement to § R1.

### FR4 — `docs/install-troubleshooting.md` documents the 3 install-UX papercuts

New file at `docs/install-troubleshooting.md`. Symptom-organized layout — strangers find their fix by searching the failure mode they see, not by reading sequentially.

**Sections (one per papercut):**

- **P1 — `/plugin install` reports "already at latest" but cached code is stale.**
  - Symptom (exact UI text + observed behavior)
  - Root cause (Claude Code's `installed_plugins.json` version-field short-circuit ignoring stale `gitCommitSha`)
  - Workaround (filesystem purge before reinstall — exact commands)
- **P2 — `/plugin` interactive menu has no Uninstall verb.**
  - Symptom (only Disable shown; cache dir survives)
  - Root cause (interactive menu UX gap; cache + `installed_plugins.json` not touched by Disable)
  - Workaround (manual `rm -rf ~/.claude/plugins/cache/<marketplace>/` + edit `installed_plugins.json` to remove dangling entry — exact commands)
- **P3 — Plugin stays disabled after uninstall + reinstall.**
  - Symptom (commands don't register on reinstall; no error)
  - Root cause (Disable state lives in `~/.claude/settings.json` under `enabledPlugins`; uninstall doesn't touch it)
  - Workaround (manually remove `<plugin>@<marketplace>` line from `settings.json` before reinstall — exact commands)

**Acceptance:**
- All 3 papercuts have a Symptom / Root cause / Workaround stanza.
- Workarounds are copy-paste-able (commands, not prose descriptions).
- Page is linked from README's existing Troubleshooting section (single new line).
- CHANGELOG entry under the patch-release header references the new file.
