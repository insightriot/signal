## Context — what E8 is solving

Three of the five documented install failure modes (P1 stale `gitCommitSha` short-circuit, P2 no Uninstall verb in `/plugin` UI, P3 Disabled state survives reinstall) are upstream Claude Code plugin-host bugs, not Signal bugs. The fourth (pre-rename `signal@signal` cache orphan) is half historical-Signal / half cache-GC. The fifth (SSH multi-identity) is environmental.

Today, a stranger upgrading Signal has to read a 280-line troubleshooting doc, self-triage which P-state they're in, and execute a 4-step canonical clean reinstall — possibly preceded by manually editing `~/.claude/settings.json` and `rm -rf`-ing a cache directory. That is not a plugin upgrade story.

E8 ships **`/sig:doctor`** — one command that detects which install-state failure mode applies, generates a remediation script the user reviews and runs, and surfaces version staleness via `/sig:status`. It also reframes `docs/install-troubleshooting.md` to honestly name upstream-vs-Signal ownership.

## Functional Requirements

### FR1 — Detect 5 documented P-states

`/sig:doctor` reads `~/.claude/settings.json`, `~/.claude/plugins/installed_plugins.json`, and `~/.claude/plugins/cache/<marketplace>/` directory contents to detect:

1. **P1** — stale `gitCommitSha` in `installed_plugins.json` vs. cache directory contents
2. **P2** — orphan plugin entry in cache without corresponding `installed_plugins.json` entry (Uninstall verb missing → cache not cleaned)
3. **P3** — entry in `~/.claude/settings.json` `enabledPlugins` (or equivalent disabled-list location) that doesn't match a currently-installed plugin
4. **Pre-rename `signal@signal`** — historical marketplace slug present in cache or settings
5. **SSH multi-identity** — environmental detection (best-effort `~/.ssh/config` inspection for `github.com-*` Host entries that might misroute)

**Acceptance:** detector returns a structured findings list with one entry per detected P-state plus a `healthy: true|false` boolean.

### FR2 — Read-only default mode

`/sig:doctor` with no flags performs detection only. **No writes, no mutations.** Output is diagnostic text printed to stdout. Exit code 0 if healthy, 1 if any P-state detected (so it can be scripted in CI / pre-commit hooks).

### FR3 — `--fix` generates surgical remediation script

`/sig:doctor --fix` runs detection, then writes `~/.claude/sig-doctor.sh` containing remediation commands **only for the detected P-states** (not a full reinstall). Prints `Generated ~/.claude/sig-doctor.sh — review then run: bash ~/.claude/sig-doctor.sh` and exits. Does not execute the script itself.

If no P-states are detected, `--fix` prints `No findings to remediate. Signal vX.Y.Z installed and healthy.` and exits 0 without writing a script.

### FR4 — `--reinstall` generates full canonical reinstall script

`/sig:doctor --reinstall` writes `~/.claude/sig-doctor.sh` containing the full canonical clean reinstall sequence regardless of detected state:

1. Remove cache directory (`rm -rf ~/.claude/plugins/cache/<marketplace>/sig/`)
2. Edit `~/.claude/settings.json` to remove stale `enabledPlugins` entry
3. Run `/plugin uninstall sig@signal` (or document that this step requires running inside Claude Code as a slash command — see OQ3)
4. Run `/plugin install sig@signal` (same caveat)
5. `/reload-plugins`

Prints the same `Generated ~/.claude/sig-doctor.sh — review then run: ...` message and exits.

### FR5 — Generated script has per-action `[y/N]` prompts

The script body wraps each mutating action in an interactive prompt:

```bash
read -p "Execute: rm -rf ~/.claude/plugins/cache/signal/sig/ ? [y/N] " ans
if [[ "$ans" == "y" ]]; then
  rm -rf ~/.claude/plugins/cache/signal/sig/
fi
```

Declining a prompt skips that action and continues to the next. Script ends with a "Done. Re-run /sig:doctor to verify install state." message.

### FR6 — `/sig:status` version-check

`/sig:status` queries `https://api.github.com/repos/InsightRiot/signal/releases/latest` (cached 24h in `~/.claude/.sig-version-cache.json`), compares `tag_name` against the installed `plugin.json` version, and prepends a staleness warning to the briefing when the installed version is older.

Warning includes the correct recommendation based on the detected state combination:

| Installed | Latest available | P-states detected | Recommendation |
|---|---|---|---|
| stale | newer exists | none | `/plugin install sig@signal` (normal path) |
| stale | newer exists | yes | `/sig:doctor --reinstall` (clean + upgrade) |
| current | none | yes | `/sig:doctor --fix` (surgical) |
| current | none | none | (no warning shown) |

### FR7 — Non-macOS stub

Invocations of `/sig:doctor` (any flag) on Linux or WSL print:

```
/sig:doctor currently supports macOS only. Linux/WSL support is in flight
(see docs/install-troubleshooting.md for the manual sequence).
```

and exit 0.

### FR8 — Troubleshooting doc reframe

`docs/install-troubleshooting.md` opens with an ownership statement:

We document them because strangers should not have to debug Anthropic's plugin host alone. **Run `/sig:doctor` first** — it will tell you which workaround applies and generate a script that runs the fix for you.

Per-section ownership tags added (e.g., `**Owner: Claude Code plugin host**` for P1/P2/P3; `**Owner: Signal (historical — pre-rename)**` for the `signal@signal` cache orphan). Doc length reduced where `/sig:doctor` collapses manual triage into automated detection.

## Acceptance Criteria

1. **Healthy install detection.** `/sig:doctor` runs on a healthy macOS install (fixture: clean `~/.claude/` with current Signal version) and reports `Signal vX.Y.Z installed and healthy — no action needed.` Exit code 0.

2. **Per-P-state detection.** `/sig:doctor` runs on each of the 5 documented P-state fixtures and reports the correct finding with the correct flag recommendation (`--fix` for one surgical state, `--reinstall` for combined or aggressive states). Exit code 1 for each.

3. **`--fix` script generation.** `/sig:doctor --fix` on a fixture with 1+ P-states writes `~/.claude/sig-doctor.sh` containing remediation commands ONLY for those P-states (verified via fixture filesystem comparison). Re-running `/sig:doctor` after script execution reports healthy.

4. **`--fix` no-op on healthy.** `/sig:doctor --fix` on a healthy fixture prints `No findings to remediate.` and does NOT write a script.

5. **`--reinstall` full sequence.** `/sig:doctor --reinstall` writes `~/.claude/sig-doctor.sh` containing the full 5-step canonical clean reinstall regardless of starting fixture state. Same script contents whether fixture is healthy or broken (the safeguard is the interactive prompts at execution time).

6. **Interactive prompts in script.** Generated script source contains per-mutating-action `read -p "Execute: ... ? [y/N]"` wrappers (verified via script-content lint test). Declining one prompt skips that action and continues.

7. **`/sig:status` version-check accuracy.** `/sig:status` prepends a staleness warning when (fixture installed version) < (mocked GitHub API latest tag). Warning text includes the correct recommendation per the state-combination matrix in FR6. No warning when versions match.

8. **`/sig:status` cache behavior.** GitHub API call hits the network on first invocation; subsequent calls within 24h read from cache (verified via mock counting API hits). After 24h, cache invalidates and a fresh call fires.

9. **Doc reframe.** `docs/install-troubleshooting.md` opens with the ownership statement (verified via grep). Per-section ownership tags present on all 5 symptom sections.

10. **Non-macOS stub.** `/sig:doctor` invoked with `process.platform === 'linux'` or `'win32'` (mocked) prints the polite stub and exits 0. No detection logic runs.

11. **Test suite green.** Existing 384 tests stay passing; new tests added under `tests/doctor.test.js` (S1) + `tests/doctor-script-gen.test.js` (S2) + `tests/status-version-check.test.js` (S3).

12. **Validator green.** `tools/validate-plugin.js` recognizes `commands/doctor.md` as required and passes.

13. **Dogfood criterion.** At least one end-to-end run of `/sig:doctor --reinstall` executed on a machine other than Mac Studio (likely the biz machine, also macOS per D-E8-2) before SHIP. Captured in a new `docs/install-verification.md` § R6 row.

## Open Questions for PLAN's research phase

OQ1: **What's the canonical path Claude Code uses for plugin state?** `~/.claude/plugins/installed_plugins.json` is referenced in `docs/install-troubleshooting.md`, but other paths may also matter (`~/.claude/settings.json` for `enabledPlugins`, `~/.claude/plugins/cache/<marketplace>/` for actual code). PLAN's codebase-researcher should empirically verify by inspecting a working install on Mac Studio.

OQ2: **Does `api.github.com/repos/.../releases/latest` work unauthenticated?** If yes: pure `curl` works on any machine. If no: requires `gh` CLI installed + authenticated, which constrains who can use the version-check. Affects FR6 reliability. PLAN should resolve via empirical test.

OQ3: **Can the generated script execute `/plugin uninstall` and `/plugin install` directly?** These are Claude Code slash commands, not shell commands. The generated script can't `bash`-execute them. Options: (a) script prints instructions for the user to run the slash commands manually after the filesystem cleanup; (b) Claude Code exposes a CLI subcommand equivalent; (c) some indirect mechanism. PLAN should resolve before S2 implementation.

OQ4: **Upstream filings timing — pre-S3 or post-E8 SHIP?** User feedback: file in the moment, don't wait. Likely answer: file alongside S1 (as soon as `/sig:doctor` ships, the upstream issues are properly cross-linkable). PLAN can lock this.

OQ5: **Cache file location.** `~/.claude/.sig-version-cache.json` was assumed in FR6 but not formally locked. Could live in `~/.claude/plugins/cache/.../signal/` instead. Minor; PLAN to confirm.

## Anti-rationalizations locked

- *"Add a `--dry-run` flag."* — No. The default behavior (write a script, print run instruction) IS the dry-run mode. Adding another flag is surface bloat.
- *"Detect 'marketplace vs. local install' automatically."* — No. Signal is marketplace-distributed by design now. If a user installed via some other mechanism, that's out of scope.
- *"Make `/sig:doctor` work for other Claude Code plugins too."* — No. Signal-specific. Generalizing is a separate Epic if it ever earns scope.
- *"Pad the generated script with explanatory comments."* — Minimal comments, not narrated tutorial. The user already audited the script before running; comments serve execution-time questions only ("what does this command do?"), not "what does this script accomplish?" (the doctor invocation prose already answered that).
- *"Auto-execute on `--fix` since it's surgical / small."* — No. Script-gen-with-review applies to both `--fix` and `--reinstall`. Consistency over per-flag UX nuance.
- *"Bundle Linux/WSL support into S1 'because the detection is similar.'"* — No. D-E8-2 explicitly defers. Adding both = doubles fixture surface + verification hardware dependency. Ship macOS first.
- *"Skip the upstream issue filings — Anthropic will probably fix it on their own."* — No. They might, they might not, on indefinite timeline. File the issues. Worst case: they're already known and our filing dups. Best case: documented repros accelerate the fix.
