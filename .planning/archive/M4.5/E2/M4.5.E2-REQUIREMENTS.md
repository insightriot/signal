## Objective

Finish `/sig:add` so capture is frictionless and captured ideas don't rot. Two clean halves:

- **Capture** (quoted or naked) — always lands in `FUTURE-IDEAS.md`, zero routing guesswork.
- **Routing** — explicit flags only (`--question`, `--milestone [N]`, `--file`). No heuristic/LLM classification.
- **Drain** — `/sig:plan` surfaces recent captures and recommends a disposition per entry (advisory, not blocking).

Audience is **self + peers** (per the E3 audience reframe, `M4.5.E3-REQUIREMENTS.md` § D-E3-11), not anonymous strangers — so power-user flags earn their keep and the naked interview stays minimal.

## Non-Functional Requirements (FULL tier)

`/sig:add` is a local CLI capture command that writes markdown — **not** a network service. The FULL-tier ops checklist (health/liveness probe, graceful shutdown, structured request logging, security headers, public rate limiting) is **N/A** — there is no server, port, or request surface. The NFRs that *do* apply are data-integrity and safety, and S1 already provides the substrate:

- **Atomic writes** — temp + `fs.rename` with `EXDEV` fallback (`atomic-write.js`). Every new destination reuses it; no `appendFile`.
- **Concurrency safety** — `.planning/.add.lock` (`file-lock.js`, 30s TTL). Reused for all destinations.
- **Secret-leak prevention** — `scrubSensitive` prompt before write; never auto-redact. Applies to every destination.
- **Input-boundary validation** — `--file` path refusals (FR3); multi-flag guard (FR4); body-length soft cap.
- **Failure logging** — errors propagate verbatim with a `git diff` recovery hint; abort paths mutate nothing.

Each new destination (OPEN-QUESTIONS, milestone holding section, `--file`) gets its own fixture test so footer/separator integrity is verified per-destination (the top risk: corrupting a destination file).
