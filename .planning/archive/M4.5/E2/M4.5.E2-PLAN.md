## Phase Goal

Ship `/sig:add` — a not-tier-gated capture command that lets any user (especially newcomers) append a stamped entry to one of 4 `.planning/` destinations via a zero-question hot path or a single-question cold path, hardened against file corruption, secret leakage, and vocabulary drift.

> **Scope cut from FUTURE-IDEAS spec:** The 5th destination (new-MILESTONE-N.md scaffold) is **deferred** per RESEARCH § 6 Q1 — hand-authoring milestones is the existing precedent (M4, M5, M4.5 all hand-authored), and including scaffold in v1 adds high-blast-radius surface for ~2% of routing volume. Captures of "new milestone idea" route to FUTURE-IDEAS instead.

## Assumed Runtime (locked at PLAN time; verified in Slice 1)

- **Node 22+** (per `package.json` engines).
- `node:fs/promises` + `node:fs.existsSync` + `node:path` only — no new ESM deps.
- POSIX `fs.rename` atomic on Mac/Linux. Windows-WSL falls back to copy-then-unlink (`EXDEV` handler).
- No native modules. No external services. No network calls.
- `npm install --dry-run` clean (verified in Slice 1).

### Slice 1 — Hardened hot path (MVP) — `[L]`

**User value:** `/sig:add "idea text"` writes a stamped entry to `FUTURE-IDEAS.md` safely. Newcomers can capture ideas without learning Signal's vocabulary.

**Scope:**
- Create `commands/add.md` (meta-command shape; no tier-gating preamble; modeled on `commands/resume.md`).
- Create `tools/lib/add.js` with: `parseInput(args)`, `captureToFutureIdeas(baseDir, entry)`, `atomicWrite(path, content)`, `acquireLock(baseDir)`, `releaseLock(baseDir)`, `scrubSensitive(text)`.
- Pre-flight: `existsSync('.planning/FUTURE-IDEAS.md')` — if missing, fail loud with "Run `/sig:init` first" message.
- `.gitignore` check (same rule as every other write-to-`.planning/` command — calibrate.md:98-102 pattern).
- Atomic write: read → build new content with entry inserted *above* `*Last updated: YYYY-MM-DD*` footer → write to `.planning/.tmp-<rand>.md` → `fs.rename`. `EXDEV` fallback (copy + unlink).
- Lock file `.planning/.add.lock`: created with `O_EXCL`, contains `${pid}\n${Date.now()}\n`, TTL 30s (stale lock auto-released). `process.on('exit', 'SIGINT')` cleanup.
- Footer rewrite: `*Last updated: YYYY-MM-DD*` → today's date, regardless of prior value.
- Sensitive-data scrub (regex): `/AKIA[0-9A-Z]{16}/`, `/ghp_[a-zA-Z0-9]{36,}/`, `/Bearer\s+[a-zA-Z0-9._-]+/`, `/\b[a-f0-9]{40}\b/i`. On hit: prompt user via `AskUserQuestion(strict-enum, [keep, abort])` with the matched span shown; never auto-redact silently.
- Entry shape (FUTURE-IDEAS template):
  ```markdown
  ## {Title-Cased first ~6 words of body}

  **Status:** Logged YYYY-MM-DD via `/sig:add`. {Trigger context from STATE.md if mid-phase, else omit.}

  {body verbatim — no LLM rewrite}

  ---
  ```
- Validator + manifest updates:
  - `tools/validate-plugin.js:27-40` — append `'commands/add.md'` to `REQUIRED_COMMANDS`.
  - `README.md:90-99` workflow table + `:134-145` per-command list — add `/sig:add` row.
  - `CLAUDE.md:78` — bump count (currently "9 slash commands"; verify and update).
- Success line: `Added to .planning/FUTURE-IDEAS.md (line {N}). Review with: git diff .planning/FUTURE-IDEAS.md`

**Acceptance criteria:**
1. `/sig:add "test idea"` (with `.planning/FUTURE-IDEAS.md` present) writes a well-formed entry above the footer and returns exit 0.
2. `/sig:add` without `.planning/` exits non-zero with "Run `/sig:init` first."
3. Concurrent runs: the second `/sig:add` invocation (while first holds lock) exits non-zero with "Another `/sig:add` is running; retry in <30s."
4. Sensitive-data hit on input prompts user; user-aborts → no file write; user-keeps → write proceeds verbatim.
5. Atomic write: if `fs.rename` fails mid-operation, the destination file is unchanged (temp file may exist, gets cleaned by lock release).
6. Footer rewrite: `*Last updated:*` date matches today after every successful write.
7. Validator (`npm run validate`) passes with new command.
8. `npm install --dry-run` clean (Node 22+).
9. **(OPTIONAL from validation)** Body > 4000 chars warns via `AskUserQuestion(strict-enum, [keep, abort])` with "Body is {N} chars — that's longer than typical capture (~500 chars). Continue?" No hard cap; user can keep.

**Test strategy (TDD, red-green-refactor):**
- `tests/add.test.js` (new):
  - Unit: `parseInput("text")` → `{body: "text", flags: {}}`.
  - Unit: `scrubSensitive("Bearer abc.123")` → `{hits: [...], scrubbed: false}` (returns hits, doesn't auto-redact).
  - Integration: temp-dir fixture with minimal `.planning/FUTURE-IDEAS.md`; `captureToFutureIdeas` writes entry above footer; footer date rewritten; entry shape matches inline snapshot.
  - Integration: concurrent-lock test — acquire lock in one promise, second `acquireLock` rejects.
  - Integration: `EXDEV` fallback path — mock `fs.rename` to throw `EXDEV` once; copy+unlink fallback executes; final file identical.
  - Integration: missing-`.planning/` → throws expected error.
- `tests/fixtures/add/future-ideas-minimal/.planning/FUTURE-IDEAS.md` (new fixture; 30-line synthetic FUTURE-IDEAS).
- Validator-test: `npm run validate` passes (existing test pattern from `tests/validate-plugin.test.js` if present, else add).

**Complexity:** **L** — Large because every safety primitive lands here. None are optional; this is the minimum shippable surface.

**Dependencies:** None (Slice 1 is the foundation).

**Risk coverage:** Top-5 risks #1, #4, #5 mitigated in this slice (footer corruption via per-destination template + fixture test; atomic-write via temp+rename+lock; sensitive-data scrub).

### Slice 2 — Force-route flags — `[M]`

**User value:** Power users who know the destination skip the interview: `/sig:add --question "should we…"`, `/sig:add --milestone "fix the foo"`, `/sig:add --milestone 5 "add this to M5"`.

**Scope:**
- Extend `parseInput(args)` to parse flags: `--question`, `--milestone [N]`, `--file <path>` (undocumented escape valve per RESEARCH § 1 #25 footnote — keep but don't surface in `--help`).
- Implement `captureToOpenQuestions(baseDir, entry)` — appends above trailing `---` separator. OPEN-QUESTIONS template differs from FUTURE-IDEAS (Status/Watch-for/Resolve-by shape per `.planning/OPEN-QUESTIONS.md:13-19`).
- Implement `captureToMilestone(baseDir, milestoneId, entry)` — appends to a new `## Captured via /sig:add` section near the end of the milestone file, before the trailing footer (`*Created ...*` if present, else `---`). If no current milestone in STATE.md and `--milestone` invoked without `N`: fail with "Specify --milestone N or run /sig:status to see active milestones."
- New helper `tools/lib/milestones.js`:
  - `listMilestones(baseDir)` — glob `.planning/MILESTONE-*.md`, return `[{id, file, exists}]` sorted numerically (handling decimals like 4.5).
  - `currentMilestone(baseDir)` — reads STATE.md (source of truth); returns milestone id or `null`. **No file-scan heuristics** (RESEARCH § 1 #20).
- `--file <path>` flag: hard-refuses paths matching `/(DECISIONS|STATE)\.md$/` (RESEARCH § 1 #13). Refuses paths outside `.planning/`. No template applied — raw body appended above last `---` or end-of-file.
- Update success line for each destination (use destination's actual path).

**Acceptance criteria:**
1. `/sig:add --question "is X right?"` writes to `OPEN-QUESTIONS.md` in OPEN-QUESTIONS template shape.
2. `/sig:add --milestone "fix bug"` writes to current milestone (from STATE.md); fails clearly if STATE.md has no current milestone.
3. `/sig:add --milestone 5 "add to M5"` writes to `MILESTONE-5.md`; fails if file doesn't exist (no auto-create — that's the deferred 5th destination).
4. `/sig:add --milestone 99 "bad"` → "MILESTONE-99.md does not exist. Use --new-milestone to scaffold (deferred to v0.2; for now, hand-author the file)." [Note: `--new-milestone` flag not implemented in v1 per Q1 deferral; error message tells the user.]
5. `/sig:add --file /etc/passwd "..."` → hard-refuse "destination must be inside .planning/".
6. `/sig:add --file .planning/DECISIONS.md "..."` → hard-refuse "DECISIONS.md and STATE.md are not valid destinations."
7. `listMilestones` returns `[{id: "4", ...}, {id: "4.5", ...}, {id: "5", ...}]` sorted (decimal-aware).
8. **(GAP-1 close)** OPEN-QUESTIONS.md and MILESTONE-*.md footer markers (`---` separator, `*Created ...*` footer if present) preserved/rewritten correctly per per-destination fixture test. Each of 3 new destinations gets a dedicated fixture under `tests/fixtures/add/`.
9. **(GAP-2 close)** Invoking with multiple destination flags (e.g., `--question --milestone`, `--milestone --file`) exits non-zero with a clear error message *before* any write or lock acquisition.
10. Validator still passes.

**Test strategy:**
- Add fixtures for OPEN-QUESTIONS, MILESTONE-current, MILESTONE-other under `tests/fixtures/add/`.
- Unit: `parseInput("--milestone 5 'foo'")` → `{body: "foo", flags: {milestone: "5"}}`.
- Unit: `listMilestones` decimal-sort assertion.
- Unit: `currentMilestone` returns STATE.md value; null when STATE.md silent.
- Integration: per-destination template tests (each of 3 new destinations).
- Integration: hard-refuse cases (DECISIONS, STATE, outside-.planning).
- Negative: `--milestone N` for nonexistent N → expected error.

**Complexity:** **M** — Adds 3 new destination templates + 1 new helper module. Reuses Slice 1's atomic-write + lock + scrub primitives.

**Dependencies:** Slice 1 (atomic-write primitives, lock, scrub all reused).

**Risk coverage:** Top-5 risk #1 extended (each new destination gets its own template + fixture test).

### Slice 3 — Cold-path interview + heuristic hints — `[M]`

**User value:** Naked invocation `/sig:add` (no args) prompts once; ambiguous input gets a single-key routing override hint.

**Scope:**
- Naked invocation handling: if `args` is empty/whitespace, issue one `AskUserQuestion(open-ended)` per `references/question-patterns.md` for "What's the idea?"; on empty response → abort cleanly (exit 0, no write).
- Heuristic hints (input parsed, no flags supplied):
  - Input ends with `?` → suggest OPEN-QUESTIONS via `AskUserQuestion(3+other)` with: `[FUTURE-IDEAS (default), OPEN-QUESTIONS (recommended — ends with ?), current milestone]`. Recommendation marker on OPEN-QUESTIONS.
  - Input matches `/^(fix|bug|TODO|broken|fixme)\b/i` AND STATE.md has current milestone → suggest current milestone: `[FUTURE-IDEAS (default), current milestone (recommended — looks like a task), OPEN-QUESTIONS]`.
  - Otherwise → silent default to FUTURE-IDEAS (zero questions, hot path).
- Heuristics live in `tools/lib/add.js#suggestDestination(body, state)` — pure function for testability.
- Plain English only — never use Signal vocabulary in question text (RESEARCH § 1 #16). E.g., "This looks like a question — route to OPEN-QUESTIONS.md?" not "This looks like an architectural question — route to OPEN-QUESTIONS.md altitude?"

**Acceptance criteria:**
1. `/sig:add` (no args) prompts "What's the idea?"; user response → routes per heuristics or default FUTURE-IDEAS.
2. `/sig:add` (no args) + empty response → exit 0, no file write.
3. `/sig:add "is this right?"` → 3+other prompt with OPEN-QUESTIONS recommended.
4. `/sig:add "fix the foo"` + current milestone in STATE.md → 3+other prompt with current milestone recommended.
5. `/sig:add "general idea"` → silent default to FUTURE-IDEAS, no prompt.
6. `suggestDestination(body, state)` unit-testable in isolation.

**Test strategy:**
- Unit: `suggestDestination` table-driven test — 12 input/state combinations → expected destination + recommendation.
- Integration: each cold-path scenario writes to expected destination.
- Mocked AskUserQuestion responses (vitest pattern for stdin-style prompts — see existing pattern in `commands/init.md` Step 5 tests if present, else add minimal mock).

**Complexity:** **M** — Heuristics are simple regex; the interface mocking + AskUserQuestion plumbing is the larger surface.

**Dependencies:** Slice 1 (write primitives), Slice 2 (multi-destination support).

**Risk coverage:** Top-5 risk #3 mitigated (plain-English routing, no vocabulary at capture; success line still prints path + line for recovery).

### Slice 4 — Stranger-safety hardening — `[S]`

**User value:** Strangers running `/sig:add` for the first time can't accidentally leak secrets or get stuck in wrong-directory mode.

**Scope:**
- First-run onboarding warning: on first `/sig:add` invocation per repo, before any write, show one-time warning via `AskUserQuestion(strict-enum, [ok, abort])`: "`.planning/` is git-tracked by convention. Entries here become permanent on commit. Treat captures as public-ish. Continue?" — persist `.planning/.add-onboarded` after `ok`; suppress on subsequent runs.
- Brownfield-without-`.planning/` improvement: enrich Slice 1's "Run `/sig:init` first" error with detection: if `.git/` exists AND source files exist → "This looks like a brownfield repo. Run `/sig:init` to scan, then `/sig:add` will work." Else → "No project detected. Run `/sig:new-project` to start fresh."
- Honor `gate_strictness` (read PROFILE.md if present; gracefully skip if absent):
  - `strict` → preview entry block (5-10 lines, shows what's about to be written) + confirm destination via `AskUserQuestion(strict-enum, [yes, abort])`.
  - `light` → confirm destination only (no preview).
  - `off` → still confirm destination once (capture is cheap; never auto-accept destination per RESEARCH § 1 #12).
- Vocabulary lint: extend `tools/validate-plugin.js` (or add new sub-check) to scan `commands/add.md` and `tools/lib/add.js` for banned terms (today: `tranche`, `Tranche`, `TRANCHE` per M4.t18 lock). Test asserts validator catches a planted "tranche" in command file.

**Acceptance criteria:**
1. First `/sig:add` in a fresh repo prompts onboarding warning; second `/sig:add` does not.
2. `.planning/.add-onboarded` is git-tracked-by-default (no special `.gitignore` rule needed; it's inside `.planning/`).
3. Brownfield detection error message includes `/sig:init` suggestion.
4. With `gate_strictness: strict`, `/sig:add "idea"` shows entry preview before write.
5. With `gate_strictness: off`, `/sig:add "idea"` still asks "Confirm destination: FUTURE-IDEAS.md?" (cheap-confirm rule).
6. Validator catches `tranche` in a planted command file.

**Test strategy:**
- Integration: first-vs-second run behavior (temp dir, two consecutive captures, assert onboarded flag).
- Integration: brownfield-detection error text matches when `.git/` exists.
- Integration: each gate_strictness value produces expected prompt count.
- Validator test: vocab-lint catches planted banned term.

**Complexity:** **S** — Small slice; mostly thin layers on top of Slices 1-3 plumbing.

**Dependencies:** Slices 1, 2, 3.

**Risk coverage:** Top-5 risk #5 (sensitive-data leak) reinforced by onboarding warning; risk #2 (vocabulary drift) mitigated by vocab-lint.

## Anti-rationalization (locks for execution time)

| Temptation | Check |
|---|---|
| "`appendFile` is simpler; atomic-write is overkill" | No. Markdown files are the entire product database. `appendFile` ≠ atomic on crash. RESEARCH § 1 #7 locks this. |
| "Lock file is paranoid; solo dev never races" | Cheap and silent — solo-dev assumption is right today; the lock costs nothing and the race-mode is silent corruption. Keep. |
| "Skip the sensitive-data scrub; user's problem" | No. `.planning/` is git-tracked by convention; public-leak is irreversible (top-5 risk #5). Even imperfect regex is worth it. |
| "Just preview before write; user confirms" | No. Capture latency dies on confirmation (Linear "C" lesson). Diff *after*, not before. |
| "Bundle the 5 destinations into one big AskUserQuestion enum" | No. 5 destinations flat is the vocabulary-wall `/sig:add` exists to eliminate (RESEARCH § 1 #17). Use 3+other dynamic narrowing. |
| "LLM-classify the input — Claude knows where it belongs" | No. Notion's AI auto-categorize is famously distrusted. Heuristic + single-key override is faster and more predictable (RESEARCH § 5 anti-pattern #3). |
| "Slice 5 is optional; ship S1-S4 first, defer S5" | No. Slice 5 closes the GTD loop. Without it, captures rot. Atomic capture-AND-process is the GTD principle — ship both halves. |
| "Number entries (FUTURE-001) for traceability" | No. GSD does this for seeds (which have trigger conditions); FUTURE-IDEAS entries are prose. Numbering adds bookkeeping without value (RESEARCH § 5 anti-pattern #5). |
| "Add `--quick` flag for power users" | No. M4.t18 + RESEARCH locks this — the interview *is* the feature for newcomers. Re-evaluate after 5+ users ask. |
