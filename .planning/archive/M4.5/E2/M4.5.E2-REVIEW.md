## Suggestions (optional improvements)

- **S1 — Q2 prose-blind verb match hid genuine candidates → RESOLVED in-loop (Q2 refined, user-approved 2026-05-31).** The original `STATUS_DISPOSED_RE = /\b(Promoted|Deferred|Merged|Deleted)\b/` marked an entry dispositioned whenever its Status line *contained* a disposition verb, even in prose. Quantified against the live file: **1 of 29 entries** (`FUTURE-IDEAS.md:224` — "Synthesizer-output validator-side sanity-check", Status "**Deferred from M4.5.E7** to FUTURE-IDEAS…") was a **false negative**. Surfaced to the user as a decision-locked tradeoff; **user chose to fix it (Option B)** rather than ship-and-log. **Q2 refined:** the rule now matches only the drain's own stamp signature — `{Verb} {date} (… drain)` — so a prose mention no longer counts; heading markers (`## ✓ SHIPPED`) still caught by `HEADING_DISPOSED_RE`. Fix is a ~1-line regex change + comment + 3 regression tests; all 762 tests green; no architectural impact (qualifies as the PASS-WITH-FIXES in-phase path). Verified live: the previously-hidden entry now surfaces (28 candidates, was 27). Commit: see `M4.5.E2.S5.t5`.
- **S2 — Minor DRY: two implementations of "first non-fenced `**Status:**` line".** `parseEntries`' inner scan (L122-131) and `statusLineIdxInBlock` (L177-187) are the same fence-aware loop over different line slices. The code comments the intentional mirroring, but a single `firstStatusLineIdx(lines, from)` helper would remove ~8 duplicated lines and guarantee surface/write can't drift. Low value (already well-documented); safe to defer.
- **S3 — `applyDispositionToFile` parses content twice.** Once for `entry` (L313) and again inside `applyDisposition` (L240). O(n) twice over a ~150 KB file — negligible, but threading the parsed entry through would halve the parse work. Defer.
- **S4 — Unbalanced code fence under-surfaces.** An entry containing an *unclosed* ``` fence would leave `inFence` true for the rest of the file, hiding all subsequent entries from the drain (under-surfacing, never corruption — no write touches them). Low risk for well-formed markdown; a "trailing-open-fence" guard (warn if `inFence` is still true at EOF) would make it self-diagnosing. Defer.

## Security Findings (OWASP + ASVS L2, scoped to a local markdown CLI)

`drain.js` is a local, offline markdown transform — no server, port, request surface, auth, DB, or network. Most OWASP categories are structurally N/A. Audited the ones that map:

- **A01 Access control / A07 auth** — N/A (no identity surface).
- **A03 Injection / path traversal** — `applyDispositionToFile(baseDir, relPath)` builds the target path from `relPath`, which is **command-controlled** (plan.md passes the fixed `.planning/FUTURE-IDEAS.md`), never user free-text. Unlike `/sig:add`'s `--file`, there is no user-supplied path here, so no traversal gate is needed. ✅
- **A04 Insecure design / A08 data integrity** — the primary risk for a file-mutating tool. Mitigated by: byte-range edits (neighbors byte-identical, proved by snapshot tests), the **R1 preview-diff-before-write** gate (command layer), the **R5 `[confirm, keep]`** gate on destructive verbs (regardless of `gate_strictness`), and full-file `atomicWrite` (temp+rename; rename-failure leaves the file unchanged, tested). ✅
- **Secret leakage** — drain introduces **no new free-text** into the file: promote/defer append only a fixed-format stamp `→ {Verb} {date} ({reason}).` where `reason` is command-controlled (`{Epic} drain`) and `date` is ISO. So `/sig:add`'s capture-time `scrubSensitive` is correctly **not** duplicated here — there is no new ingestion path to scrub. ✅
- **A06 Vulnerable dependencies** — none added; `drain.js` is pure Node (`node:fs/promises`, `node:path`) + the existing `atomic-write.js`. `npm install --dry-run` unchanged. ✅

**Security verdict: PASS.**

## Performance Findings

- `parseEntries` is **O(n)** over lines: one heading-scan pass + a per-entry Status scan whose spans sum to n. At ~1,500 lines / 29 entries it's sub-millisecond.
- `applyDispositions` parses once, then O(B) range-splices for B dispositions — the right shape for "defer all remaining" (avoids B re-parses).
- `applyDisposition` re-parses per call; a per-entry command loop is O(E·n) ≈ 29·1,500 line-ops — trivial. No N+1, no unbounded growth, not frontend.

**Performance verdict: PASS — no anti-patterns.**
