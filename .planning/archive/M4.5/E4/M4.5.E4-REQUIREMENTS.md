## Objective

Make a newcomer *get* Signal fast, two ways:

- **A worked example** — one complete `calibrate → ship` run, committed so a reader can see exactly what each phase produces and watch the **calibration router** (Signal's wedge) decide the rigor. Runnable, so "clone and `npm test`" is real, not claimed.
- **A comparison page** — a short, honest "when to reach for which" guide across the plugin landscape Signal is built from, with the calibration router as the differentiator.

Both must be linked from the README.

- **FR1 — `examples/url-shortener/` worked example.** Contains: runnable source (`src/` + `tests/` + `package.json`; **no** `node_modules`), the refreshed `.planning/` chain (PROFILE, STATE, CONTEXT, PROJECT, REQUIREMENTS, config.json, and the numbered phase artifacts 1-RESEARCH / 1-PLAN / 1-VALIDATION / 1-PROGRESS / 1-VERIFICATION / 1-REVIEW / 1-SHIP), and an example `README.md`.

- **FR2 — Annotated tour in the example README.** Walks the reader through (a) what the project is, (b) that it was calibrated **FULL** and *why* (which calibration answers fired which rule), and (c) what each phase artifact represents. Makes the calibration → right-sized-rigor link explicit — this is the example's reason to exist.

- **FR3 — Artifacts match current Signal conventions.** Every refreshed `.planning/` file uses today's shapes: STATE.md `schema_version: 1` frontmatter, Milestone/Epic vocabulary, current PROFILE schema. The drift audit (what changed vs. the 2026-04-26 originals) is documented (in the example README or a short audit note), so a reviewer can confirm faithfulness.

- **FR4 — Currency guard test(s) in the root suite.** Assert the example's `PROFILE.md` parses via `readProfile` and `STATE.md` via `readState` (from `tools/lib/`) without error, and run a structural check over the example `.planning/`. Effect: a future schema change that the example doesn't track breaks the root test run instead of silently rotting the example.

- **FR5 — `docs/vs.md` comparison page.** Prose decision guide, ~60–80 lines, covering Signal vs. the 5 comparators (D-E4-6), organized as "reach for X when…, reach for Signal when…", leading with *when to use Signal*, naming the calibration router as the wedge, heritage-respectful tone (names what Signal ports/borrows from each). Links to `analysis/REPO-ANALYSIS.md` for depth.

- **FR6 — README links both artifacts.** A pointer to `examples/url-shortener/` ("see a full run") and to `docs/vs.md` ("how Signal compares"). This is the Epic's exit criterion.

## Risks & Assumptions to validate (in PLAN / EXECUTE)

- **R1 — Native-dependency caveat (settle in PLAN).** The url-shortener dogfood hit `better-sqlite3` native-module / Node-version friction (logged in STATE.md, M3 Task 2 findings). A *runnable* example (D-E4-3) with a native dep can fail `npm install`/`npm test` on a stranger's machine — undercutting the credibility that motivated D-E4-3. **PLAN must choose:** (a) swap the example's storage to a zero-native-dependency store during the refresh (recommended — the example exists to show Signal's flow, not to be a production URL shortener), (b) keep `better-sqlite3` and document the build step + supported Node range, or (c) scope the guard to artifact-parsing only and label "runnable" with documented caveats. Affects FR1/FR3/FR4.

- **A1 — Salvageability.** Assumes the `.dogfood/url-shortener-fulltier/` artifacts are complete and internally consistent enough to refresh rather than regenerate. Validate at the start of EXECUTE; D-E4-2's escape hatch covers the fallback.

- **A2 — Example suite currency.** Assumes the example's ~39 tests still pass on the current Node version after the refresh (coupled to R1). Validate before claiming "clone and `npm test` passes."
