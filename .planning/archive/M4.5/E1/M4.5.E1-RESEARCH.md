## 1 — Load-bearing findings (the 9 facts the plan must respect)

1. **Anthropic ships a documented HTTPS escape hatch already.** Claude Code 2.1.141 (Apr 2026) added env var `CLAUDE_CODE_PLUGIN_PREFER_HTTPS=1` — "to clone GitHub plugin sources over HTTPS instead of SSH, for environments without a GitHub SSH key." This means (a) Anthropic considers HTTPS the install-everywhere default, (b) a documented stopgap exists today for *any* user hitting our break, and (c) we should cite this in user-facing troubleshooting.

2. **`"source": "github"` shorthand resolves to SSH (`git@github.com:owner/repo.git`).** Confirmed by Claude Code GitHub issues #47088, #29722, #31930, #18001, #26588, #27771, #52234, #20589 — all open or closed-as-duplicate reports of the same defect; #52234 (Apr 2026) is the most recent. `plugin install` does NOT have the HTTPS fallback that `marketplace add owner/repo` shorthand has. The asymmetry is the bug.

3. **Anthropic's own catalog avoids `"source": "github"` shorthand.** In `claude-plugins-official/.claude-plugin/marketplace.json` (172 plugins), distribution is: 83 use `"source": "url"` + explicit `https://...git`, 38 use `"source": "git-subdir"` + explicit URL, only **2** use `"source": "github"`. Both of those 2 (`jfrog`, `fullstory`) still pin `commit` + `sha`. **Zero** plugins use unpinned `"source": "github"`. Signal sits in a population of one with floating-tip `"source": "github"`.

4. **Agent subdirectory recursion is undocumented.** Claude Code docs describe `agents/` as a flat directory of `.md` files. Across 8 surveyed Anthropic plugins shipping agents (`hookify`, `code-simplifier`, `feature-dev`, `plugin-dev`, `pr-review-toolkit`, `code-modernization`, `agent-sdk-dev`, `skill-creator`), **100% use flat `agents/<name>.md`; 0% use nested category subdirs**. Signal's 26 agents under `agents/{executors,planners,researchers,scanners,specialists,support,verifiers}/` is unprecedented. M4.t15 dogfood data (dev-mode Task tool can't see scanners) is corroborating evidence: F2 outcome is almost certainly (b) or (c), not (a).

5. **Agent namespacing convention is `<plugin-name>:<agent-name>`.** Per docs: "agent `agent-creator` for the plugin with name `plugin-dev` will appear as `plugin-dev:agent-creator`." Signal's plugin name is `sig`, so registered names will be `sig:stack-scanner`, `sig:planner`, etc. — NOT `signal-stack-scanner` (the stale hedge in `commands/init.md:170`).

6. **Pin policy gap is real.** `marketplace.json` has no `ref` or `sha`. Every stranger install pulls HEAD of `main`. Installs are non-reproducible across collaborators and across same-machine retries. Pairs with auto-update behavior: third-party marketplaces have **auto-update disabled by default** — users must manually run `/plugin marketplace update signal` to see new versions. Combined with the documented "if plugin.json `version` field doesn't change, the cache sees the same version and refuses to update" rule, this is a high-severity silent-staleness landmine.

7. **Validator is install-blind.** `tools/validate-plugin.js` checks file *existence* but never reads `marketplace.json` content (so today's source-block break is invisible), never models the loader's directory walk, never asserts post-install state. The "local-green-but-install-breaks" failure mode is wide open.

8. **No CHANGELOG, no versioning policy, no install troubleshooting doc.** `docs/` contains only `migration-vocab-v0.1.0.md` and `docs/map/` (Vercel hosting artifacts). `CHANGELOG.md`, `CONTRIBUTING.md`, `SECURITY.md`, `docs/versioning.md`, `docs/install.md` — all absent. README's install section uses stale `/plugin install signal` syntax (pre-M4.t19 slug rename) and does not mention SSH/HTTPS caveats.

9. **Failure surface beyond today's break is wide.** Risk catalog enumerates ~30 distinct install-time failure modes across 7 categories (auth/network, cross-platform, version skew, plugin schema, post-install, update path, uninstall). The minimum verification matrix is 8 rows covering OS × SSH-config × auth-state × Node-version × pre-existing-state. Most failure modes have a documented mitigation; many have no test coverage at all.

## 3 — `source` block design options

Eight options surveyed (risk-researcher Q2, prior-art-researcher Q1). The top three:

| Option | Solves today's break? | Pinning? | Cross-platform? | Maintenance | Prior-art alignment |
|---|---|---|---|---|---|
| **(b) `"source": "url"` + explicit HTTPS URL** | **Yes** — HTTPS bypasses SSH entirely | Yes (`ref` + `sha`) | Yes | Low | **Strong** — 83/172 official plugins (~40%) |
| (d) `"source": "git-subdir"` with HTTPS URL + `"path": "."` | Yes | Yes | Yes | Low | Strong — 38/172 official plugins (~19%). Adds indirection without benefit since Signal is whole-repo |
| (h) Keep `"source": "github"` but add `ref` + `sha` pinning | No — protocol still SSH | Yes | Yes | Low | Weak — only 2/172 plugins use this; both with `commit`+`sha` pin |

**Recommendation (the planner accepts at gate):** Option **(b)** — `"source": "url"` + explicit `https://github.com/InsightRiot/signal.git` + `ref: "v0.1.1"` + `sha: "<40-char-commit>"`. Reasoning: solves today's break, matches dominant Anthropic convention, gives explicit version pinning, supports per-release cadence control.

Floating-vs-pinned ref is **orthogonal**: pinning to `v0.1.1` tag gives us per-release control (must bump marketplace.json on every release), unpinning makes every push to main a release. Recommend pinning — release cadence should be intentional, not coupled to commit cadence.

## 6 — Versioning + deprecation policy options

Five options surveyed (risk-researcher Q5). Tradeoff summary:

| Option | Strictness | Friction (author) | Friction (users) | User-count-1 fit |
|---|---|---|---|---|
| **(a) Strict semver + CHANGELOG, no deprecation** | High | Low-medium | Medium (read CHANGELOG) | **Good fit** — minimal overhead, honest pre-1.0 signaling |
| (b) Deprecation period for every breaking change | Highest | High (2× release per breaking change) | Low | Premature — needs users first |
| (c) "Move fast" — no policy until E5 | Lowest | Lowest | Highest | Contradicts E1 exit criterion ("versioning policy committed") |
| (d) Hybrid — deprecation for user-surface, semver for internals | Medium-high | Medium | Low for surface, none for internals | Premature; needs API surface stable |
| (e) SemVer + `[BREAKING]` tags in CHANGELOG | Medium | Lowest formal | Medium | Subset of (a) |

**Recommendation (planner accepts at gate):** **(a) Strict semver from v0.1.0 + CHANGELOG discipline, no deprecation window**. Rationale: pre-1.0 (`0.x.y`) conventionally allows breaking changes at `x` bumps; user-count is 1 today (premature to gate ourselves on deprecation cycles); CHANGELOG is the universal substrate every option needs. Revisit when E5 brings ≥3 external users — at that point, option (b) or (d) becomes warranted.

`docs/versioning.md` (Slice 4) codifies (a) + a release checklist (bump `plugin.json.version`, update `marketplace.json` `ref` + `sha`, tag git, write CHANGELOG entry, push tag + marketplace.json).
