## FR1 — `/sig:sweep`: on-demand doc-hygiene sweep over the invoking project (D-M5E6-1)

A new meta command (same class as `/sig:status` / `/sig:migrate-memory` — no tier-gating preamble, no skills, no agents) that scans **the invoking project** (`process.cwd()`, never hard-coded Signal paths) for doc rot and prints a human-readable report. **Read-only** — it detects and reports; it never mutates (`--fix`/auto-remediation is Not-Doing, deferred). The name is the one reserved by the BR-1 `/sig:audit` split; `/sig:audit` (Sprint 5) keeps the readiness scorecard.

**Scan scope (the crux — differs from the existing test-suite guard):** `.planning/` (its own docs — STATE, CONTEXT, BACKLOG, BUGS, ISSUES-INBOX, DECISIONS, INDEX, retros, milestone files), **exempting `archive/`** (mirroring migrate's R7 "historical, not links to repair" exemption), plus repo docs (`docs/`, `README`, `analysis/`) and `CLAUDE.md`. The existing `doc-hygiene.js` `WALK_IGNORE` deliberately skips `.planning/`; FR1 parameterizes scan scope so sweep can include it while the test-suite guard is unchanged.

**Portable-vs-Signal-only check split:** portable checks (meaningful in any Signal-managed repo, including a stranger's) run everywhere; Signal-repo-only checks (need a command roster / `plugin.json`) **auto-skip** when `plugin.json` is absent — no false positive on a stranger repo, no no-op noise.

**Acceptance criteria**
- **AC1.1** `/sig:sweep` run in an arbitrary project dir scans *that* project's `.planning/` + docs (asserted against a fixture repo with a distinct baseDir), never Signal's own paths. No hard-coded Signal path in the code path.
- **AC1.2** `.planning/` docs are scanned; `.planning/archive/**` is exempt (a broken link inside `archive/` is NOT reported).
- **AC1.3** With no `plugin.json` present, the Signal-only checks (roster, version, command-frontmatter) are skipped — the report shows only portable findings and states they were skipped, not failed.
- **AC1.4** With `plugin.json` present (Signal itself), all checks run.
- **AC1.5 (read-only)** A sweep run mutates zero files — asserted by a before/after tree hash over the fixture.
- **AC1.6** The report groups findings by severity: **structural** (dead link, index drift, roster/version drift — the things the test-suite guard hard-fails on) vs. **advisory** (bloat, stale-inbox nudges). Exit is report-only; no process-exit contract in v1 (deferred with `--fix`).
- **AC1.7** The command markdown (`commands/sweep.md`) exists and is registered; roster count bumps 17 → 18 and the FR4 hygiene guard + README reference reconcile (release-cut task, tracked).

## FR2 — Sweep check set (D-M5E6-1)

The checks FR1 runs, reusing `doc-hygiene.js` internals where they exist and adding the new ones.

**Portable (any repo):**
- Dead internal links — reuse `checkInternalLinks`. It matches **only `](*.md)` inline links** (line 117) — prose `D-ID` mentions and the ~669 index-resolvable prose refs (M5.E3 catalogue) are out of scope *by construction*, so there is **no prose false-positive flood**. **But scope-widening to `.planning/` is a *validated* change, not assumed-free:** PLAN must confirm against Signal's real `.planning/` that `D-ID` **anchor** links (`](DECISIONS.md#d-…)`) and post-eviction `](archive/…)` links resolve-or-are-exempt, not mis-flagged. This may split the portable set into "ports cleanly" (`[FILL IN]`) vs. "needs `.planning/`-aware validation" (dead-links).
- `[FILL IN]` stubs — reuse `checkFillInStubs`, scope widened (clean port — no anchor/D-ID semantics).
- **`.planning/INDEX.md` freshness (new)** — the **pure compose-and-diff** of AC2.1 (`enumeratePlanningDocs` → `renderPlanningIndex` → diff on-disk), **never** the writing Core; drift is a structural finding.
- **Stale-inbox nudge (new, advisory)** — count undrained `ISSUES-INBOX.md` entries; report as advisory (a "consider draining" nudge), never a hard failure. Deterministic count, no judgment pass.
- **CLAUDE.md bloat (new, advisory)** — size threshold (advisory finding); no auto-edit.

**Signal-only (needs `plugin.json` / roster):**
- Roster counts — reuse `checkRosterCounts`.
- Version consistency — reuse `checkVersionConsistency`.
- **Command-frontmatter freshness (new)** — each `commands/*.md` frontmatter present + consistent with the roster (no stale/missing description).

**Acceptance criteria**
- **AC2.1** Index-freshness uses the **pure compose path** — `enumeratePlanningDocs` → `parseExistingAnnotations(on-disk)` → `renderPlanningIndex(docs, annotations)`, diffed against on-disk `INDEX.md`. It **MUST NOT** call `regeneratePlanningIndexCore` (which `atomicWrite`s on drift — `planning-index.js:323`) or `regeneratePlanningIndex` (its FR7 self-locking wrapper): a read-only sweep cannot invoke a writer (would violate AC1.5). Drifted index → structural finding; fresh → pass; **zero writes** either way.
- **AC2.2** Stale-inbox: N undrained inbox entries → advisory finding reporting N; 0 entries → no finding. Never structural.
- **AC2.3** CLAUDE.md bloat: a CLAUDE.md over the threshold → advisory; under → none. Threshold recorded as a named constant.
- **AC2.4** Command-frontmatter freshness runs only under the Signal-only gate; a command md with a missing/empty frontmatter description is flagged.
- **AC2.5** Every portable check runs on a fixture stranger repo (no `plugin.json`) and produces correct findings; the Signal-only checks are absent from that run.
- **AC2.6** Determinism + offline: two sweeps of the same fixture yield byte-identical reports; no network access (mirrors the existing guard's contract).
- **AC2.7 (`.planning/` dead-link surface validated)** The dead-link check run over Signal's own `.planning/` produces **zero *false* positives** — every flagged link is a genuine dead link; `D-ID` anchor refs and index-resolvable prose refs are not mis-flagged. (Real dead links, if any, are reported as intended — this AC is about no false flood, and forces the FR2 PLAN validation.)

## FR4 — Migrate dangling-gate: flag-not-abort for historical link shapes (B27 + B28, D-M5E6-4)

`applyMigrate`'s dangling-link gate must **flag** (not abort) a *pre-existing* dangling link of these two shapes, matching R7's existing archive-**prose** exemption:
- **B27** — an archive-doc **inline** `](*.md)` link whose target was renamed by an FR6 rename (e.g. `](../../FUTURE-IDEAS.md)` → `ISSUES-INBOX.md`).
- **B28** — an **absolute-path** `](/abs/foo.md)` `.md` link that is a pre-existing dangle inside an evicted closed block (the `scanDanglingLinks` `resolve()` vs `computeLinkEdits` `posix.join()` abs-key divergence).

The gate must still **abort** on a genuinely migrate-*introduced* dangle. The exemption is scoped to archive/closed-block + absolute-path shapes, never a blanket relaxation.

**Acceptance criteria**
- **AC4.1 (B27)** A stranger-repo fixture whose archive doc inline-links an FR6-renamed target **migrates successfully** (flag emitted, no abort/rollback). RED-first: fails against `main`.
- **AC4.2 (B28)** A fixture with a pre-existing absolute-path `.md` dangle inside an evicted closed block **migrates successfully** (flag, no false-abort). RED-first.
- **AC4.3 (gate not weakened)** A genuinely migrate-introduced dangle (relative OR absolute) still aborts + rolls back byte-identical — a non-injected discriminating test.
- **AC4.4** The flag surfaces in `renderDryRun`'s "pre-existing dangling links" section (so the user sees it), consistent between dry-run and apply.
- **AC4.5** B27/B28 move `needs-triage` → `fixed` in `BUGS.md`; the R7-consistency rationale is recorded.

## FR5 — Retro-gate fires on a fresh REVIEW→SHIP flow (B30, P2)

The FR1 retro pre-check (`/sig:ship` Step 0.5) must enforce the retro requirement on a fresh REVIEW→SHIP flow, where today it skips because it runs at `phase: REVIEW` (before the SHIP transition), and B26's STATE-based fallback only fires once `phase === SHIP`.

**Fix approach → PLAN decides between:** (a) `/sig:ship` transitions to SHIP *before* its own FR1 pre-check; (c) re-run the pre-check *post-transition*. **PLAN must not pick blind:** approach (a) means a subsequent halt leaves the project already-transitioned to SHIP (recovery implication); (c) re-run-post-transition avoids that and is the cleaner default. (Approach (b) — broadening `isEpicCloseByState` to "about-to-close" — risks false-fires and is disfavored.)

**Acceptance criteria**
- **AC5.1** On a fresh flow (`phase: REVIEW`, `completed_phases` through the tier's last pre-SHIP phase, retro missing), `/sig:ship` **halts** with `NO_RETRO_FILE` rather than shipping. RED-first: reproduces the M5.E5-SHIP dogfood miss.
- **AC5.2** Once a valid retro exists, the same flow proceeds.
- **AC5.3** No regression to B26's Layer-1/Layer-2 behavior (all `tests/ship-fr1.test.js` cases pass).
- **AC5.4** The chosen approach's recovery implication (already-transitioned-on-halt, if (a)) is documented or avoided (if (c)).
- **AC5.5** B30 moves `needs-triage` → `fixed`.

## FR6 — `_afterRead` seam prototype-pollution guard (B29, defense-in-depth)

The 6 RMW Cores that read `opts._afterRead` must gate on an own-property + `typeof` check (`Object.hasOwn(opts, '_afterRead') && typeof opts._afterRead === 'function'`), or key the seam on a `Symbol`, so an inherited `Object.prototype._afterRead` can never reach the awaited-under-lock path. - **AC6.2** The legitimate test seam (own-property function) still works — the B25 interleaving test is unaffected.
- **AC6.3** Prod path byte-identical (default `undefined`, no caller).
- **AC6.4** B29 moves `needs-triage` → `fixed`.

## FR7 — Concurrency-lock the doc-runtime RMW paths (BACKLOG Sprint-3, D-M5E6-3) — *sequenced last*

Add compare-and-swap/locking to the unlocked read-modify-write paths so two concurrent writers can't lost-update: `checkpoint.js` (`captureCheckpointContext`), `drain.js` (`promoteDrainEntry`, `evictTerminalToLedger`), `retro-index.js` (`regenerateIndex`, `generateMilestoneMetaRetro`), `planning-index.js` (`regeneratePlanningIndex`). These are `atomicWrite` (torn-write-safe) but have no lock, so cross-session concurrent writes can lost-update. **This is a wanted feature** — parallel Claude sessions on one repo do occur (D-M5E6-3).

**The naive fix is unsafe** (documented §9 hazard): `migrate-memory.js:2375` calls `regeneratePlanningIndex` *inside* `applyMigrate`'s coarse `.state.lock`, so self-locking that function re-enters the non-reentrant lock → deadlock. **Required pattern** (the established migrate split): each locked entry = a lock-free **Core** + a self-locking **wrapper**; lock only true command entries; keep inner helpers (`backlog.js`, `applyDispositionToFile`) lock-free. Reuse `tools/lib/file-lock.js`.

**Acceptance criteria**
- **AC7.1** Each of the 6 RMW command entries acquires the lock; a concurrent-writer interleaving test (the B25 `_afterRead` harness shape) shows no lost update against the locked version and a lost update against a lock-free twin.
- **AC7.2 (no deadlock)** `applyMigrate` still completes — the migrate re-entrancy path routes through the lock-free Core, proven by a test that would hang on a naive self-lock.
- **AC7.3** Inner helpers (`backlog.js`, `applyDispositionToFile`) remain lock-free (no nested acquire).
- **AC7.4** Single-session (sequential) behavior byte-identical; no perf regression on the common path.
- **AC7.5** The concurrency-lock item moves out of BACKLOG Sprint-3 residual (marked done).
