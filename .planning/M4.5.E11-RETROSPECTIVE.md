## What changed mid-flight

- **S3.t3 grew a command-surface leg not in its file list.** `formatTierLine` alone is inert — the override never renders until `resume.md`/`status.md` actually read the effective *and* project profiles and pass both. That wiring landed as a labeled `(cont)` commit (`e1d3644`), and it also fixed a latent bug it surfaced: `resume.md`'s documented `nextActionForPhase(state.phase, profile)` example passed the profile object where a `phases_skipped` array was expected — it would have thrown "object is not iterable." Named as a scope addition, not silently folded.
- **Consolidated "Artifact naming" note over per-line edits.** Threading `artifactName` into 6 commands could have meant rewriting every `{phase}-*.md` reference; instead one authoritative note per command governs all of them. A mechanics call made at the altitude gate, not a product change — documented in the S2.t2 commit.

## What assumptions broke

- **"999 green ⇒ FR3 works at runtime" — false, and the advisor caught it.** A fully green suite proves the four helpers in isolation but says nothing about whether the *rigor consumers* call `readEffectiveProfile` or re-read the project `PROFILE.md` directly. The headline FR3 AC — "a SKETCH Epic in a FULL project runs its phases at SKETCH" — could be false at runtime with every test passing. Only a grep of every `readProfile(` call site (and confirming `execute.md` threads the effective `profile` into `dispatchTaskWithState`, which keys tier off `profile.tier`) actually proved it. It came back clean — but the check was load-bearing, not decorative.
- **The `artifactName` round-trip test is self-consistency, not correctness-vs-convention.** Because `resolveArtifactPath` tries both `1-REQUIREMENTS.md` and `REQUIREMENTS.md`, a *wrong* linear REQUIREMENTS mapping would round-trip cleanly (write it → resolve finds it). The round-trip alone would have blessed a name that forks every existing linear project. Fixed by pinning the exact output with equality assertions, not just the round-trip.

## What surprised us

- **Both review agents independently flagged the read/write Epic-ID regex asymmetry — and the "fix" would have broken back-compat.** The security auditor offered narrowing `resolveArtifactPath`'s permissive pattern-0 to the strict regex as optional tidiness. Applied, it would have made `/sig:resume` unable to resolve the `v0.1.6-*.md` artifacts sitting in *this very repo* (v0.1.6 fails the strict shape). A plausible-sounding cleanup that breaks the thing — caught only by checking the suggestion against real on-disk state, not by reasoning. Surfaced and rejected-with-reason rather than silently applied.
- **E11 could not be its own zero-hand-edit dogfood.** Its Epic was opened by hand (`current_epic: M4.5.E11` hand-typed at DISCUSS) *before* the write-half it builds existed. The bootstrap-as-evidence AC resolved to "prove the new machinery via an e2e fixture opening a *fresh* Epic (`setCurrentEpic('M4.5.E77')`, zero hand-edited STATE)," not "re-run E11 clean" — which would have been meaningless mid-flight. Named honestly in the S4.t1 attestation.

## What we'd do differently

- **Treat command-surface wiring as part of the render task, not a follow-on.** S3.t3's helper was done but the feature wasn't visible until the commands passed it data. Scoping the render AC to include the command wiring from the start would have avoided the `(cont)` split.
- **Record phase completion per-phase.** `completed_phases` was under-recorded at EXECUTE close (EXECUTE wasn't appended), caught + corrected at the VERIFY close. A disciplined `transitionPhase` at each phase boundary — rather than a hand-edit at the next close — would keep the resume "N/7 done" count honest without catch-up.

## What to feed back into Signal

- **The `markFresh` 1-commit staleness floor is a recurring papercut.** Every phase close leaves `last_updated_commit` pointing one commit behind HEAD (the freshness-write commit references the not-yet-existing commit it lives in), so `/sig:resume` always shows "1 commit behind" right after a close. It bit at S1's close (the very first resume this session). Worth a FUTURE-IDEAS: teach `isStateStale` to ignore a pure-`markFresh` STATE-only commit, or annotate the resume banner so a 1-behind reads as "the freshness marker, benign" rather than drift.
- **The "consolidated naming note" is a reusable pattern** for threading a JS helper into command-markdown prose — worth documenting as a convention for future command-prose retrofits (M5's v2 ports will do a lot of this).

## Anti-rationalization moment

The temptation was to declare FR3 done on "the unit tests are green and the e2e passes against disk." That's the exact rationalization the strict tier exists to catch — *tests verify the helper, not that the system wires it in*. The advisor's push ("a green suite says nothing about whether rigor consumers call the effective profile") was the moment; grepping every `readProfile(` call site and confirming the effective profile flows all the way into `dispatchTaskWithState` is what actually earned the FR3 checkmark. Same discipline in REVIEW: rejecting the auditor's tidiness suggestion after testing it against real repo artifacts, instead of applying a clean-looking change on trust.
