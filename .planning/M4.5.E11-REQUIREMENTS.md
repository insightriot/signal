## Objective

Make **Epic mode first-class**. Today Signal has two unreconciled modes: **linear** (what the phase commands implement — one `calibrate→ship` pass, phase-named artifacts, no Epic concept) and **Epic** (what Signal-on-Signal runs *by hand* — Milestones→Epics, each Epic runs its own `discuss→ship`, `{EpicID}-*.md` artifacts, `current_epic` hand-typed). This Epic teaches the commands to drive off the locked Epic/Milestone/Slice/Task vocabulary (M4.t18) instead of dogfooding it manually. Per-Epic calibration — an Epic inside a FULL project honestly being SKETCH — falls out of Epic-scoping. FR1 from E10 (`resolveArtifactPath`) is the forward-compatible **read-half**; E11 supplies the **write-half**.

> **Bootstrap-as-evidence.** This Epic's own DISCUSS did the STATE transition + Epic-ID assignment by hand (`current_epic: M4.5.E11` hand-typed; STATE prose hand-written). Every by-hand step here is a concrete requirement for what E11 automates. PLAN should treat this session's manual bootstrap as the canonical worked example.

## Scope

**In (tight core — D-E11-1):**
- Epics are first-class: commands create/track them, assign IDs, auto-populate `current_epic`, write Epic-scoped artifacts.
- Per-Epic calibration: an Epic carries its own tier, overriding the project PROFILE for that Epic's phases.

**Out (parked, not cut — D-E11-1):** the FUTURE-IDEAS "Multi-feature project lifecycle" layer — `features[]` block in STATE, per-feature `.planning/features/{slug}/` subdirs + PROFILE overrides, feature-aware `/sig:status` + `/sig:resume`. Gated on the second-dogfood evidence that entry's own "resolve by" names (BR-9).

## Functional Requirements

### FR1 — Epic creation & `current_epic` write-half
A user can open/switch an Epic; the tooling records the Epic ID and writes it to STATE `current_epic` **automatically** (no hand-typing). `current_epic` tracks the live Epic through its `discuss→ship`, and rolls/clears at Epic close or when the next Epic opens. The E10 read-half (`resolveArtifactPath`, `EPIC_ID_RE`) consumes it unchanged.
- *AC-seed:* opening an Epic sets `current_epic`; no phase command requires a hand-edited STATE; the ID passes `EPIC_ID_RE`.

### FR2 — Epic-scoped artifact naming & resolution
When an Epic is active, phase commands write `{EpicID}-{ARTIFACT}.md` (RESEARCH / REQUIREMENTS / PLAN / VALIDATION / VERIFICATION / REVIEW / PROGRESS / RETROSPECTIVE); resume/status resolve them via the E10 read-half. Adopts the established Signal-on-Signal convention.
- *AC-seed:* an Epic-mode PLAN writes `{EpicID}-PLAN.md`; `/sig:resume` finds it with no "artifact not found" papercut.

### FR3 — Per-Epic calibration
An Epic can carry its own tier (SKETCH / FEATURE / SPIKE / FULL) that overrides the project PROFILE **for that Epic's phases only**. Default composition: Epic tier wins for the Epic's phases; the project PROFILE is the fallback when no Epic override exists. Storage location + composition mechanics = PLAN-open.
- *AC-seed:* an Epic calibrated SKETCH inside a FULL project runs its phases at SKETCH rigor; the project PROFILE is unchanged.

### FR4 — Linear/Epic mode detection (linear stays default)
Commands detect which mode they're in (signal: an active Epic / non-null `current_epic`) and branch accordingly. **Linear mode is the unchanged default** — a project with no Epic runs exactly as today with phase-named artifacts.
- *AC-seed:* a linear-mode project (no `current_epic`) writes/resolves `{phase}-*.md` with byte-identical behavior to pre-E11.

## Non-Functional Requirements (mirrors D-E10-10 / D-v016-6)

- **Additive / opt-in (D-E11-3):** linear mode keeps working untouched; no migration of existing linear-mode projects.
- **Back-compat:** no `schema_version` bump anticipated (confirm at PLAN); existing `{phase}-*.md` projects and the 894-test suite stay green.
- **No new runtime dependencies.**
- **Platform-agnostic:** Signal runs in stranger repos where the STATE hook fires — nothing macOS/`~/.claude`-scoped in the flow paths.
- **Fail-open:** any new Epic-mode check that could break `/sig:resume` or `/sig:status` degrades gracefully (never throws on a linear-mode or malformed STATE).
- **Strict Nyquist (FULL):** every acceptance criterion maps to a RED-first test.

## Assumptions to validate in PLAN

1. The E10 `resolveArtifactPath` read-half already covers every Epic-ID shape E11 will emit (`EPIC_ID_RE = /^[A-Za-z0-9._-]+$/` — verified permissive; `M4.5.E11`, `M5.E0` pass).
2. A single scalar `current_epic` is sufficient for tight core (no `features[]`; one active Epic at a time).
3. Per-Epic tier can be represented **without** a `schema_version` bump — e.g., a STATE field or an Epic-scoped PROFILE file. Confirm which, and whether it forces a migration.
4. Epic-creation UX can be additive without a new required command (a required new command would change the install contract / validator's `REQUIRED_COMMANDS`).

## Open for PLAN (design/plumbing — sensible defaults, not product gates)

- **(a) Epic-creation UX** — dedicated `/sig:epic <name>` meta-command vs a flag on `/sig:discuss`/`/sig:new-project` vs auto-detect-and-offer when a phase command runs with no active Epic. *Surfaced to Brett for optional input at the DISCUSS gate; else PLAN decides.*
- **(b)** `current_epic` write points — which command(s) own the write, and the close/roll semantics.
- **(c)** Per-Epic calibration storage (Epic-scoped PROFILE override vs STATE field) + composition rules with the project PROFILE.
- **(d)** Mode-detection signal + where it's read.

- Multiple Epics in flight simultaneously — one active Epic at a time for tight core.
