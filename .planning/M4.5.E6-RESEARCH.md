### 1.5 — Implementation pattern: orchestrator-wrapped, not agent-internal

D3's auto-protocol calls (`setCurrentTask` at task start, `clearCurrentTask` at task end) live in the `commands/execute.md` orchestrator, NOT in `agents/executors/executor.md`. Three reasons (all from phase-researcher):

1. **Crash resilience (D9 + D12).** Orchestrator is still alive if the agent crashes; can call `clearCurrentTask({status: aborted})` to prevent orphan in_progress entries.
2. **Tool scoping.** Executor agent's tool budget (Read, Write, Edit, Bash, Grep, Glob) doesn't need to know about state mutation; orchestrator already manipulates `.planning/`.
3. **Tier-gating (D9).** Orchestrator reads PROFILE.md at preamble; has `gate_strictness` in hand for the failure-policy branch. Agent would need re-read or prompt-parameter.

**Trade-off acknowledged:** More wiring in `execute.md`. Executor agent's "6-step process" stays 6 steps; state mutation is *outside* it via a new wrapper `dispatchTaskWithState(baseDir, task, profile)`.

## 3. External findings (libraries + prior art)

### 3.1 — Frontmatter parsing: hand-roll, don't add `gray-matter`

`gray-matter` pulls 5 transitive deps including `js-yaml`. Hand-rolled splitter using existing `yaml` is ~20 lines:

```js
import { parse as parseYaml, stringify as stringifyYaml } from 'yaml';

const FM_RE = /^---\r?\n([\s\S]*?)\r?\n---\r?\n?([\s\S]*)$/;

export function parseFrontmatter(raw) {
  const m = raw.match(FM_RE);
  if (!m) return { data: null, body: raw };
  let data;
  try {
    data = parseYaml(m[1], { schema: 'core' });
  } catch (e) {
    throw new StateSchemaError(`STATE.md frontmatter parse error: ${e.message}`);
  }
  if (data === null || typeof data !== 'object' || Array.isArray(data)) {
    throw new StateSchemaError('STATE.md frontmatter must be a YAML mapping');
  }
  return { data, body: m[2] };
}

export function stringifyFrontmatter(data, body = '') {
  const yaml = stringifyYaml(data, { lineWidth: 0 });
  return `---\n${yaml}---\n${body}`;
}
```

Handles CRLF, empty body, missing trailing newline. Doesn't handle BOM or custom delimiters (Signal doesn't need either).

### 3.2 — Schema versioning: follow Jest Snapshot's pattern

Surveyed Terraform, Jest Snapshot, kubectl, gh CLI, git internals. **Jest Snapshot's single-integer header marker** is closest fit:

- Single integer `schema_version: 1` (D1 locks this)
- Fail-closed on unknown future versions (D14 locks this)
- Auto-migrate forward only (D4 locks this)
- No migration chains until v3 ships (YAGNI; one-version-so-far reality)
- Migrations are pure functions, idempotent (`if (data.schema_version === N) return data` guard at top)

Terraform's MigrateState chain is over-engineered for Signal's stage. gh CLI's "no version field" is the anti-pattern D14 explicitly avoids.

### 3.3 — Prior art for session-handover state

git's rebase/cherry-pick sequencer is closest UX analog. Patterns to borrow:

- **Status command surfaces in-flight state first** — `/sig:status` should lead with `current_tasks[]` if any
- **Continue command is explicit** — `/sig:resume` is Signal's analog
- **Abort path always available** — `/sig:checkpoint`'s orphan-detection (D12) is Signal's analog to `git rebase --abort`
- **ORIG_HEAD pattern** — store `previous_tasks` for a future `/sig:checkpoint --undo`; logged to FUTURE-IDEAS, not E6

### 3.4 — Atomic writes: extract from add.js, don't adopt `write-file-atomic`

`add.js`'s pattern is gold-standard. `write-file-atomic` adds 3 deps for features Signal doesn't need (uid/gid, signal-exit cleanup). `steno` is wrong tool (designed for 100s of writes per second). Extraction is the right move; library adoption is not.

### 3.5 — D3 direct-call is correct; reject git post-commit hook approach

Hook approach fails Signal's specific context:
- Installation footprint (where does Signal write the hook? what if Husky is already installed?)
- `.git/hooks/` not in version control (fresh clone = no hook)
- Hook chaining is fragile (data loss vs complex error semantics)
- Hook runs are NOT durable (git swallows hook crashes silently)
- Hook does not know which task it's for (no `current_task.id` context)

D3 direct-call is the right design. Plan notes "post-commit hook considered and rejected" with link to this research.

**Migration flow (D4 + D14):**

```
readState(baseDir):
  raw = readFile('.planning/STATE.md')
  parsed = parseFrontmatter(raw)
  if parsed.data === null:
    // Case 3 in D14: no frontmatter → triggers migration on next write
    return { ...legacyParse(parsed.body), _schema: 'legacy' }
  if !parsed.data.schema_version:
    // Case 4 in D14: frontmatter exists but no version → refuse to auto-upgrade
    throw new StateSchemaError('STATE.md has frontmatter but no schema_version...')
  if parsed.data.schema_version !== 1:
    // Case 2 in D14: unknown future version → fail closed
    throw new StateSchemaError('STATE.md was written by a newer Signal version...')
  // Case 1: parse normally
  return parsed.data

upgradeStateFile(baseDir):  // called by setCurrentTask et al. on first write to legacy STATE.md
  raw = readFile('.planning/STATE.md')
  body = raw  // preserved verbatim
  newFrontmatter = {
    schema_version: 1,
    phase: inferPhase(raw),  // grep for "## Current Phase\n(.+)" else 'EXECUTE'
    current_epic: null,
    current_wave: null,
    current_tasks: [],
    completed_phases: inferCompletedPhases(raw),
    blockers: [],
    last_decision_at: null,
    last_updated_commit: getMostRecentStateCommit(),
    last_updated: new Date().toISOString(),
  }
  newContent = stringifyFrontmatter(newFrontmatter, body)
  await atomicWrite('.planning/STATE.md', newContent)
  console.error('📋 STATE.md upgraded to schema_version 1. Original content preserved below frontmatter.')
```

**Test-first ordering for S1:**
1. `readState` legacy-detection (returns `_schema: 'legacy'` for no-frontmatter)
2. `readState` strict three-way (D14 cases 2/3/4)
3. `upgradeStateFile` (preserves body verbatim)
4. `setCurrentTask` (appends to array; triggers upgrade if legacy)
5. `clearCurrentTask` (removes by id; idempotent if already removed)
6. `getCurrentTasks` + `detectOrphans` (read-side)
7. `isStateStale` (mocked git output)
8. Remaining: addBlocker/clearBlocker, appendDecision, markFresh

Pair (1+2+3) is the must-have — every other helper depends on the upgrade-on-first-write contract.

### 4.3 — S3: Executor agent + execute.md orchestrator

