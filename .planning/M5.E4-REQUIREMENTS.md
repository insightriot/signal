## Functional Requirements

### FR1 — B19 (P2): migrate must never silently clobber a hand-curated INDEX.md
*The one live P2 — a released defect on the **default** migrate path for every external stamp-null project.* `applyMigrate`'s tail `regeneratePlanningIndex` wipes an old-format curated INDEX to `_(note pending)_` because `parseExistingAnnotations` can't read the old table format. **Chosen fix (D-M5E4-2): detect old-format, skip + flag** (not the fragile auto-migrator, not consent-to-clobber).
- **AC1.1** — when INDEX.md is present and in the old/unportable format, `applyMigrate` **skips** the auto-regen: the curated file is left byte-identical.
- **AC1.2** — the skip is surfaced in the dry-run output *and* the apply summary, with an explicit pointer to run `/sig:index` manually to upgrade the format.
- **AC1.3** — a portable/absent/new-format INDEX regenerates exactly as today (no behavior change for the common path).
- **AC1.4** — RED-first test proves an old-format curated INDEX survives a full migrate with 0 notes lost; a new-format INDEX still regenerates.

### FR2 — Path-confinement + scan-integrity hardening (B14, B18, B15, B22)
- **B14** (P3, security) — apply the proven `realpathNearestExisting` + `assertRealInsidePlanning` guard (from `migrate-memory.js`, commit `ab2242d`) to the three remaining symlink-blind write/read gateways: `evict.js:350-351`, `add.js:830`, `resume.js:74` (read-side, lower severity — still in scope).
- **B18** (P3) — `regeneratePlanningIndex` / `enumeratePlanningDocs` must `resolve(baseDir)` at entry (or key via `path.relative`) so a `'.'` baseDir no longer chops keys / corrupts INDEX.md.
- **B15** (P3) — the migrate dangling-link scanners (`scanDanglingLinks` et al.) must not let the 1 MB `FILE_SCAN_CEILING` silently defeat the blocking gate: scan the whole file for dangle detection, or **flag** when truncation occurs so a silent pass is impossible.
- **B22** (P3) — `doc-hygiene.js checkInternalLinks` must bound the resolved link target to repo root before `existsSync`/`readFile` (no disk touch outside the walk root).
- **AC2.x** — each carries a RED-first test: a directory-symlink escape attempt is rejected (B14); `regeneratePlanningIndex('.')` produces correct keys + no corruption (B18); a >1 MB single doc with a dangle past the cap is caught or flagged, never silently passed (B15); a `](../../../x.md)` target is refused resolution outside root (B22).

### FR3 — Migrate rollback & perf polish (B16, B21)
- **B16** (P3) — a rolled-back `--apply` in git mode must **not** leave a lingering `pre-migrate-memory-<stamp>` tag: clean it up in the rollback catch (or make it idempotent per-run). AC: after a forced-rollback apply, no stray tag remains; a *successful* apply still leaves the intended revert pointer.
- **B21** (P3) — build the D-ID map **once** before the FR5 anchor-gate loop and resolve in-memory (the eviction has already written to disk before the gate → the map is stable across the loop); leave the mechanical-fresh `resolveDecisionId` for standalone callers. AC: anchor-gate resolution result is unchanged; `buildDecisionIdMap` is called once, not per-ID.

### FR4 — Resume-trust banner accuracy (B6)
`isStaleVsOrigin` / the origin-drift banner must gate on a **genuine** remote push (`git rev-list HEAD..origin/main > 0` — local actually behind origin), not merely on STATE's `last_updated_commit` lagging HEAD by the bookkeeping "+1". Same correction for the local-stale side of the "+1" false-positive.
- **AC4.1** — right after a clean checkpoint+push where `HEAD == origin/main`, neither the origin-drift nor the local-stale banner fires (the benign "+1" is silent).
- **AC4.2** — a real remote push (local genuinely behind origin) still fires the origin-drift banner. RED-first on both.

### FR5 — Concurrency-lock the doc-runtime RMW paths *(the isolated refactor slice)*
The unlocked read-modify-write paths (`checkpoint.js captureCheckpointContext`, `drain.js promoteDrainEntry`/`evictTerminalToLedger`, `retro-index.js regenerateIndex`/`generateMilestoneMetaRetro`, `planning-index.js regeneratePlanningIndex`) are atomic-write-safe but have no compare-and-swap/lock → concurrent cross-session writers can lost-update. **Fix via the prescribed safe split, NOT the naive `file-lock.js` reuse** (which re-enters the non-reentrant `.state.lock` via `migrate-memory.js:2375` and deadlocks migrate — the documented §9 hazard): split each locked entry into a **lock-free core + a self-locking wrapper**; lock only true command entries; keep inner helpers (`backlog.js`, `applyDispositionToFile`) lock-free.
- **AC5.1** — `/sig:migrate-memory` still completes (no re-entrant deadlock) — the existing migrate suite stays green.
- **AC5.2** — a test demonstrates two concurrent writers to a doc-runtime RMW path no longer lost-update (serialized via the new wrapper lock).
- **AC5.3** — wave-executor paths are unchanged (these are orchestrator-only; single-session writes stay sequential).
- **This is the risky slice — it gets its own PLAN slice and does not ride a parallel quick-fix wave** (D-M5E4-3).
