## Context — what M4.5.E9 is solving

Two pain points, surfaced together 2026-05-25:

1. **Retros got skipped because context cleared.** A retrospective intended to follow an Epic close was never written because the conversation context was cleared before SHIP completed. The learning evaporated. There is no enforcement mechanism today — `commands/ship.md` contains zero references to "retrospective."
2. **No formal retro exists.** What's been treated as "the normal summary RETROSPECTIVE" is purely ad-hoc — written by whoever closed the phase, in whatever shape they felt like, with quality varying widely. There's no template to follow, no required sections, no minimum content bar.

The combination means: every Epic's most valuable learning artifact (what surprised us, what assumptions broke, what to feed back into Signal's own design) is at the mercy of context-window timing and individual session discipline.

M4.5.E9 fixes this with two slices:
- **S1** — SHIP hard-blocks until a `RETROSPECTIVE.md` file exists on disk with a tier-appropriate template. Stubs are backfilled for already-shipped M4.5 Epics so the gap is visible.
- **S2** — `RETROSPECTIVES.md` index at `.planning/` root catalogs all retros (stub or complete), so future-you can traverse them without loading every file.

This is the smallest possible ship that closes the failure mode and establishes the pattern. Workstreams 3 + 4 from the FUTURE-IDEAS entry (full wiki restructure + doc-runtime + migration tooling) are deferred to **M5.E1** with its own DISCUSS — those decisions are too consequential to bundle with the small enforcement Epic.

## Functional Requirements

### FR1 — SHIP hard-block check

`commands/ship.md` adds a pre-check at the start of the SHIP phase: read the current Epic ID from `STATE.md` frontmatter (`current_epic`), compute the expected retro path (`.planning/M4.5.E{N}-RETROSPECTIVE.md` per D-E9-6 / generalized to whatever Epic ID format is current), and verify the file exists.

If the file is missing, SHIP halts with: *"M4.5.E9 enforcement: `RETROSPECTIVE.md` required before SHIP can close. Create `.planning/M4.5.E{N}-RETROSPECTIVE.md` from the tier-appropriate template (see `references/retrospective-template.md`), then re-invoke `/sig:ship`."* No `--no-retro` bypass flag exists (D-E9-3).

**Acceptance criteria:**
- **AC1** — SHIP halts with the documented message when no retro file exists. (Test: fixture project, invoke `/sig:ship`, assert exit code + message.)
- **AC2** — SHIP halts even with `--force` or other flag combinations. No bypass exists. (Test: try every flag in the command's spec, assert all fail.)
- **AC3** — SHIP proceeds normally when the retro file exists and passes FR3's sanity check.

### FR2 — Tier-aware retrospective template

`references/retrospective-template.md` (new file) defines four template shapes keyed by PROFILE.md tier. Templates are markdown with section headings that downstream FR3 validates against.

- **FULL** — full template: Timeline, What changed mid-flight, What assumptions broke, What surprised us (positive + negative), What would be done differently, What to feed back into Signal's own design, Anti-rationalization moment (at least one "we almost rationalized away X but didn't"), Links to commits/PRs/artifacts.
- **FEATURE** — medium template: Timeline (briefer), What surprised us, What would be done differently, What to feed back into Signal, Links to commits/PRs.
- **SPIKE** — exploratory template: What we learned, Whether the spike resolved its question, What to do next (build / abandon / continue).
- **SKETCH** — 3-question stub: What worked, What didn't, Anything to feed back into Signal.

**Acceptance criteria:**
- **AC4** — Each tier's template renders with the documented section headings. (Test: read template file, parse markdown, assert section list matches per-tier spec.)
- **AC5** — Template file is discoverable from `commands/ship.md` via documented relative path. (Test: parse `commands/ship.md`, find template reference, resolve path, assert file exists.)

### FR3 — Minimum-content sanity check

`tools/lib/retrospective.js` (new) exports `validateRetroContent(content, tier)` which checks:

1. File is non-empty (≥ a tier-specific minimum byte count to be locked in PLAN; rough cut: SKETCH ≥ 200 bytes, FEATURE ≥ 800, FULL ≥ 2000).
2. All template section headings are present (no missing required sections).
3. No section is content-empty (heading present but no body text — guards against "I typed `## What surprised us` and called it done").
4. `[FILL IN]` markers are flagged but not blocking for the M4.5.E9 retro itself — they DO block for retros that should be complete. Behavior for stub vs. complete retros: see FR4 / FR5.

**Acceptance criteria:**
- **AC6** — `validateRetroContent` rejects empty files. (Unit test.)
- **AC7** — `validateRetroContent` rejects files missing any required section heading for the given tier. (Parametrized unit test, all 4 tiers.)
- **AC8** — `validateRetroContent` rejects files with required-section headings but empty bodies. (Unit test.)
- **AC9** — `validateRetroContent` passes well-formed retro files. (Per-tier fixture test.)

### FR4 — Stub backfill for already-shipped M4.5 Epics

A one-shot mechanism (CLI subcommand under `/sig:ship` or a separate `tools/backfill-retros.js` — decision deferred to PLAN) iterates through M4.5 Epics that have already shipped (E1, E2, E3, E6, E7 per CONTEXT.md as of 2026-05-25), and for each one generates `.planning/M4.5.E{N}-RETROSPECTIVE.md` if it doesn't already exist.

Each stub contains:
- Auto-extracted links to the Epic's PLAN, REVIEW, PROGRESS, VERIFICATION artifacts (whichever exist on disk).
- Auto-extracted commit range for the Epic (parsed from git log filtered by file paths touching `M4.5.E{N}-*.md`).
- The tier-appropriate template's section headings (per FR2; FULL tier for all M4.5 retros since project is FULL tier).
- `[FILL IN]` markers in every reflection section.
- A header note: *"Stub generated 2026-05-DD by M4.5.E9 backfill. Fill in opportunistically — index in `RETROSPECTIVES.md` surfaces stub-vs-complete status."*

**Partial-Epic handling (deferred to PLAN):** E1 and E2 shipped only some slices (E1.S1 + E1.S2-A; E2.S1) while the rest are shelved or pending. Per-Epic granularity (D-E9-5) means one retro per Epic — but a partial Epic doesn't fit cleanly. **PLAN decides:** generate stub for the shipped-portion only with explicit note ("Epic incomplete; this retro covers shipped slices only"), or defer those Epics' stubs until they fully close. Recommendation tilts toward the first; final call in PLAN.

**Acceptance criteria:**
- **AC10** — Backfill generates stubs for all currently-shipped M4.5 Epics that don't already have a retro file. (Integration test: fixture M4.5 state, run backfill, assert per-Epic stub files exist.)
- **AC11** — Each stub contains auto-extracted artifact links that resolve to real files. (Test: parse generated stub, follow links, assert each target exists.)
- **AC12** — Backfill is idempotent — re-running produces no changes if all stubs already exist. (Test: run twice, diff after second run, assert empty.)
- **AC13** — Backfill never overwrites a retro file that's been edited (has content beyond the stub markers). (Test: edit a stub, re-run backfill, assert no overwrite.)

### FR5 — RETROSPECTIVES.md index (S2)

`.planning/RETROSPECTIVES.md` is the root-level index for all per-Epic retros. Structure mirrors `MEMORY.md`'s shape — one-line entries per Epic, sorted reverse-chronologically (newest first), with a status marker (stub vs. complete) and a one-line hook.

Example shape:
```markdown
# Signal — Retrospectives Index

- [M4.5.E9 — Retro Foundations](M4.5.E9-RETROSPECTIVE.md) — *complete* — established the SHIP enforcement gate; first dogfooded retro.
- [M4.5.E7 — Synthesizer prose-quality + install-UX](M4.5.E7-RETROSPECTIVE.md) — *stub* — backfilled by M4.5.E9; reflection [FILL IN].
- [M4.5.E3 — Public-facing docs rewrite](M4.5.E3-RETROSPECTIVE.md) — *stub* — backfilled by M4.5.E9; reflection [FILL IN].
- ...
```

**Index maintenance (auto vs. hand-curated — deferred to PLAN):** The index could be auto-generated from a script that walks `.planning/M4.5.E*-RETROSPECTIVE.md` files and produces the listing, OR it could be hand-curated and validated by a link-health check, OR a hybrid. **PLAN decides** — leaning toward hybrid (auto-structural list + hand-written hook line per Epic), but final call is PLAN's.

**Acceptance criteria:**
- **AC14** — `RETROSPECTIVES.md` exists at the documented path after S2 ships. (Test: assert file exists.)
- **AC15** — Index lists every retro file in `.planning/M4.5.E*-RETROSPECTIVE.md` (and the non-namespaced equivalent for future milestones). (Test: enumerate retro files, assert each appears in index.)
- **AC16** — Each index entry's link resolves to the actual retro file. (Test: parse links, assert each target exists.)
- **AC17** — Each index entry surfaces the stub-vs-complete status accurately based on retro file content (presence of `[FILL IN]` markers in reflection sections → "stub"; absence → "complete"). (Test: per-fixture.)

### FR6 — Optional milestone-close meta-retro

When SHIP closes the FINAL Epic of a Milestone (heuristic: no further planned Epics in `MILESTONE-{N}.md`'s status snapshot), SHIP additionally prompts: *"Milestone {N} appears complete. Generate optional milestone-close meta-retrospective at `.planning/MILESTONE-{N}-RETROSPECTIVE.md`? [y/N]"*

If yes, generate a stub that pulls together all Epic retro headers + a synthesis template for cross-Epic patterns. If no, exit normally. **No hard block.**

**Acceptance criteria:**
- **AC18** — Final-Epic detection works correctly for at least one fixture milestone. (Test: fixture MILESTONE.md with all-Epics-shipped status, invoke SHIP, assert prompt fires.)
- **AC19** — User-rejected meta-retro prompt doesn't block SHIP completion. (Test: simulate "n" response, assert SHIP completes.)

## Deferred to PLAN

The following design questions are real but don't block DISCUSS — PLAN's job to resolve:

1. **Backfill mechanism location.** CLI subcommand under `/sig:ship` (e.g., `--backfill-retros`) vs. standalone `tools/backfill-retros.js` vs. `/sig:retros-backfill` slash command. Trade-off: discoverability vs. command surface area.
2. **Partial-Epic backfill behavior (FR4).** Generate retro for shipped-slices-only with explicit note, vs. defer until Epic fully closes. Recommendation tilts toward the first.
3. **Index maintenance pattern (FR5).** Auto-generated from retro files, hand-curated with link-health check, or hybrid (auto-structural + hand-written hooks). Recommendation tilts hybrid.
4. **Per-Epic vs. per-Slice SHIP behavior (FR1 interaction).** Today, SHIP fires per-Slice in some Epics (e.g., E1.S1 shipped 2026-05-15) and per-Epic in others (E3 shipped as one event). D-E9-5 locks per-Epic granularity for retros — PLAN must decide whether the FR1 hard-block fires on every SHIP invocation, or only on Epic-close SHIP. If the former, Slice-level retros become de-facto required despite D-E9-5; if the latter, SHIP needs to detect "Epic-close" vs. "Slice-close." This is the load-bearing implementation decision.
5. **Tier-specific minimum-content thresholds (FR3 byte counts).** Rough cuts in FR3 are placeholders. PLAN should pick values based on rendered token cost of the templates themselves (template floor + reasonable user fill).
6. **Template section heading exact wording.** FR2 lists conceptual sections. PLAN locks the exact heading text (matters because FR3 validates against them).
7. **Promotion of D-E9-* decisions to `DECISIONS.md`.** Per the E3 follow-up flagged in CONTEXT.md (2026-05-25 "Recommended follow-ups" #1) — decisions should live in DECISIONS.md, not requirements files. PLAN should promote.
8. **Enforcement mechanism — Claude Code hook vs. command-internal check vs. both.** *Added post-DISCUSS 2026-05-25 after reviewing Anthropic's "Claude Code in large codebases" post (2026-05-14):* the post argues that hooks enforce rules more deterministically than command-internal logic ("hooks enforce the rules deterministically and produce more consistent results than relying on Claude to remember an instruction"). FR1's hard-block could be implemented as: (a) inline check in `commands/ship.md` (current implicit assumption — bypassable by skipping `/sig:ship` and editing STATE.md directly, or by Claude "forgetting" the check under context pressure), (b) a Claude Code hook bound to SHIP-phase invocations or STATE.md writes (harder to bypass, harness-managed), (c) both layered (defense-in-depth). **Important corollary:** a `Stop` hook (fires at session end regardless of phase) would address the ORIGINAL failure mode more directly than SHIP-phase enforcement does — the motivating incident was context cleared *before* `/sig:ship` was invoked, meaning a SHIP-phase check never fires for the exact scenario it's meant to prevent. PLAN should evaluate hook-based and stop-hook variants alongside the command-internal option. Reference: `references/hook-patterns.md` if it exists; otherwise this Epic may also need to scope writing one.

## Anti-rationalization

| Temptation | Check |
|---|---|
| *"Skip the milestone meta-retro (FR6) to keep scope tighter."* | No — it's an optional, no-hard-block addition that lets future-you synthesize across Epics. Costs ~50 lines of code; pays back the first time a milestone closes. |
