## Architecture decisions (from research + DISCUSS)
1. **Decompose by file-contention, not by FR.** Two files are each touched by ~5 tasks ‚Äî `migrate-memory.js` (B15-scanners, B16, B19, B21, FR5) and `planning-index.js` (B19, B21, FR5). Parallel executors on those files would collide. So: **Wave 1 = genuinely disjoint files in parallel; Wave 2 = the migrate/index cluster serial in one context; Wave 3 = FR5 last.** (Advisor's #1 ‚Äî the thing that breaks EXECUTE if wrong.)
2. **FR5 wraps already-corrected code.** The concurrency-lock refactor runs *after* the migrate/index bug fixes (B19/B21 restructure the same functions), so the core+wrapper split operates on green, corrected code ‚Äî not a moving target.
3. **Reuse, don't rebuild.** B18 is already fixed (`8f1dda8`); B19's mechanism is shipped (`497267a`) but holed; B8's regression exists but one-sided. Plan targets the *residuals*, not greenfield.
4. **RED-first against `main` is mandatory** ‚Äî B19 shipped broken *because* its fixture was defanged. Every fix proves RED against current `main` before the GREEN.
