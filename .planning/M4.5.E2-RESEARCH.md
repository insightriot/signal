## 2. Domain research — productivity/UX prior art

**Canonical pattern:** capture-to-inbox first, route later. Validated by Things 3 Quick Entry, Obsidian Inbox plugin (tmfelwu, Zachatoo variants), Reflect, Notion Quick Capture, GTD literature (David Allen, facilethings.com, super-productivity.com).

**Key load-bearing findings:**
- **GTD principle: mixing capture with routing is the textbook anti-pattern.** Every routing question violates it. Signal's design (FUTURE-IDEAS default + planning-phase promotion) is the canonical answer; locked in research.
- **AskUserQuestion adaptive shape** (Jeremy Watt, Claude Code docs): if a question's answer is inferable, skip it. Don't ask "Is this actionable now?" if input contains "TODO" or "fix bug" — infer.
- **POSIX `O_APPEND` makes byte-races moot** for single-user CLI (nullprogram.com source). Focus instead on positional failures (YAML, code fences) — atomic-write discipline still required for crash-recovery.
- **Vocabulary teaching is not done at capture time** in any mainstream productivity tool. Linear/Jira teach by post-capture review (Triage, Inbox processing). Signal must absorb the vocabulary burden via plain-English routing questions, surface vocabulary only on confirmation.
- **Missing back-half (cross-Epic insight):** `/sig:plan` must explicitly reference reviewing FUTURE-IDEAS.md and promoting/culling entries. Without this loop being documented, FUTURE-IDEAS.md becomes a write-only graveyard. Linear's daily Triage ritual is the model. **This adds a sub-task to M4.5.E2's exit criteria** — see § 6 below.

### Anti-patterns from prior art (lock in command's anti-rationalization table)

1. **Don't ask for priority/criteria/category at capture.** GSD's `add-todo` is the heaviest path; it turns a 2-second thought into 30-second form. Inbox-style structuring happens at `/sig:plan` time.
2. **Don't preview before write.** Cookiecutter/`gh --web` do this for irreversible scaffolds; capture-append is not in that class. Show diff *after* — print path + line.
3. **Don't auto-classify with LLM call on default path.** Notion's AI auto-categorize is famously distrusted. Use heuristic hints (e.g., `?` → suggest OPEN-QUESTIONS) with single-key override, not a model call.
4. **Don't force routing at capture for the unflagged case.** Linear "C" learned this; force-routing kills adoption.
5. **Don't number entries** (FUTURE-001, IDEA-042). GSD seeds do this; appropriate for seeds (referenced by ID with trigger conditions). FUTURE-IDEAS entries are prose — numbering invites bookkeeping.

## 6. Open sub-questions for plan to resolve

Three sub-questions surfaced during research that the plan must decide:

### Q1. New-milestone scaffold path — keep or defer to v0.2?

FUTURE-IDEAS routing table assigns ~2% to "new MILESTONE-N.md scaffold." It's the highest-blast-radius destination (creates new top-level planning file). Two options:
- **(a) Keep in v1**, gated behind two-stage confirmation (filename + skeleton preview → explicit "yes scaffold"). Refuse if N collides; refuse decimal numbers without explicit flag (anchors precedent that 4.5 was a one-off).
- **(b) Defer to v0.2**, route the 2% case to FUTURE-IDEAS.md with a note, let user hand-author milestone files (M4 + M5 + M4.5 all hand-authored — no breaking precedent).

**Recommendation: (b) — defer.** Hand-authoring milestone files is the existing precedent; capturing "new milestone idea" to FUTURE-IDEAS preserves the safe-asymmetry principle. Saves Epic scope.

### Q2. Cross-Epic missing back-half — does `/sig:plan` get a "review FUTURE-IDEAS.md" update in this Epic?

Domain agent flagged that without an explicit promotion loop, FUTURE-IDEAS.md becomes a write-only graveyard. Two options:
- **(a) In-scope:** add a sub-task to M4.5.E2 — update `commands/plan.md` to include a "review FUTURE-IDEAS.md and surface candidates for promotion" step.
- **(b) Out-of-scope:** log as a separate Epic (`M4.5.E2.1` or new entry).

**Recommendation: (a) in-scope, but as a small sub-task.** ~20 line edit to `commands/plan.md`. Without it, `/sig:add` ships into a system that doesn't read from where it writes. Atomic capture-AND-process is the GTD principle; ship both halves together.

### Q3. Heuristic routing hints — which patterns trip a single-key override?

Domain + prior-art both endorse heuristic suggestion with single-key override. Initial heuristic set proposed:
- Input ends with `?` → suggest OPEN-QUESTIONS (press `q` to confirm, Enter to default FUTURE-IDEAS)
- Input matches `/^(fix|bug|TODO|broken)\b/i` → suggest current MILESTONE (if STATE.md has one; press `m`)
- Otherwise → silent default to FUTURE-IDEAS

**Open:** should the heuristic set be hardcoded in v1 (recommended) or extensible via `.planning/.add-rules.json`? Plan recommendation: hardcoded; user-extensible later if real demand surfaces.

## 7. Environment / runtime assumptions (for environment-check step in plan)

- **Runtime:** Node 22+ (per `package.json` engines). `fs.rename` POSIX-atomic on Mac/Linux; Windows note — semantics differ but Signal targets Mac primary, Linux secondary, Windows-WSL only.
- **Native modules:** None. `/sig:add` is pure JS (`node:fs/promises`, `node:path`, `Date`).
- **External services:** None.
- **Existing deps:** `yaml` is the only non-node `tools/lib/` dep — `/sig:add` doesn't need it (no YAML frontmatter handling; markdown-only).
- **`npm install --dry-run` check:** required as part of plan's environment-check step.
- **Cross-FS rename caveat:** if `.planning/` is symlinked across filesystems, `fs.rename` returns `EXDEV`. Fall back to copy-then-unlink with documented caveat.
