## Important Issues

### I-1 ✅ FIXED IN-PHASE — `evict.js` B14 guard anchored one dir too shallow (leaf-symlink escape)
**Confirmed by security + code-quality reviewers, PoC-verified.** `evictEpicNarrative` passed the *directory* `archiveDir` to `assertRealInsidePlanning`, so the guard realpath'd `dirname(archiveDir)` = the archive **parent** and never inspected the Epic-leaf `archive/M{n}/E{n}` — the dir the write actually passes through. A checked-in directory symlink at that leaf (parents real) slipped the guard → `atomicWrite` wrote *through* it, landing `STATE-NARRATIVE.md` outside `.planning/`. Every other write gateway (`add.js`, `migrate-memory.js`, `archive-tree.js`) passes the **file** path (so `dirname` = the write's real parent) — evict was the lone deviant. The existing test planted the symlink at `.planning/archive` (which the shallow anchor *does* catch by walking up), never at the leaf.
- **Impact (bounded):** create/overwrite a file outside `.planning/` with *semi-controlled* content (attacker picks *where* via the symlink target, not *what* — the bytes are Signal's own narrative section). Not arbitrary-write, not RCE. Preconditions: a hostile repo the victim clones + runs `/sig:ship`|`/sig:checkpoint` on, with a valid `*-RETROSPECTIVE.md` for the target Epic. Raw escape ~P3; elevated to Important because a false-green test shipped it as "fixed."
- **Fix (`evict.js`):** compute `narrativeAbs` *before* the guard and pass the **file** path — `assertRealInsidePlanning(baseDir, narrativeAbs, …)` — so `dirname(narrativeAbs)` = the archive leaf gets realpath-resolved (matches `archive-tree.js`). **RED-first:** new `tests/evict-on-close.test.js` "LEAF … Site A, REVIEW" plants the symlink at `archive/M5/E1` (parents real) → RED (narrative escaped to `<outside>`) → GREEN after. 1528 → **1529**.

### I-2 ⏸ DEFERRED (justified) — FR5 AC5.2 tests prove throw-on-contention, not read-enclosure
**Test-integrity reviewer.** The 8 `rmw-lock.test.js` tests assert a second writer *under a held lock* rejects — but a wrapper that read *outside* the lock (the lost-update FR5 kills) would pass them identically. So AC5.2's literal property ("no lost-update") has **no behavioral test**; it rides on **inspection**.
- **Why deferred, not FAIL:** read-enclosure is **confirmed correct by inspection by 3 independent parties** (me, the advisor pre-REVIEW, and both the test-integrity + security reviewers swept all 6 wrappers) — each is `withStateLock(baseDir, () => XxxCore(…))` with the read inside `Core`; `promoteDrainEntryCore` calls the lock-free `applyDispositionToFileCore` (no re-entrancy). VERIFICATION.md already discloses this. So **AC5.2's behavior is met**; the gap is test-strength only.
- **Why not fix-in-phase:** a true behavioral proof needs an *interleaving* test (writer A pauses after its read, B commits, A resumes) which requires an **injectable pause hook in each Core** — a product-code change + new test infra, >50 LOC with design impact → per the PASS-WITH-FIXES rule that's a loop-to-EXECUTE, not an in-phase patch. **Follow-up:** add the pause-hook + interleaving test if FR5's guarantee ever becomes load-bearing (candidate v0.1.10).

### I-3 ⏸ DEFERRED (justified) — B6 local-stale suppression is broader than the "+1" bookkeeping scope
**Code-quality reviewer.** `isStateStale` now suppresses whenever *every* commit in `lastCommit..HEAD` touches only `STATE_AFFECTING_PATHS` (which includes `*-PLAN/PROGRESS/VERIFICATION/REVIEW.md`). So a PLAN/PROGRESS-only work commit with no co-committed non-state file reads as *not stale*, where the old code flagged it.
- **Why deferred:** the behavior is **deliberate** (the code comment states it) and **matches the ACs I wrote** (FR4's "real work" = touches a non-state file; the 3-case test encodes exactly this). Blast radius is **low + advisory** (fail-open; in the canonical flow `markFresh` advances past PROGRESS commits, so the range holds only the STATE-only +1). It's a *semantic* question ("does a PROGRESS-only commit mean STATE is stale?") better settled with usage data than rushed.
- **Candidate refinement (recorded):** the reviewer's tighter gate — suppress only when `commits.length === 1 && <HEAD is the STATE-only bookkeeping +1>` — is more precise to the true "+1" intent. Deferred as a v0.1.10 refinement; **surfaced to Brett** for a semantics call.
