## Security findings
- **ReDoS: none** — every regex against attacker/agent-controlled content is linear-time, anchored, no nested quantifiers / overlapping alternation. Empirically: 10 MB no-close frontmatter → 3.1 ms; 5 M newlines → 5.1 ms. FR3/FR4 regexes and loops are single-quantifier / window-capped.
- **RegExp injection: none** — `new RegExp(\`^${key}:\`)` interpolates only the string literals `'completed_phases'`/`'blockers'`; `proposedContent` never reaches `key`.
- **Fail-open posture: sound** — FR1 is a doc-integrity nudge, not a security boundary (anyone with Edit/Write already controls the repo). Fail-*closed* would wedge legit writes; the accepted false-negatives defer to FR2. `checkStateFrontmatterShape` can't throw (whole body in try/catch); the hook allows on every stdin/JSON/path/missing-file error.
- **FR2 `readStateSize`**: no path traversal (trusted baseDir, fixed suffix), `statSync` reads metadata only (no content amplification via symlink), no meaningful TOCTOU (read-only advisory).
