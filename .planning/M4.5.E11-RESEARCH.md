# M4.5.E11 — Epic-native flow — Research

**Phase:** PLAN (research pass). **Tier:** FULL / `research_parallelism: 4`. Synthesized from 4 parallel research agents (codebase integration, design tradeoffs, assumptions, risk+slices). All file refs verified against `main`.

## Headline: the write-half is *prose*, and there's a *regex schism*

Two findings reframe the whole Epic:

1. **The artifact "write-half" is model-substituted markdown prose, not code.** Phase commands emit literal filenames (`commands/plan.md:89` "Generate `.planning/{phase}-PLAN.md`", `execute.md:46`, `verify.md:64`, `review.md:83`); no `tools/lib/*.js` constructs artifact names — `resolveArtifactPath` (`resume.js:50`) is read-only. So FR2 is markdown edits across ~6 commands + a mode signal the model honors at write time. **Implication:** to satisfy strict Nyquist, the branching logic must be extracted into a testable pure helper (`artifactName()`), symmetric to `resolveArtifactPath`, that both commands and tests call. Prose is covered by prose-review + validator + `{phase}-VERIFICATION.md` attestation (the E10.S1.t4 house pattern).

2. **The regex schism (the constraint everything hangs on).** Three helpers read `current_epic` with *disagreeing* legality rules:
   - `EPIC_ID_RE` (`resume.js:28`) — **permissive** `/^[A-Za-z0-9._-]+$/` (accepts `v0.1.6`, `myfeature`, `M4.5.E11`). E10's read-half was built against this.
   - `deriveRetroPath` (`retrospective.js:133`) — **strict** `/^M\d+(\.\d+)*\.E\d+$/` and **throws** on non-match.
   - `CURRENT_EPIC_RE` (`milestones.js:20`) — **strict** `/^M(\d+(?:\.\d+)?)\.E\d+/`, returns null on non-match (and can't parse 3-deep `M4.5.6.E1`).

   The read-half resolves permissive shapes fine, but retro-path derivation + milestone lookup (which `/sig:ship` and the E9 retro-gate hook depend on) demand the strict `M{N}[.{N}]*.E{N}` shape. This is exactly the `current_epic: v0.1.6` friction from that lightweight patch. **E11's write-half must emit only strict-shape IDs (version-strings stay a hand-managed exception), OR widen the two strict regexes.** → PLAN gate decision (see §7).

## Definitive fact: no command writes `current_epic` today

Confirmed (not just cited). Init-to-`null` at `state.js:167` (`upgradeStateFile`) + `:213` (`initState`); every other occurrence is a read (`ship.md:34,117`, `resume.md:56-63`, `add.md:68`, `milestones.js:40`, `retrospective.js` ×several). No mutating helper assigns it — they `stripStateMeta`-spread, so a hand-typed value is *preserved* but never *set*. **Gotcha:** `setCurrentTask({id, epic, wave})` writes `epic` into the per-task sub-field `current_tasks[].epic` (`state.js:443`), NOT the top-level scalar. The scalar is genuinely orphaned — Signal-on-Signal hand-types it.

## Design decisions (resolved by the tradeoff agent; all pivot on `current_epic`)

- **(A) Epic-creation UX → `--epic <name>` flag on `/sig:discuss` + `/sig:new-project`.** Strictly cleanest on the install contract (no new file, no `REQUIRED_COMMANDS` change — the locked constraint holds by construction) and it fits the grain: an Epic *always begins at DISCUSS*. Rules out a pure auto-detect prompt (an offer injected on the null-`current_epic` linear path violates FR4 byte-identical). A dedicated `/sig:epic` command is the additive follow-on **only if** real use needs an explicit abort-without-opening-next (log to FUTURE-IDEAS, don't pre-build).
- **Close semantics = roll-on-open, NEVER clear to null** (reconciling a conflict: the risk agent proposed clear-at-SHIP; the design agent's roll-on-open wins because **clear-at-SHIP strands SPIKE Epics** — SPIKE skips SHIP (`calibrate.md:237`) so the clear never fires — *and* clearing to null wrongly reverts an Epic-native project to linear). `current_epic` holds the current/last Epic; opening the next **rolls** it (and must atomically reset the coupled `current_wave: null` + `current_tasks: []` — this also covers the *abandon* case).
  - **"Epic complete" done-signal = `{EpicID}-RETROSPECTIVE.md` existence, NOT `phase=SHIP`.** Signal's STATE never moves SHIP into `completed_phases` (v0.1.6 shipped with `phase: SHIP` and SHIP *absent* from `completed_phases` — this repo's live STATE proves it), so `phase=SHIP` means "in/past SHIP," not done. The retro file is the last SHIP step and is unambiguous.
  - **Unspecified transition roll-on-open creates (must be in S1):** a *writing* phase command run while `current_epic` points at a **done** Epic (its retro exists) with **no `--epic`** must **error and require `--epic`** — never silently re-run and clobber `{EpicID}-REQUIREMENTS.md`/`{EpicID}-PLAN.md`. Opening/rolling to a new Epic is *only* via explicit `--epic`.
- **(B) Per-Epic calibration → `.planning/{EpicID}-PROFILE.md`, whole-file shadowing.** The fork (partial-override rejected by strict `readProfile`) **dissolves**: Signal's PROFILEs are already written *complete* ("not inherited by reference," DECISIONS 2026-04-22), so an Epic PROFILE is a full self-contained profile that simply *shadows* the project one — no merge, no tier→defaults mapping, no STATE field, no `schema_version` bump. `readProfile`'s strictness becomes a *feature*. Add `readEffectiveProfile(baseDir, currentEpic)` probing the Epic file first (PROFILE is NOT in FR2's artifact list → read via `readProfile`, not `resolveArtifactPath`). **Composition rule:** effective profile = `{EpicID}-PROFILE.md` if it exists (with `current_epic` set + `EPIC_ID_RE`-valid), else `.planning/PROFILE.md`; no Epic PROFILE → inherits project unchanged (keeps per-Epic *optional* + linear byte-identical). status/resume must show *which* tier is active + why: `Tier: SKETCH (Epic M4.5.E11 override; project default FULL)`.
- **(C) Mode detection → non-null `current_epic` is the sole signal.** Atomic (one field can't self-disagree), already de-facto everywhere, zero new surface. Canonical predicate (use everywhere): `typeof current_epic === 'string' && current_epic.trim() !== ''`. Fail-open to linear on null/absent/`''`/malformed — never throw (matches the resolver guard + the "never throws on malformed STATE" NFR).

**How they lock:** `--epic` writes `current_epic` (A) + triggers calibrate-logic to write `{EpicID}-PROFILE.md` (B); every command branches on non-null `current_epic` (C) to resolve both the artifact name (FR2) and the effective profile (B). No new command, no contract change, no schema bump, linear byte-identical.

## Assumptions (severity-ranked, load-bearing only)

- **HIGH — FR2 has no code to RED-test:** resolved by extracting `artifactName()` (above); the pure helper carries the logic + gets the RED test, prose gets attestation.
- **HIGH — "add Epic #2 to a shipped project" straddles tight-core vs the parked layer:** resolved by the design — sequential Epics with flat `{EpicID}-*.md` (exactly what Signal-on-Signal does) ARE tight-core; the parked layer is only concurrent multi-feature with per-feature *subdirs*. `--epic` on the next DISCUSS is the entry point. Residual (a project's *first* linear→Epic adoption leaving old `1-*.md`) = documented default: leave as history (read-half serves the active Epic's `{EpicID}-*` correctly; R4 safe-by-construction).
- **MED — no schema bump is round-trip-safe but has a rigor-drift edge:** dissolved by design B (calibration lives in a sidecar PROFILE file, NOT a STATE field → no new STATE field to drift; old Signal without Epic support just uses project PROFILE). Documented limitation, not a blocker.
- **MED — coupled fields on Epic-roll:** roll must reset `current_wave`/`current_tasks` atomically (covered in A's close semantics).
- **MED — mode predicate:** one canonical truthiness+trim predicate everywhere (covered in C).
- **LOW-MED — fixture coverage hole:** all fixtures carry a `current_epic`; add the linear golden fixture (R9).

## Test strategy (the Nyquist spine)

Every FR gets a pure helper carrying the branching logic, unit-testable RED-first; markdown prose that calls it is covered by prose-review + validator + attestation (E10.S1.t4 pattern):
- `setCurrentEpic`/`clearCurrentEpic` (FR1) — strict-shape validation at write; clears to YAML `null` not `"null"`; idempotent; preserves other frontmatter; resets coupled `current_wave`/`current_tasks` on roll. Mirror `setCurrentTask` test shape.
- `artifactName(artifact, {currentEpic, phase})` (FR2) — pure, symmetric to `resolveArtifactPath`; round-trip test (write name → resolver finds it) for all 8 artifact kinds; `null` → `1-{ARTIFACT}.md`.
- `detectMode(state)` (FR4) — non-null strict → epic; null/absent/`''`/malformed → linear (fail-open).
- `readEffectiveProfile(baseDir, {currentEpic})` (FR3) — Epic-file-if-exists-else-project; no-`currentEpic` byte-identical golden (R7); malformed Epic PROFILE → `ProfileSchemaError` (fail-closed, existing posture).
- Hook throw-safety (R2) — malformed `current_epic` SHIP write → exit 0.
- Fixtures: `tests/fixtures/epic-native/{linear,epic}/.planning/` (byte-identity anchor + Epic golden). Existing 894-test suite staying green with no fixture edits *is* the linear byte-identity proof.

## Decisions for the PLAN gate

**Product/positioning — surface to Brett (§ gate):**
1. **Epic-ID shape (real fork, presented neutral):** (a) **M-shaped work units + version tags as separate labels** — `current_epic` is always `M{N}[.{N}]*.E{N}`; version numbers (`v0.1.6`) become separate release tags decoupled from `current_epic`. Cleaner, but a change from current habit (Brett has been putting `v0.1.6` *in* `current_epic`), so every lightweight patch going forward must be M-named to get the automation. **Shrinks S1** (no regex-widening). (b) **Widen the two strict regexes** so version-string patches stay first-class `current_epic` values, matching current habit. Grows S1. *Neutral — Brett's release-cadence call, not plumbing.*
2. **Retro-gate coupling (R3) — block / warn / off:** what happens when a stranger reaches Epic-close SHIP without a `{EpicID}-RETROSPECTIVE.md`. *Lean **warn*** — a hard exit-2 block cuts against the M4.5 stranger-adoption thesis AND contradicts the hook posture already shipped in v0.1.6 (D-v016-2: "block malformed frontmatter; *flag* everything else"). A missing retro isn't unambiguously wrong the way malformed YAML is, so warn (not block) is consistent with the codebase; `off` is the third option.

**PLAN self-decides (documented defaults, not gated):** design A/B/C above; `artifactName()` extraction; roll-on-open close semantics; canonical mode predicate; R10 (Epic-active → calibrate/escalate target the Epic PROFILE); the two golden fixtures.
