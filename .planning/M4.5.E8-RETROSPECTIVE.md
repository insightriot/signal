## What changed mid-flight

**`/releases/latest` → `/tags` endpoint switch.** REQUIREMENTS specified `https://api.github.com/repos/InsightRiot/signal/releases/latest`. Research empirically verified that endpoint returns HTTP 404 today — Signal pushes git tags but has never cut a GitHub Release object. The fix isn't to start cutting Releases (process change with no other consumer) — it's to switch the endpoint to `/repos/.../tags`. PLAN locked the substitution at D-E8-7 and the S3 implementation used the new endpoint everywhere. REQUIREMENTS document retained as written (historical capture of DISCUSS intent); the working endpoint lives in PLAN + RESEARCH + DECISIONS.

**FR9 upstream-filings scope reframed.** REQUIREMENTS planned 3 new issue filings (P1, P2, P3 against anthropics/claude-code). Research found 2 of 3 already filed by other people (#56740 P1 from `vermont42` on 2026-05-06; #62497 P2 from `xdm67x` on 2026-05-26). Filing duplicates would embarrass Signal in front of M4.5.E5's launch audience. PLAN promoted cross-linking work into S1 (`docs/install-troubleshooting.md` headers) and reduced new-filing scope to just P3 (Disabled-state-survives-reinstall — no upstream match found). Filed as anthropics/claude-code#63624 at S1.t14 ship.

**Helper-script split deferred (S2.t4/t5 PLAN deviation).** PLAN locked an 80-char threshold for inline `node -e` payloads — anything longer was to be promoted to a companion `~/.claude/sig-doctor-helper.js`. S2 implementation kept the inline approach despite ~200-char payloads. Tradeoff: two emitted files double the user's audit surface; one self-contained bash script reads as a flat sequence of independently-auditable actions. The 80-char threshold was an aesthetic readability proxy; the actual correctness gate ("does the embedded JS work?") is now enforced by `tests/doctor-script-gen.test.js` running every payload through `new Function()` to catch syntax errors at script-gen time, not at user-run time. Deviation documented in `.planning/FUTURE-IDEAS.md` with explicit revisit triggers.

**`docs/install-troubleshooting.md` grew, not shrunk.** PLAN said "doc length reduced where /sig:doctor collapses manual triage." Implementation went the other way (272 → 306 lines) because the reframe added an Ownership-at-a-glance table + per-section `/sig:doctor` lead-in notes. Manual workarounds retained as fallback prose for `/sig:doctor`-unavailable environments (older Claude Code, Linux, WSL until E1.S3 lands). Net value-add per line; trimming the manual sequences would have harmed users without `/sig:doctor` access.

**S3.t13 dogfood destructive leg deferred.** PLAN's AC #13 expected an end-to-end run including the actual `rm -rf` destructive steps, plus a follow-up `/sig:doctor` confirming healthy. Dogfood ran cleanly through detection + script-generation + script-execution but the user declined all three `[y/N]` prompts (legitimate — the orphan dirs are historical context worth keeping for the dev machine). Filesystem state pre/post-script-execution is identical. AC #13 documented in `docs/install-verification.md` § R6 as "execution-mechanism end-to-end verified; destructive leg covered structurally by unit-test content lint + per-step `[y/N]` safety; first real rm-rf execution waits for a stranger encountering a broken install."

## What assumptions broke

- **FR6 endpoint assumption.** Research assumed `/releases/latest` was the canonical version source. Empirical check (`curl` against the live repo) returned 404. Fix above. **Lesson:** for any external-API integration, the first PLAN-phase task should be a one-command empirical probe, not a doc-read.
- **`installed_plugins.json` shape.** REQUIREMENTS treated it as `plugins["sig@signal"]: {...}`. Reality is `plugins["sig@signal"]: [{...}]` (per-scope array). Detector signatures absorbed the array shape; tests + fixtures had to model it correctly. Caught early during S1.t6 fixture authoring; would have been a quiet production bug if missed.
- **Aggregate-recommendation severity rules at the P5 edge.** Initial cut had P5 (SSH multi-identity) being a finding that flipped `healthy: false`. RESEARCH § 7 Risk 1 reframed P5 as `info-only` — it's an environmental observation, not a broken-install symptom. The carve-out for `recommendation: 'info-only'` not changing `healthy` got added to `runAllDetectors`; tests cover the path.
- **Generated-script line-length aesthetic.** PLAN's 80-char threshold was a readability proxy; the actual user concern is "do the inline JS payloads work?" Reframed as well-formedness gate (parse via `new Function()`). Aesthetic proxies in PLANs are worth re-evaluating at GREEN time.
- **Dogfood destructive-leg execution.** PLAN expected an end-to-end including the actual rm-rf. Reality: when the dev machine is the dogfood substrate AND the orphan dirs are context worth keeping, the user-decline path IS the natural validation. The structural safety mechanisms (`[y/N]` default-skip, no `set -e` abort) carry the load.

## What surprised us

**Mac Studio's accumulated install history was a perfect free fixture.** The dev machine had three orphan version dirs (`0.1.0`, `0.1.1`, `signal/`) sitting on disk from the M4.5.E1.S1 upgrade and the M4.t19 pre-rename rename. RESEARCH flagged this as a "bonus" finding during codebase research; the dogfood at S3.t13 ran against real accumulated cruft, not synthetic state. The P2 (manifest-narrowed orphan detection — excludes the current `0.1.2`) and P4 (pre-rename slug literal hit) detectors both fired correctly on first try against this real data. Free integration test bigger than any fixture.

**`/reload-plugins` hot-loaded the in-flight code.** S3.t13's final step (`/reload-plugins`) picked up the new `commands/doctor.md` + the freshly-built `tools/lib/doctor.js` directly from the dev environment. `/sig:doctor` went live as a slash command mid-Epic — useful for the AC #13 follow-on, and a reminder that the dev-mode hot-reload path means dogfood can move faster than a marketplace publish cycle.

**The 3 conflicts surfaced at PLAN, not at EXECUTE.** Research-phase agents (4 of them) caught all three REQUIREMENTS-versus-reality mismatches before any code was written. PLAN reconciled rather than escalating back to DISCUSS — the right move because each conflict had an obvious technical resolution (`/tags` endpoint, cross-link instead of file, handle the array shape). **Lesson:** the PLAN-phase research investment paid for itself many times over relative to the cost of catching these at EXECUTE or VERIFY.

**Auto-mode classifier blocked the gh issue create AND the local node script.** Both events were appropriate friction — filing an issue on a third-party org and running an arbitrary script against `~/.claude/` are exactly the kinds of actions that should require user-initiated execution. The escape hatch (user typing `! <command>`) worked cleanly. The cost was minor turn-taking; the safety property was load-bearing.

**Test-count forecast was the most accurate dimension we measured.** PLAN forecast +38 tests; actual landed +73. PLAN's slice-level breakdowns (S1 +13, S2 +14, S3 +10) were each off by 2x. **Lesson:** forecast accuracy improves with explicit "per detector / per matrix row" counting, not "rough estimate per slice."
