**Epic:** M5.E3 (final doc-runtime Epic). **Tier:** FULL/strict (inherited; no Epic PROFILE). **Opened:** 2026-07-18 via `/sig:discuss --epic M5.E3`.

**Lineage.** This Epic delivers the original doc-runtime flagship's **FR4** (all-docs hygiene) + **FR5** (living BACKLOG) from `archive/M5/E1/M5.E1-REQUIREMENTS.md`, *plus* the **append-log hygiene** extension opened by **D-M5E2-6** (the doc-runtime ships as ONE `0.1.x` = E1+E2+E3; E3 extends auto-hygiene to the append-logs the E2 migrate protects-but-doesn't-clean). ## The doc-lifecycle model (locked this DISCUSS)

Four role-named files, each with one job:

| File | Role |
|---|---|
| **`ISSUES-INBOX.md`** (renamed from `FUTURE-IDEAS.md`) | The **raw capture inbox** — everything untyped lands here first, unsorted. Familiar word ("ISSUES") but the `-INBOX` suffix marks it as *not* the system of record. |
| **`BACKLOG.md`** (new) | The **groomed, sequenced roadmap** — the single "what's planned, in what order" source. Roadmap-vs-hygiene is a **tag inside** it, not a separate file. |
| **`BUGS.md`** (exists) | Confirmed **defects**. |
| **`OPEN-QUESTIONS.md`** (exists) | Unresolved **design questions**. |

Flow: `ISSUES-INBOX` → the drain **classifies** (work→`BACKLOG`, bug→`BUGS`) and **dispositions** (promote/defer/merge/delete), human-confirmed → promoted items **physically evict** from the inbox.

## Functional Requirements

### FR1 — `ISSUES-INBOX` rename + smart capture
Rename `FUTURE-IDEAS.md` → `ISSUES-INBOX.md` across code (`add.js`, `drain.js`, `migrate-memory.js` append-log set), tests, docs, and existing content (move-never-delete). `/sig:add` captures the **body verbatim** (the sacred contract — never rewritten) but writes an **agent-authored one-line title** (a clean summary written by the in-loop agent; the mechanical clause-slice `deriveHeading` stays as the deterministic fallback for non-interactive / hook captures). Typed fast-paths still skip straight to their homes: `--question`→`OPEN-QUESTIONS`, a **new `--bug`**→`BUGS`; untyped default → `ISSUES-INBOX`.
- **AC1.1** `/sig:add "text"` (untyped) writes to `.planning/ISSUES-INBOX.md`, not `FUTURE-IDEAS.md`.
- **AC1.2** The captured body is byte-identical to the input; only the derived title is agent-authored.
- **AC1.3** When no agent title is supplied (programmatic call), the entry still gets a deterministic clause-slice title (existing `deriveHeading`), never an empty/placeholder heading.
- **AC1.4** `--bug "text"` writes to `.planning/BUGS.md` (new fast-path); `--question` unchanged.
- **AC1.5** A repo with a pre-existing `FUTURE-IDEAS.md` has it renamed to `ISSUES-INBOX.md` with content preserved verbatim (handled by the FR6 migrate, not `/sig:add`).

### FR2 — Living `BACKLOG.md` + drain classify/promote
`BACKLOG.md` becomes the single sequenced roadmap; `BACKLOG-REVIEW-2026-07-04.md`'s content migrates in and the dated snapshot is archived behind a pointer (move-never-delete). The `/sig:plan` drain is extended to **classify** each inbox entry (work→`BACKLOG` / bug→`BUGS`) in addition to its existing **disposition** verbs (promote/defer/merge/delete); a promote may **retitle** the item as it moves; promoted items **evict** from `ISSUES-INBOX` (reusing M5.E1's FR3 eviction). The drain is anchored at `/sig:plan` (primary) plus a light inbox-sweep at `/sig:ship` (Epic-close); **not** `/sig:execute`.
- **AC2.1** `BACKLOG.md` exists as the sequenced roadmap; roadmap-vs-hygiene is a tag on each entry, not a separate file.
- **AC2.2** `BACKLOG-REVIEW-2026-07-04.md` content is migrated into `BACKLOG.md`; the snapshot is archived with a one-line pointer; nothing deleted.
- **AC2.3** The drain classifies at least one seeded feature entry → `BACKLOG` and one seeded bug entry → `BUGS`, human-confirmed.
- **AC2.4** A promoted entry no longer appears in `ISSUES-INBOX.md`; a second drain does not resurface it (convergence).
- **AC2.5** Promote supports a retitle; the item lands in `BACKLOG` under the (possibly cleaned) title.
- **AC2.6** The drain runs at `/sig:plan` and a light sweep at `/sig:ship`; `/sig:execute` does not invoke it.

### FR3 — Auto-generated `/sig:index`
`INDEX.md` becomes **fully auto-generated** by a Signal-native `/sig:index` generator (hand-curation retired); it is the **load-bearing traversal layer** for FR5. It **preserves hand-curated annotations** (the "Gotcha:" notes, tier legend, per-file one-liners) by file/Epic ID across regeneration — the same survive-by-ID pattern `RETROSPECTIVES.md` hooks already use. One-line notes are derived per file (mechanism in PLAN).
- **AC3.1** Running the generator on `.planning/` produces an `INDEX.md` covering every tracked doc + archive.
- **AC3.2** A curated annotation (e.g. INDEX's ID-is-identity "Gotcha") survives a regeneration that changes unrelated rows (idempotent by ID).
- **AC3.3** The index resolves every live **and archived** decision ID → its current home (required by FR5).
- **AC3.4** Regeneration is idempotent — a no-op run produces no diff.

### FR4 — All-docs hygiene runtime (in the test suite)
Extend hygiene beyond `.planning/` to the whole doc surface (`docs/`, `README`, `CLAUDE.md`, `analysis/`). **Structural drift = hard test failure**; **soft findings = reported, non-blocking**. Checks are **deterministic + offline** — internal link-health + roster/count/version drift only; **no external URL checks** (the B17 CI-flakiness lesson). Absorbs the queued **map-drift guard**.
- **AC4.1** Adding an agent/command/skill without updating the roster/count turns the suite **red** (structural drift = hard fail).
- **AC4.2** A dead **internal** link, or a `[FILL IN]` stub in a shipped (non-template) doc, fails the suite; a prose-staleness hint is reported, not failed.
- **AC4.3** No hygiene check makes a network request (grep the check code for fetch/curl/http → none); external links are never validated.
- **AC4.4** The guard only reads/asserts — it never auto-writes prose.
- **AC4.5** The map-drift guard candidate is folded in (the `docs/map` roster/count checks live here).

### FR5 — Append-log eviction (evict-with-anchors) — *the risky, migrate-shaped piece*
Closed-milestone `DECISIONS.md` blocks (and closed milestone-doc blocks) **relocate** to a history file (`DECISIONS-HISTORY.md` / per-milestone) behind a dated pointer, with **every `D-…` anchor preserved** so the ~669 bare-ID prose references stay resolvable via the FR3 index; the live `DECISIONS.md` keeps only the current milestone's decisions. Reuses E2's **relocate-never-delete spine + dangling-link gate**; a **faithfulness gate** (distill→verify-against-source→evict, per the E1/E2 §5 pattern) guards the move — the risk is silent *degradation*, not loss (nothing deleted).
- **AC5.1** A closed-milestone decision block moves to the history file behind a dated pointer; the live `DECISIONS.md` shrinks; nothing is deleted (original survives in history).
- **AC5.2** Every `D-…` anchor for an evicted decision still resolves via the index (a seeded `D-…` reference is findable post-eviction).
- **AC5.3** The dangling-link gate (E2) runs over the result — zero migrate-introduced dangles; the current milestone's decisions stay in the live file.
- **AC5.4** The append operation (`/sig:checkpoint --context`) still targets the **live** `DECISIONS.md` — new decisions append there, eviction only moves *closed-milestone* blocks.
- **AC5.5** Idempotent — a re-run on an already-evicted `DECISIONS.md` is a no-op.

### FR6 — Layout v2→v3 migration + rollout (covers new *and* existing projects)
Bump `CURRENT_LAYOUT_VERSION` **2 → 3**. Extend `/sig:migrate-memory` to handle the **v2→v3** transition — rename `FUTURE-IDEAS.md`→`ISSUES-INBOX.md` (with reference updates), create `BACKLOG.md` from the migrated snapshot, run the FR5 append-log eviction — all under E2's dry-run/rollback/relocate-never-delete/idempotent contract. New projects are **born on v3** (new-project/init create the new-named files on-demand). Existing projects get the E2 auto-flow: `/plugin update` → SessionStart/resume/status banner sees layout `2 < 3` → prompts `/sig:migrate-memory` → dry-run → **user approves** → apply. **Retire the `ship.md` §8 external-Curator step** (native hygiene replaces it, per D-M5E1-3).
- **AC6.1** `CURRENT_LAYOUT_VERSION === 3`; a v2-stamped project banners on resume/status/SessionStart; a v3 project is silent.
- **AC6.2** `/sig:migrate-memory` on a v2 project dry-runs the full v2→v3 plan (rename + BACKLOG create + append-log evict) and applies only on confirm; re-run is a no-op.
- **AC6.3** A fresh `/sig:new-project` (or `/sig:init`) project uses the new file names — `/sig:add` creates `ISSUES-INBOX.md`, never `FUTURE-IDEAS.md`.
- **AC6.4** The `ship.md` §8 Curator step is removed; SHIP reconciles docs via the native runtime (FR3/FR4) instead; no Signal command invokes `curator`.
- **AC6.5** The migrate is faithful on Signal's own `.planning/` dogfood (rename + BACKLOG + append-log evict, 0 content dropped, dangling-gate green).

## Non-Functional Requirements
- **Deterministic + offline** — hygiene checks never touch the network (FR4); no flaky CI (the B17 lesson made concrete).
- **Cross-platform** — checks run in the test suite / CI on macOS + Linux (no macOS-only calls in the hygiene path).
- **Fail-open** — the layout-drift banner + hygiene-at-orientation never block a session (E2 posture inherited).
- **Safety (inherited from E2)** — the migrate is dry-run-first, git-reversible, relocate-never-delete, idempotent.
- **Faithfulness (the risk safety-NFRs don't cover)** — append-log eviction (FR5) and any distilled pointer must faithfully represent the source; verified by a human-approved dry-run diff + the Signal dogfood eyeball, not green tests alone (E1/E2 §5 pattern).

## Out of scope (E3)
- Full knowledge-graph / `graphify` traversal beyond link-health + the reference-index (deferred with M5.E1).
- v2 feature ports (IDEATE/VALIDATE/STRATEGIZE, compound replay, gstack/superpowers) — later M5, behind the re-audit.
- Smarter capture heuristics beyond agent-authored title + clause-slice fallback.

## Assumptions to validate in PLAN/EXECUTE
- The auto-`/sig:index` (FR3) is thorough enough to keep archived decisions findable — **load-bearing** for FR5; if it isn't, evict-with-anchors regresses to detect-only.
- E2's migrate engine (dangling-gate + relocate-never-delete spine) is reusable for FR5's append-log eviction (it already tags `DECISIONS.md` as a protected append-log — E3 adds the eviction path).
- An in-loop agent is present for the auto-title in interactive `/sig:add`; the deterministic fallback covers hook/script captures.
