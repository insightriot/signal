## Phase Goal

Ship Signal v0.1.1 with a marketplace install that survives any reasonable stranger's machine — including multi-identity SSH configs, corporate proxies blocking SSH port 22, and `IdentitiesOnly yes` hardened setups — verified on at least 4 fresh-machine matrix rows, with the F2 agent-registration unknown empirically resolved and a written versioning policy governing future releases.

> **Out of scope (deferred to other Epics):**
> - Full README rewrite → E3
> - CONTRIBUTING.md / SECURITY.md → E3
> - `examples/` worked example → E4
> - External tester recruitment → E5
> - `.planning/` schema migration story (D4 / F4 risk) → FUTURE-IDEAS

---

## Assumed Runtime (locked at PLAN time; verified in Slice 1)

- **Node 22+** (per `package.json` engines, advisory only — Claude Code does not enforce).
- Pure JS, no native modules, no install hooks. Single runtime dep: `yaml@^2.8.3`.
- Test runner: `vitest` (devDep). Validator: `tools/validate-plugin.js` (pure-Node CLI).
- POSIX filesystem assumed; Windows-native path handling deferred to E5.
- Claude Code latest stable. Older versions (< 2.1.141, which shipped `CLAUDE_CODE_PLUGIN_PREFER_HTTPS`) are not guaranteed-supported but should not be broken — they'll just fall back to status quo.

## Vertical Slices

Five slices. **S1 is critical-path and ships first** (stops the bleeding for stranger installs). S2/S3 depend on S1's v0.1.1 being installable; they unblock once S1 is tagged. S4 and S5 are independent and can ship in parallel with S2/S3. Each slice ends at green tests + validator pass + manual smoke + commit. Each is independently shippable as a v0.1.x patch.

```
S1 (critical) ─────┬──> S2 (F2 verify + maybe restructure)
                   ├──> S3 (verification matrix)
                   └─── ships v0.1.1 ─────────────────> S4 (versioning policy)
                                                       └── S5 (validator hardening) [scope-beyond-exit]
```

### Slice 1 — Source-block fix + v0.1.1 ship — `[S]`

**User value:** Strangers with multi-identity SSH configs, `IdentitiesOnly yes` hardening, or corporate-proxy-blocked SSH ports can install Signal via marketplace without intervention. Closes today's break (failure mode A1).

**Scope:**

- `.claude-plugin/marketplace.json` — change `plugins[0].source`:
  - **From:** `{"source": "github", "repo": "InsightRiot/signal"}`
  - **To:** `{"source": "url", "url": "https://github.com/InsightRiot/signal.git", "ref": "v0.1.1", "sha": "<40-char-commit-sha-of-the-S1-commit>"}`
- `.claude-plugin/plugin.json` — bump `version`: `"0.1.0"` → `"0.1.1"`.
- `README.md` — patch the Install section only (minimum-viable, NOT a full rewrite — that's E3):
  - Remove stale `/plugin install signal` line.
  - Confirm 3-line install block: `/plugin marketplace add insightriot/signal` → `/plugin install sig@signal` → `/reload-plugins`.
  - Add a short "Troubleshooting" subsection naming `CLAUDE_CODE_PLUGIN_PREFER_HTTPS=1` as the documented escape hatch (citing Claude Code 2.1.141), plus the manual-clone + local-marketplace workaround that Brett used today.
- `CHANGELOG.md` (new file) — initial seed with v0.1.0 entry (M4 close + signal→sig rename + marketplace install layout) and v0.1.1 entry (source block fix + version bump).
- `tools/validate-plugin.js` — minimal addition: enforce `plugin.json.version` is semver-shaped (`/^\d+\.\d+\.\d+$/`) so future drift is caught. (Full validator hardening lives in S5; this is just the version regex.)
- Tag v0.1.1 (the sha in marketplace.json must match the commit that contains the marketplace.json update — chicken-and-egg handled by amending after first commit, or pre-computing via `git commit-tree`; documented in DECISIONS.md).

**Acceptance criteria:**

1. **Install succeeds on Brett's business machine** (R1 — multi-identity SSH, no default `Host github.com`) via the documented 3-line install block, with zero git/SSH workaround commands typed. *Test method: manual verification on the actual machine that broke today; capture exit codes + stdout in `docs/install-verification.md`.*
2. `npm run validate` passes with the new semver-format check on `plugin.json.version`. *Test method: `tests/install-contract.test.js` adds 2 cases (`0.1.1` passes; `0.1.x` fails) — see `M4.5.E1-VALIDATION.md`.*
3. `marketplace.json.plugins[0].source.source === "url"` and `.source.url` matches `https://github.com/InsightRiot/signal.git`. *Test method: `tests/install-contract.test.js` adds a fixture-based content assertion (S1's new test file).*
4. `marketplace.json.plugins[0].source.sha` exists and is a 40-char hex string. *Test method: same as #3.*
5. `marketplace.json.plugins[0].source.ref` matches `plugin.json.version` (prefixed with `v`). *Test method: same as #3.*
6. README's Install section contains `CLAUDE_CODE_PLUGIN_PREFER_HTTPS=1` reference. *Test method: grep-based smoke in `tests/readme-content.test.js` (new).*
7. `CHANGELOG.md` exists at repo root with `## [0.1.1]` and `## [0.1.0]` sections. *Test method: existence + heading regex in `tests/install-contract.test.js`.*
8. `npm test` green (existing 209 + new install-contract tests). *Test method: CI / local `npm test`.*
9. Git tag `v0.1.1` exists locally and is pushed (verified by user before merging gate). *Test method: manual; documented in CHANGELOG.*

**Test strategy (TDD, red-green-refactor):**

New file: `tests/install-contract.test.js` (~8 tests covering criteria 2–7).
- Unit/fixture: parse `.claude-plugin/marketplace.json` → assert source-block shape.
- Unit/fixture: parse `.claude-plugin/plugin.json` → assert semver-format on `version`.
- Fixture: malformed marketplace.json (e.g., `source: "github"` no ref) → expect validator failure.
- Integration: invoke `tools/validate-plugin.js` against current repo, expect success.

New file: `tests/readme-content.test.js` (~3 tests covering criterion 6).
- Smoke: README contains expected install block.
- Smoke: README contains `CLAUDE_CODE_PLUGIN_PREFER_HTTPS` mention.
- Smoke: README does NOT contain stale `/plugin install signal` (the bare slug).

**Dependencies:** None. Slice 1 is the entry point.

**Complexity:** **S** — file changes are small; logic is straightforward; complexity is in the SHA chicken-and-egg (handled with `git commit-tree` + a one-paragraph rationale doc).

**Ship gate (Slice 1):** All 9 acceptance criteria met → user explicitly approves → tag v0.1.1 → push tag + branch. Slice 1 ships before S2/S3/S4/S5 start work.

### Slice 2 — F2 verification + agent restructure (conditional) — `[M]`

**User value:** Resolves the F2 unknown that's been deferred since v0.1.0. Either confirms Signal's nested-agents layout works post-install (and stops dev-mode-fallback-as-primary), or restructures to flat so agents are first-class discoverable via `/agents`.

**Scope:** Two-phase. **Phase A is verification only (no code changes); Phase B is conditional on Phase A's result.**

**Phase A — Verification (must run before Phase B):**
- Install Signal v0.1.1 (from S1) on a fresh machine via marketplace.
- Run `/reload-plugins`.
- Run `/agents` — capture exact list of Signal-registered agents.
- Run `/sig:init` on a small public test repo (e.g., a fresh clone of `expressjs/express` or similar).
- Inspect `.planning/scan/{stack,structure,activity,quality}.md` for parity with dev-mode dogfood output (`/Users/macstudio/dev-biz/signal/.dogfood/M4-INIT-DOGFOOD/.planning/scan/` is the gold reference).
- Document outcome in `docs/install-verification.md` (created in Slice 3, but Slice 2 contributes the F2-section content).

**Outcome decision tree (Phase B):**

| Phase A outcome | Phase B action |
|---|---|
| **(a)** All 26 agents visible in `/agents`, namespaced as `sig:<name>`; named-subagent spawn works in `/sig:init` | Update `commands/init.md:170` (stale `signal-` → `sig:`). Update DECISIONS.md F2 entry. No restructure. Mark fallback path as "dev-mode only." |
| **(b)** 0 (or partial) visible; fallback fires; scan output parity OK | Promote fallback to primary in init.md prose. Add `tools/lib/agents.js` helper that loads agent definitions from disk for the fallback's inline-spawn pattern (single source of truth instead of duplicating definitions inline). Update DECISIONS.md. Agent files stay nested as source-of-truth. |
| **(c)** 0 visible; fallback brittle (tools not granted, token budget exceeded, scan output degraded) | Restructure all 26 agents from `agents/<category>/<name>.md` → `agents/<category>-<name>.md` namespaced-flat (preserves category in filename). Rewrite init.md Step 2 to use direct named-subagent spawn. Update validator `REQUIRED_AGENTS` list to all 26. Update tests. |

**Acceptance criteria:**

1. **Phase A run is documented** in `docs/install-verification.md` with: exact `/agents` output, exact `/sig:init` log including any "Agent type X not found" lines, side-by-side comparison of new `.planning/scan/` files vs dogfood-reference, dated + machine-signature. *Test method: manual; the doc file is the artifact.*
2. **Outcome label (a/b/c) is recorded** in `DECISIONS.md` with a new 2026-05-15-ish entry titled "F2 resolved: outcome (X)". *Test method: grep `DECISIONS.md` for "F2 resolved".*
3. **`commands/init.md:170`'s stale `signal-` prefix is fixed** to match actual outcome (no prefix, `sig:` prefix, or per-outcome-specific). *Test method: grep `commands/init.md` for stale `signal-` ref; should return 0.*
4. **If outcome (b) or (c) chosen:** `tools/validate-plugin.js` REQUIRED_AGENTS expanded from 4 to 26 entries. *Test method: validator runs, all 26 enforced.*
5. **If outcome (c) chosen:** all 26 agent files moved to flat namespaced layout; no file in `agents/{subdir}/` except per the new convention. *Test method: `tests/install-contract.test.js` adds a path-shape assertion for `agents/`.*

**Test strategy:**
- Phase A is empirical verification — output goes to `docs/install-verification.md`, not code.
- If outcome (c) restructure: rename tests cover both old (gone) and new (present) paths; existing `tests/init-fixtures.test.js` already uses fixture-data and won't break.
- New unit test in `tests/install-contract.test.js`: assert every file matched by `glob('agents/**/*.md')` ALSO matches the chosen layout pattern (caught at validate-time going forward).

**Dependencies:** Slice 1 must ship (v0.1.1 installable from marketplace).

**Complexity:** **M** — Phase A is fast but blocks Phase B. Phase B has three outcomes ranging from trivial (a) to moderate (c — 26-file rename + init.md rewrite). Worst-case is mechanical.

### Slice 4 — Versioning + deprecation policy — `[S]`

**User value:** Strangers can trust how Signal handles breaking changes. Author gains a release-checklist that prevents M4.t19-style "silent rename" landmines.

**Scope:**
- Create `docs/versioning.md` codifying option (a) from research (strict semver from v0.1.0; pre-1.0 `0.x.y` allows breaking changes at `x` bumps; CHANGELOG entries flagged `[BREAKING]` where applicable; no formal deprecation window until ≥3 external users — revisit at E5).
- Embed a "Release checklist" section in `docs/versioning.md`: (1) bump `plugin.json.version`, (2) update `marketplace.json` `ref` + `sha`, (3) update `CHANGELOG.md` with entry + `[BREAKING]` tags, (4) git tag `v<version>`, (5) push tag + branch, (6) optionally notify users to `/plugin marketplace update signal` for time-critical fixes.
- Append a `DECISIONS.md` entry locking the policy.

**Acceptance criteria:**

1. `docs/versioning.md` exists and describes: (a) pre-1.0 vs post-1.0 strictness, (b) CHANGELOG format, (c) release checklist, (d) the "revisit at E5 if user count ≥3" trigger. *Test method: grep for each load-bearing section heading.*
2. `DECISIONS.md` has a new entry titled "Versioning + deprecation policy locked" with the rationale + 2026-05-15-ish date. *Test method: grep.*
3. The release checklist is followed for S1's v0.1.1 tag (retroactive smoke test of the policy). *Test method: cross-check that v0.1.1 ship steps match the checklist as written.*

**Test strategy:** Documentation artifact. Tests are existence-and-content grep checks in `tests/docs-presence.test.js` (new or merged into install-contract).

**Dependencies:** Slice 1 (CHANGELOG.md exists). Otherwise independent.

**Complexity:** **S** — pure documentation; logic is the policy decision, captured.

### Slice 5 — Validator hardening (scope-beyond-exit-criteria) — `[S]`

**User value:** Future install breaks of today's class (marketplace.json drift, plugin.json drift, agent layout drift, file-path semantics drift) get caught at `npm run validate` time instead of by strangers' broken installs.

**Note:** This slice is **beyond** the E1 exit criteria as written in `MILESTONE-4.5.md`. Including in E1 because the class of bug it prevents is exactly what motivated E1's existence. If gate-time scope pressure exists, this slice is the safe punt → FUTURE-IDEAS.

**Scope:** Add to `tools/validate-plugin.js`:
- **marketplace.json content validation:** `plugins[0].name === "sig"`, `plugins[0].source.source === "url"` (locks the new convention), `plugins[0].source.url` matches `https://github.com/...git$`, `plugins[0].source.sha` is 40-char hex.
- **plugin.json ↔ marketplace.json consistency:** `plugin.json.name === marketplace.json.plugins[0].name`.
- **plugin.json semver format:** already added in S1; extend with a parity check vs git tag if available.
- **Per-command frontmatter sanity:** every `commands/<name>.md` has `name: sig:<name>` in YAML frontmatter; filename matches.
- **REQUIRED_AGENTS expanded** (from S2 outcome) to cover all 26.
- **Agent count assertion:** total file count in `agents/**/*.md` equals 26 (catches accidental deletion / orphan additions).

**Acceptance criteria:**

1. New validator checks all pass against current repo state. *Test method: `npm run validate` exits 0.*
2. Each new check has ≥1 negative-case test in `tests/install-contract.test.js` (fixture with malformed content → expect validator failure). *Test method: vitest run.*
3. The 4 "extra" agents on disk vs the 22-agent count in `DECISIONS.md:90-100` are reconciled (either count updated to 26 or extras removed). *Test method: validator count assertion + manual decision logged in DECISIONS.md.*

**Test strategy:** Pure-unit. Each validator check pairs with one positive fixture (passes) + one negative fixture (fails with expected error message). ~10-12 new tests.

**Dependencies:** Slice 1 (validator semver check) + Slice 2 (REQUIRED_AGENTS final list).

**Complexity:** **S** — adds ~80-100 LOC to `validate-plugin.js`; fixture-test pattern already established in `tests/install-contract.test.js`.

## Cross-slice Notes

- **Single source-of-truth for the SHA-pin chicken-and-egg.** The S1 marketplace.json contains a SHA pointing to its own commit. Resolution: commit S1 changes EXCEPT marketplace.json's `sha`. Get the commit SHA. Amend marketplace.json with the SHA. Force-push the amended commit (only safe on a feature branch). Document the dance in `docs/versioning.md` § Release checklist. Alternative: `git commit-tree` to pre-compute, but the amend dance is simpler and well-understood.
- **Slice 1 ship triggers a small Slice 4 dependency:** the release checklist embedded in `docs/versioning.md` should describe the very sequence S1 just executed. Slice 4 can ship after S1 with the checklist as written-after-the-fact.
- **Slice 5's "agent count drift" check exposes the 22-vs-26 question.** `DECISIONS.md` line 90-100 locked 22; current disk count is 26. The 4 extras (`ui-researcher`, `ui-auditor`, `ui-checker`, `phase-gate-enforcer`) are present, in use (UI agents bound to skills), and not malicious. Decision: reconcile by updating DECISIONS.md count to 26 with a note. This is a Slice 5 sub-task.
