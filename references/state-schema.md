# STATE.md — Schema Reference

`.planning/STATE.md` is Signal's machine-readable record of where the project is, what's in flight, what's blocked, and when work last touched ground. As of `schema_version: 1` (M4.5.E6), STATE.md uses YAML frontmatter for structured state with a freeform body for human-readable narrative.

The frontmatter is authoritative. The body is preserved across writes but is not parsed by Signal's commands — write whatever notes are useful to you below the closing `---`.

---

## Format

A schema_version-1 STATE.md starts with a YAML frontmatter block (`---` fences), followed by an optional freeform body.

### Example — fresh project, just calibrated

```yaml
---
schema_version: 1
phase: CALIBRATE
current_epic: null
current_wave: null
current_tasks: []
completed_phases: []
blockers: []
last_decision_at: null
last_updated_commit: null
last_updated: 2026-05-17T18:00:00.000Z
---

# Project State

Initialized by /sig:new-project. Run /sig:calibrate next.
```

### Example — mid-EXECUTE, one task in flight, one just completed

```yaml
---
schema_version: 1
phase: EXECUTE
current_epic: M4.5.E6
current_wave: 2
current_tasks:
  - id: M4.5.E6.S4.t3
    epic: M4.5.E6
    wave: 2
    status: in_progress
    startedAt: 2026-05-17T22:00:00.000Z
completed_phases:
  - CALIBRATE (2026-05-14)
  - DISCUSS (2026-05-16)
  - PLAN (2026-05-17)
blockers: []
last_decision_at: 2026-05-17T20:00:00.000Z
last_updated_commit: abc123def456ship
last_updated: 2026-05-17T22:00:00.000Z
last_completed_task:
  id: M4.5.E6.S4.t2
  status: done
  commit: prior-sha-789
  completedAt: 2026-05-17T21:45:00.000Z
---

# Project State

Optional freeform narrative below. Signal never parses this.
```

## Frontmatter Schema

### Top-level fields

| Field | Type | When populated | Notes |
|---|---|---|---|
| `schema_version` | integer (1) | Always | Schema marker. Reading code refuses unknown versions (fail closed). |
| `phase` | string enum | Always | One of `CALIBRATE` / `DISCUSS` / `PLAN` / `EXECUTE` / `VERIFY` / `REVIEW` / `SHIP`. |
| `current_epic` | string \| null | When mid-Epic | E.g. `M4.5.E6`. Auto-migration sets to `null` since legacy STATE.md has no machine-readable epic. |
| `current_wave` | integer \| null | When mid-wave | Optional even within an epic; meaningful for wave-parallel execution. |
| `current_tasks` | array of `Task` | Mid-EXECUTE | See `Task` below. D10: array, not scalar — supports wave parallelism (Slice 1 of M4.5.E6 ships with sequential-only call sites but the schema accepts multi-entry). |
| `completed_phases` | array of strings | After any `transitionPhase` | Entries shape `"PHASE_NAME (YYYY-MM-DD)"`. Dedupe by phase name on re-transition. |
| `blockers` | array of `Blocker` | When blocked | See `Blocker` below. |
| `last_decision_at` | ISO 8601 timestamp \| null | After `appendDecision` / `clearCurrentTask` | When the last decision-shaped event happened. |
| `last_updated_commit` | string (git sha) \| null | After `markFresh` / `clearCurrentTask({commit})` / `upgradeStateFile` | The commit hash relative to which staleness is measured. |
| `last_updated` | ISO 8601 timestamp | Always (any mutation) | When STATE.md was last written. |
| `last_completed_task` | `Task` shape \| absent | After first `clearCurrentTask` | Single-entry (not array) — only the most recent. Read by `renderResumeBriefing` for the "Last completed" line. |

### `Task` sub-shape

```yaml
- id: M4.5.E6.S4.t3       # required; the canonical Signal ID
  epic: M4.5.E6           # nullable
  wave: 2                 # nullable
  status: in_progress     # 'in_progress' | 'done' | 'aborted' | 'blocked'
  startedAt: 2026-05-17T22:00:00.000Z
```

`last_completed_task` adds:

```yaml
last_completed_task:
  id: M4.5.E6.S4.t2
  status: done            # 'done' | 'aborted' | 'blocked'
  commit: prior-sha-789   # nullable
  completedAt: 2026-05-17T21:45:00.000Z
```

### `Blocker` sub-shape

```yaml
- id: blk-a1b2            # 4-hex-char auto-generated by addBlocker
  text: Marketplace install hangs on first run; tracked under F2
  raisedAt: 2026-05-16T10:00:00.000Z
```

## Migration policy (D4 + D14)

`tools/lib/state.js#readState` performs strict three-way detection:

1. **No frontmatter at all** (legacy STATE.md) → returns the legacy-parsed shape with a `_schema: 'legacy'` sentinel. Downstream mutating helpers see the sentinel and call `upgradeStateFile` on the next write.
2. **Frontmatter + `schema_version: 1`** → parses normally. Returns native snake_case fields (`completed_phases`, `last_updated`) AND back-compat camelCase aliases (`completedPhases`, `lastUpdated`) so existing readers keep working.
3. **Frontmatter + unknown `schema_version`** (e.g., `999`) → throws `StateSchemaError`. Fail closed; don't silently mis-parse.
4. **Frontmatter present but no `schema_version` key** → throws `StateSchemaError`. Refuses to auto-upgrade. The user must remove the frontmatter (to let migration run) or hand-edit `schema_version: 1` in.

### `upgradeStateFile` behavior

Triggered automatically on the first mutating write to a legacy STATE.md. Preserves original content verbatim under an HTML comment marker:

```
---
schema_version: 1
phase: <inferred from "## Current Phase\n…" or 'EXECUTE'>
…
---
<!-- Original STATE.md content preserved verbatim from pre-schema_v1 migration on YYYY-MM-DD. The YAML frontmatter above is the authoritative machine-readable state; everything below is human-readable history. -->

# (original legacy STATE.md content)
…
```

Field inference:

- `phase` — regex `^## Current Phase\s*\n+([^\n]+)`; defaults to `EXECUTE` (most pre-S1 STATE.md files were mid-EXECUTE).
- `completed_phases` — slice the `## Completed Phases` section, strip `-` bullets, treat `(none)` as `[]`.
- `last_updated_commit` — `git rev-parse HEAD` on `baseDir`; `null` on git failure (D6 graceful degradation).
- `last_updated` — ISO timestamp at migration time.
- Other fields → defaults (`null`, `[]`).

Migration is idempotent. Re-running on an already-upgraded file returns `{upgraded: false, reason: 'already-frontmatter'}` and does not re-write.

## Staleness check (D11 + D6)

`tools/lib/state.js#isStateStale(baseDir, {bypassGrace})` returns:

```ts
{
  stale: boolean,
  commitCount: number,
  commits: Array<{sha: string, subject: string}>,
}
```

A commit counts as state-affecting if it touches any of these paths (D6 path scope):

```
.planning/STATE.md
.planning/CONTEXT.md
.planning/*-PROGRESS.md
.planning/*-PLAN.md
.planning/*-VERIFICATION.md
.planning/*-REVIEW.md
```

Editing FUTURE-IDEAS.md / OPEN-QUESTIONS.md / DECISIONS.md / MILESTONE-*.md / PROJECT.md / LANDSCAPE.md does **not** trigger staleness — those are metadata curation, not "ground state has moved."

`isStateStale` skips the git call entirely when `last_updated` is within 60s of now (resume-after-write isn't stale). `/sig:checkpoint` passes `bypassGrace: true` because the user is explicitly asking "what changed?"

## Orphan detection (D12)

`tools/lib/state.js#detectOrphans(baseDir, {thresholdMs, execFn})` returns in-progress tasks that:

- Have aged past `thresholdMs` (default 30 minutes) and
- Have no matching commit subject since `startedAt` referencing the task's ID.

Used by `/sig:resume`'s `handleOrphansAtResume` and `/sig:checkpoint`'s `handleCheckpointOrphans`. The prompt fires regardless of `gate_strictness` — the prompt IS the recovery mechanism; suppressing it re-introduces the failure mode that motivated this Epic.

---

## Tier-aware auto-protocol (D9 + S5)

`/sig:execute` wraps each task with `dispatchTaskWithState` (in `tools/lib/execute.js`):

| Tier | `gate_strictness` | Behavior on `setCurrentTask` failure |
|---|---|---|
| SKETCH | off | **Auto-protocol entirely disabled.** STATE.md updates only via manual `/sig:checkpoint`. |
| FEATURE / SPIKE | light | Log stderr warning; dispatch anyway. State-tracking for that task is skipped; clearCurrentTask doesn't run. |
| FULL | strict | Halt: throw before agent runs. State integrity wins over progress under strict. |

`markFresh` calls in `commands/verify.md` + `commands/review.md` follow the same SKETCH-opt-out rule but never halt phase exit on failure — the verification / review work is already done; a state-write blip is a recovery item, not a phase failure.

## Validation rules

- `schema_version` is required when frontmatter is present; readers refuse to auto-upgrade frontmatter that lacks it.
- `phase` must be one of the seven canonical phases.
- `current_tasks[].id` is required for every entry; idempotent on double-call.
- `blockers[].id` follows the `blk-XXXX` shape (4-hex auto-generated).
- All ISO 8601 timestamps are validated by the writer (`new Date().toISOString()` shape); readers tolerate any string but caller code that does date math should guard against `null` / parse failure.
- Lock semantics: `.planning/.state.lock` has a 5s TTL (state writes are millisecond-shape; longer TTLs would falsely block concurrent reads). `/sig:add` keeps its 30s TTL on `.planning/.add.lock` because its sensitive-data prompt can sit open.

## How commands consume STATE.md

| Command | Reads | Writes |
|---|---|---|
| `/sig:status` | `state.phase`, `state.completed_phases`, `state.blockers`, `state.current_tasks` | (read-only) |
| `/sig:resume` | All fields + `last_completed_task` for "Last completed" line | (read-only) |
| `/sig:execute` | (orchestrator delegates) | `setCurrentTask` → agent dispatch → `clearCurrentTask` |
| `/sig:verify` | (any prior fields) | `markFresh` at phase exit |
| `/sig:review` | (any prior fields) | `markFresh` at phase exit |
| `/sig:checkpoint` | All fields | `setCurrentTask` / `clearCurrentTask` for proposed transitions; `markFresh` on apply |
| `/sig:calibrate`, `/sig:discuss`, etc. | (state.phase only) | `initState` / `transitionPhase` |
